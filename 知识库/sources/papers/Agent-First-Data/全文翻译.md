# Supporting Our AI Overlords: Redesigning Data Systems to be Agent-First
## 论文全文翻译

### 摘要（Abstract）

大语言模型（LLM）Agent 代表用户操作和分析数据，很可能成为未来数据系统的主导工作负载。Agent 处理数据时采用我们称之为 **agentic speculation** 的高吞吐量探索与方案形成过程。agentic speculation 的巨大规模和低效性可能对当今数据系统构成挑战。我们主张数据系统需要适应以更原生地支持 agentic 工作负载。基于 agentic speculation 的特性——规模、异质性、冗余性、可引导性——我们勾勒出新的 agent-first 数据系统架构的研究机会，涵盖新的查询接口、查询处理技术到新的 agentic 内存存储。

---

### 1. 引言

LLM 能够推理、调用工具、编写代码、相互通信——我们正处在新的 agentic 革命的边缘，它将彻底改变数据系统的使用方式。

**Agentic Speculation**：发现（metadata discovery）与方案形成（solution formulation）的高吞吐量组合过程——大量探索性查询以确定最佳行动方案。

**三个例子**：
1. **咖啡豆销售分析**：Agent 军队同时发出海量查询——许多是探索性的"接地"（grounding），纯粹浪费
2. **航班延误机组重组**：需并行探索多种假设性事务，每个都有数十次数据库更新
3. **数据转换**：Agent 在决策前探索多种"what-if"方案

**四个可优化特性**：
1. **高吞吐（scale）**：成百上千请求/秒
2. **异质（heterogeneity）**：从粗粒度元数据探索到精确方案验证
3. **冗余（redundancy）**：大量请求访问相似数据或重叠操作
4. **可引导（steerability）**：数据系统可主动引导 agent 请求方向

### 2. 案例研究（Case Studies）

#### 2.1 BIRD text2SQL

**并行设置**：50 个独立 LLM 尝试，agent-in-charge 选最佳
- 成功率随尝试数 K 增长 **14-70%**（GPT-4o mini / Qwen2.5-Coder-7B）
- **冗余发现**：不同查询间 distinct sub-plan 占比仅 10-20%

**顺序设置**：单个 agent 多轮迭代，成功率随轮数增长

#### 2.2 多异质后端任务

22 个任务（PostgreSQL + SQLite + MongoDB + DuckDB），44 条 o3 trace

**阶段模式**：
- 早期：探索表和列
- 中期：列统计
- 后期：部分/完整查询
- 但各阶段**并不清晰分离**，在整个 trace 中重叠

**可引导性验证**：注入人类专家 hints
- 表探索查询：减少 14%
- 列探索查询：减少 28%
- 部分查询尝试：减少 37%
- 整体 SQL 查询：减少 18%

### 3. Agent-First 数据系统架构

```
User Task
  ↓
[LLM Agent(s)] → probes (+ briefs)
  ↓
┌──────────────────────────────────────┐
│ Query Interfaces — Sec.4 （粉色）      │
├──────────────────────────────────────┤
│ Probe Optimizer — Sec.5 （蓝色）       │
├──────────────────────────────────────┤
│ Storage & Transactions — Sec.6 （橙色） │
└──────────────────────────────────────┘
```

**Agentic Interpreter**：解析 agent 的 probe + brief

**目标转变**：不是最大化吞吐，而是 **satisficing**——提供"刚好够"的信息让 agent 决策下一步

### 4. 查询接口（Query Interfaces）

#### 4.1 Agent → 数据系统

**Probe（探针）> Query（查询）**的原因：

**提供背景信息（Brief）**：
- 自然语言 goals/intents
- 当前 phase（exploration / solution formulation）
- 近似度需求 / 优先级
- 开放目标：如 "n 个省份中挑 k 个代表东西海岸"

**语义搜索能力**（非 SQL）：
- "找跟 electronics 语义相关的所有表/列/行"
- LIKE 无法替代的语义相似性

**终止条件函数**：
- Agent 定义函数，数据系统在 partial results 上评估，触发提前终止

#### 4.2 数据系统 → Agent

**Sleeper Agent（睡眠探员）**：
- 随探针并行触发的轻量 LLM agent
- 提供两类输出：

**辅助信息**：
- 关联表推荐（join discovery）
- why-not provenance：解释空结果原因（例如"你用 'CA' 但表里存的是 'California'"）
- 类似的已物化答案

**成本估算反馈**：
- 预测查询成本 → 建议缩小范围 / 增加近似度
- 建议批处理多个探针

### 5. 处理与优化探针（Probe Optimization）

核心思想：优化目标从 **最大化吞吐** 转为 **最小化 agent 完成任务的总体时间**。

#### 5.1 支持探索

**挑战**：agent 用自然语言 + SQL 混合探索
- 元数据查询返回成千上万行 → 浪费 agent 上下文长度
- 探索阶段查询不应被完全评估

**关键能力需求**：
- 识别查询 phase → 决定执行精度（admission control）
- 语义缓存已探索的元数据 → agent 查询内存存储而非重复 SQL

#### 5.2 探针优化

##### Intra-Probe（批次内）

**决定执行什么**：
- 语义分析 brief → 剪枝不相关查询/列
- 探针间比较 → 合并重复信息
- Phase-based 近似度（探索用 coarse grain，方案验证用 high accuracy）

**高效执行**：
- MQO（多查询优化）+ AQP（近似查询处理）+ 结果缓存
- 增量查询：先给部分回答，逐步提高精度
- 查询规划与处理阶段可重新决策近似度

##### Inter-Probe（跨批次/跨代理）

**决定执行什么**：
- 与历史已答查询比较 → 无新信息则跳过
- 预判后续探针 → 当前多花时间完全执行以节省后续总时间
- **探索 vs 利用**：有时优先探索未充分评估的方案空间

**高效执行**：
- 预物化：预测后续探针会用到的 join → 提前物化
- 跨 agent 缓存共享

---

### 6. 索引、存储与事务

#### 6.1 Agentic Memory Store（语义缓存）

**存储什么**：
- 历史探针结果 + 部分方案
- 列的编码格式、缺失值、时间/位置粒度
- 元数据嵌入表本身 → 查表即得

**实现方式**：
- 向量索引：支持"探索语义相似的先前探针"
- 但结构化查找也需要直接支持

**更新挑战**：
- 新探针可能让旧信息过时
- schema 变更使历史探针结果无效
- 多用户隐私共享博弈

#### 6.2 分支事务（Branched Updates）

**观察**（Neon）：agent 创建 20× 更多分支，50× 更多 rollback

**核心需求**：Multi-World Isolation（逻辑隔离 + 物理共享）

**借鉴**：
- 分支一致性：Bayou / Dynamo / Tardis
- 版本数据库：ORPHEUSDB
- CoW：Neon / Aurora / Bauplan

**新挑战**：
- Agent 间分支 reconciliation（不仅是与 mainline，还有 branch vs branch）
- **MVCC on steroids**：成千近相同快照，fast abort
- 相似分支之间的计算共享（但逻辑隔离不交叉污染）

---

### 7. 结论

我们描述了 agent-first 数据系统的愿景：
- agentic speculation 的四个特性（scale / heterogeneity / redundancy / steerability）为全新优化提供机会
- 三层架构：查询接口（probe + sleeper agent）→ 探针优化器（satisficing）→ 存储（agentic memory + branch TX）
