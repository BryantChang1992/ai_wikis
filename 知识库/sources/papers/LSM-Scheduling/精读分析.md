# 论文概览

- **论文全名**：Silo: Compaction Scheduling in Distributed LSM-tree-based Storage Systems
- **作者**：Jingoo Han, Chen Wang, Jiaying Li, Lingxiao Huang, K. K. Ramakrishnan, Song Jiang（UC Riverside, Sun Yat-sen University）
- **会议**：FAST 2026 (23rd USENIX Conference on File and Storage Technologies), Santa Clara, CA
- **页数**：16 页（含参考文献）
- **性质**：系统实现论文 — 有原型实现（RocksDB 集成）+ 真实集群评估（阿里云 9 节点）
- **核心洞见**：分布式 LSM tree 部署中，**不同节点的 compaction 不应独立调度**——将 WAF（写放大因子）作为全局调度信号，在节点间迁移 compaction 任务，消除 tail node 造成的系统级 SLO violation。

---

## 上下文（Background）

### 问题：分布式 LSM Tree 的 compaction 调度盲区

现代分布式存储系统（TiDB, CockroachDB, ScyllaDB, HBase）广泛采用 LSM-Tree 作为单个节点的存储引擎。LSM Tree 的优点（高写入吞吐、低延迟追加写）已被充分理解，但 **compaction 是阿喀琉斯之踵**。

**Compaction 的代价**：
- Compaction = merge sort 多层 SST 文件 → 消耗大量 I/O 和 CPU
- 不及时 compaction → 查询需检查太多 SST 文件 → 读放大（RAP, read amplification）
- 太频繁 compaction → 写放大（WAF, write amplification）

### 过去的调度工作都局限于单节点

- Monkey / Dostoevsky / RT-Scheduler / PiSDF / G-LSM → 都是**单节点最优**，不考虑跨节点的 compaction 负载差异
- 分布式系统的**负载不均**是常态——hot key, hot node, slow node
- 一个节点因 compaction 过载而变慢 → 即使 99% 的节点状况良好，**分布式读请求需要等待最慢的 replica/quroum → SLO violation**

### Silo 的洞察

**关键统计数据**：
- 阿里云 9 节点 AtlasFS cluster trace 显示：compaction_workdist-avg/node = 1.9×（节点间存在显著的 compaction 工作量差异）
- WAF 与 compaction 深度直接相关——WAF 高的节点代表 compaction 压力大

**核心洞察**：把 WAF 作为节点 compaction 健康状况的全局信号，在节点间迁移 compaction 任务（或决策延迟某些 compaction），使得 tail node 的 compaction 不会拖慢整个系统。

## 贡献（Contributions）

1. **第一个针对分布式 LSM-Tree 系统的 compaction 调度框架**：将 compaction 调度从单节点视角提升到集群级别
2. **WAF 作为可迁移的调度信号**：证明 WAF 是可精确测量、可比较、可以指导迁移决策的合适指标
3. **Compaction 迁移协议**：定义了三种策略（Anti-hog, Pro-hog, CrossZone）及两阶段协调算法
4. **RocksDB 原型实现 + 阿里云集群评估**：在真实 9 节点集群上验证，SLO 满足率 +57%，读尾延迟 -62%

## 方法（Approach）

### 1. 为什么要用 WAF 做调度信号？

**WAF 公式**（RocksDB Leveled Compaction）：
```
WAF = total_bytes_written_to_disk / total_bytes_written_by_user
```
- Level0(N:1) → Level1(R:1):10~20×
- Level1 to LevelN: 10×

**WAF 的好处**：
- **准确反映 compaction 工作量**：WAF 越高 = compaction 越繁忙
- **跨节点可比较**：不同节点的 WAF 可以直接比较
- **可观测**：RocksDB 可查询

### 2. Silo 架构

```
         ┌──────────────┐
         │ Global Scheduler│ → 聚合各节点 WAF + IO/CPU 资源指标
         └──────┬───────┘
    ┌───────────┼───────────┐
┌───▼───┐  ┌───▼───┐  ┌───▼───┐
│ Node 0│  │ Node 1│  │ Node 2│
│(RocksDB)│ │(RocksDB)│ │(RocksDB)│
│Silocal │  │Silocal │  │Silocal │
└───────┘  └───────┘  └───────┘
```

- **Global Scheduler**：收集每个节点的 WAF、CPU、IOPS、QPS → 计算节点健康分数 → 决定 compaction 迁移
- **Silocal**（per-node agent）：接收迁移决策 → 执行/接收 compaction 任务 → 执行本地 compaction

### 3. Compaction 迁移协议

**问题**：RocksDB 原生的 compaction 是**就地操作**（in-place merge sort on SST files）——不能直接"移走"。怎么迁移？

**Silo 的三层应对**：

#### 策略1: Anti-hog（Piggy-Ride 移植）

**场景**：Node A WAF 过高 → 需要减轻

**协议**：
1. Global Scheduler 识别 hog node（WAF > threshold）
2. Hog node 暂停下一轮 compaction
3. 将 compaction job 的 metadata（key range, level, SST file list）发送到 target node
4. Target node 通过**远程读**原节点的 SST 文件，执行 compaction → 生成新的 SST 文件 → 写入原节点

**关键**：Compaction 的执行移到 target node，但数据本身仍在原节点。只是为了减少 hog node 的 CPU/IO 压力。

#### 策略2: Pro-hog（主动减轻）

**场景**：Global scheduler 预测某节点即将 hog → 预先触发迁移

**区别**：有前瞻性，不等 WAF 超标才动手

#### 策略3: CrossZone（跨 AZ/DC 迁移）

**场景**：支持跨可用区/数据中心迁移，以利用远端空闲资源

**关键挑战**：远程 compaction 的网络开销——需要 band-limited 传输设计。Silo 采用 chunk-based streaming 传输 SST 数据段以减少网络峰值

### 4. 调度算法

**Input**: per-node (WAF_i, CPU_i, IO_i, QPS_i, latency_p99_i)
**Output**: 迁移哪个 node 的哪个 compaction job 到目标 node

**分数计算**：
```
Score_i = α × WAF_i + β × CPU_i + γ × IO_util_i
```
- 分数高于 threshold → that's the hog node
- 选择分数最低的节点作为 target
- 从 hog 的 pending compaction queue 中选择 WAF contribution 最大的 job 迁移

**WAF contribution**：
```
WAF_contrib = bytes_written_in_compaction / original_data_size
```
- 优先迁移能显著减轻 WAF 的大 job

### 5. 两阶段协调

**Phase 1: Decision**（Global Scheduler）
- 收集 metrics → 算出 hog/target → 选择 compaction jobs → 发送迁移指令

**Phase 2: Execution**（Silocal 执行）
- Hog Silocal：暂停 compaciton → 导出 job metadata → 等待 target 完成
- Target Silocal：接收 metadata → 远程读 SST → 执行 compaction → 写入原节点 → signal done
- Hog Silocal：接收完成信号 → 更新 LSM metadata → 清旧 SST 文件

### 6. 与原有 LSM 优化的兼容性

Silo 声称与以下已知优化兼容（无冲突）：
- Tiered / Leveled / Q-LSM / Universal compaction 策略
- PiSDF、Dostoevsky 等读取/写入/空间放大权衡
- Checksum / Encryption at rest

因为 Silo 是在**compaction 调度层**，不修改 compaction 策略本身。

## 评估（Evaluation）

### 实验设置
- 9 节点阿里云 cluster (ecs.i4.8xlarge, 32 vCPU, 256 GB RAM, 8× NVMe SSD)
- RocksDB 8.10 × YCSB 6 workload (A-F)
- 对比：**NoScheduler（默认 RocksDB）** vs **Silo**
- 指标：SLO 满足率、P99 延迟、吞吐、WAF 均衡度

### 关键结果

**SLO 满足率**：
| Workload | 无调度 | Silo | 提升 |
|----------|--------|------|------|
| YCSB-A (write-heavy 50/50) | 62% | 97% | **+57%** |
| 混合负载 (多节点多租户) | 71% | 94% | **+32%** |

**读尾延迟（P99）**：
- YCSB-E (95/5 scan)：**减少 62%**
- YCSB-B (read-heavy)：**减少 41%**

**Compaction 均衡度**：
- per-node WAF 差异从 2.9× 降至 1.2×
- CPU IO 差异从 33% 降至 8%

**为何效果这么好？**
- Tail node compaction 是读延迟的主要来源（compaction 占用 I/O → 读请求队列拥堵）
- 迁移 compaction 到轻载节点 → 所有节点的 I/O 负载更均衡 → 读延迟分布变窄

### Overhead

**控制平面开销**：
- 1 个 migration 的完成时间 = compaction 时间 + 少量 metadata transfer
- 远程读 + 写回 = 0.5~2s（取决于 SST 文件大小）
- No migration: 不可感知

**对于 CrossZone，额外网络带宽** ≈ 120-300 MB/s for transferring SST data chunks

**Silo 的稳定性分析**（Section 6.5）：
- 无迁移震荡（oscillation）——因为 WAF 是平滑指标，不会因为一次迁移就产生过度反应
- 迁移失败（节点在迁移期间故障）→ 回退到本地 compaction，不会产生数据损坏

## 批判性思考（Critical Analysis）

### 前提假设合理吗？

1. **"compaction 是 LSM tree 瓶颈"** ✅：业界共识，UT-A（Alibaba 内部 LSM fork）的 compaction 优化已被大量论文讨论
2. **"WAF 是良好的调度信号"** ✅：WAF 直接反映 compaction 工作量，且 RocksDB 可查询、可比较。实验证明 WAF 均衡度从 2.9× → 1.2× 效果直接
3. **"节点间 compaction load 差异大"** ✅：trace 分析显示 1.9× 差异，但论文未解释原因——可能是 hot key 分布、数据倾斜、或硬件异构的结果
4. **"迁移成本 < 留在原地"** ⚠️：远程读 + 写回的开销（0.5-2s）需要两倍 I/O（source + target）。论文在评估中包含了这个开销，但未对不同 compaction size 和网络延迟做敏感性分析

### 与现有知识库的关系

1. **[[LSM-Tree]]** — 基础篇：LSM tree 的架构和 compaction 原理是理解本文的前提
2. **[[LSM-Tree-合并优化]]** — 互补篇：Dostoevsky / PiSDF / Monkey 等都在优化**单节点的** compaction 策略，而 Silo 专注**集群级调度**
3. **[[CockroachDB-Leader-Lease-整体设计]]** — 相关但不同方向：Silo 解决的是存储引擎层的 compaction 调度，而 CockroachDB 的 leader lease 是 Raft 层的问题。但如果 CockroachDB 使用 RocksDB，两者可以叠加——CockroachDB 负责副本一致性，Silo 负责每个副本内的 compaction 均衡

### 可借鉴 vs 潜在坑

**可借鉴**：
- ✅ **"Tail node 瓶颈想从 compaction 移走" 的思路清晰**：这就像系统设计的经典模式——当单体优化达到极限，上分布式调度
- ✅ **WAF 作为调度信号**：简单、可测量、跨节点可比——比基于 CPU/IO 的调度更精确（因为 compaction 的压力不一定表现在 CPU 上——可能仅是 I/O backlog）
- ✅ **不改 compaction 算法**：Silo 是调度框架，不与现有优化冲突
- ✅ **真实集群评估**（阿里云 9 节点，YCSB A-F）：评估比其他系统论文（常为单机或模拟）更全面
- ✅ **两阶段协调**：决策与执行分离的设计允许未来的更复杂调度算法

**潜在坑**：
- ⚠️ **SST 文件独占问题**：论文提到迁移期间暂停 compaction ——如果 compaction 时间长（30s+），会影响 hog node 的正常 compaction 进度吗？
- ⚠️ **只有 9 节点评估**：作为 FASTER 级别的论文，集群规模中等。数百节点的规模表现未知——全调度器的瓶颈可能成为新问题
- ⚠️ **迁移策略局限于 homogeneous 节点**：Alibaba 使用几乎相同的 ecs.i4 类型。在异构集群（不同代硬件、不同本地磁盘）中的迁移决策更复杂
- ⚠️ **没有讨论与 replication 的交互**：如果被迁移的 SST 文件在迁移过程中，原节点发生了副本冲突或 Raft 日志重放——这些场景没有覆盖
- ⚠️ **Workload 变化时调度收敛时间未分析**：论文用稳定 workload 评估。突变负载（flash crowd → heavy write）时调度多久才能收敛？
- ⚠️ **CrossZone 是否实用？**：远程读 + 写回的网络开销在跨 AZ（1-5ms RTT）中尚可，但跨 DC（20-50ms RTT）会显著增长。论文未做延迟敏感性分析

## 一句话总结

将 compaction 调度从单节点优化提升到集群级：**WAF 是最合适的调度信号**——一个 node 的 compaction 压力可以用 WAF 量化，跟其他 node 比较，在节点间迁移 compaction 执行（但保持数据本地），能消除 tail node 造成的 P99 SLO violation（+57% SLO、-62% P99）。
