# 《A Survey on the Evolution of Stream Processing Systems》精读分析

## 1. 论文概况

- **作者**：Marios Fragkoulis (Delivery Hero) / Paris Carbone (RISE/KTH) / Vasiliki Kalavri (Boston University) / Asterios Katsifodimos (TU Delft)
- **发表**：arXiv 2008.00842v2，2023年1月更新版
- **篇幅**：30页
- **类型**：综述 (Survey)
- **覆盖范围**：流处理系统 20+ 年演化，从 1992 年 Tapestry 到 2022 年 Stateful Functions

## 2. 核心分类框架

论文将流处理系统分为三个代际：

| 代际 | 时间 | 代表系统 | 特征 |
|------|------|---------|------|
| 1st Gen (DSMS) | '00–'10 | STREAM, Aurora/Borealis, TelegraphCQ, Gigascope | Scale-up，有序处理，关系型流模型 |
| 2nd Gen (Dataflow) | '11–'22 | Storm, Flink, Spark Streaming, Beam/Dataflow, Kafka Streams | Scale-out，数据流模型，无序处理，强状态管理 |
| 3rd Gen (Emerging) | '22– | Ray, Stateful Functions, Neptune, Ambrosia | 事件驱动架构，定点计算，硬件加速，微服务集成 |

## 3. 六大功能域

论文将流处理系统的功能演进归纳为六大领域：

### 3.1 乱序数据管理 (Out-of-Order Data Management)

**核心命题**：乱序是流处理的常态而非异常。

- **乱序根因**：网络延迟不均、多源汇聚、Join/Window 操作的副作用
- **两类架构**：In-Order Processing（先排序再处理，DSMS 主流）vs Out-of-Order Processing（原生支持乱序，Dataflow/Beam 引入）
- **关键机制**：Sliding Windows + Slack（低水位标记 LWM）/ Punctuations（嵌入流中的进度标记）/ Pointstamps（Naiad：时间戳+位置的二维进度标记）/ Dataflow Model（事件时间/处理时间分离 + Watermark + 触发条件 + 修正）

**开放问题**：循环查询的时间进度追踪；多时间域窗口聚合的通用抽象

### 3.2 状态管理 (State Management)

**核心命题**：状态管理是区分 1st Gen 和 2nd Gen 的决定性能力。

- **状态类型演化**：Synopsis（1st Gen：系统内部算子状态）→ Application-Managed（Storm：用户自行管理）→ User-Defined, System-Managed（Flink/Beam：用户定义+系统管理持久化）
- **三种架构**：In-Memory / Out-of-Core（LSM-Tree: RocksDB/FASTER）/ External（Millwheel 模式，状态在 BigTable/Spanner）
- **持久化粒度**：Epoch-Level（Chandy-Lamport 分布式快照）/ Batch-Level（Spark Streaming/Trident）/ Record-Level（Millwheel：每条输出+状态写入事务）
- **可扩展状态操作**：Partitioned State（Keyed）vs Global/Broadcast State

**开放问题**：工作负载感知的状态管理；状态独立性的形式化；MLSys/图分析等新负载的状态需求

### 3.3 容错与高可用 (Fault Tolerance & High Availability)

**核心命题**：Exactly-Once ≠ 端到端 Exactly-Once；Output Commit 问题是关键。

- **处理语义**：At-Most-Once（已过时）/ At-Least-Once（Storm 默认）/ Exactly-Once on State（系统内状态一致，但可能输出重复）/ Exactly-Once on Output（输出也要保证无重复——此即 Output Commit Problem）
- **复制策略**：Active Replication vs Passive Replication
- **恢复数据**：State Snapshots / Output Buffers / Input Log Offsets
- **存储介质**：Resilient Store (Local/Remote) / In-Memory
- **高可用定义缺失**：论文指出流处理领域没有标准的高可用定义，提出基于处理进度的定义方案

**开放问题**：Exactly-Once on Output 的通用方案；高可用的形式化量化指标；非确定性计算+容错

### 3.4 负载管理、弹性与重配置

**核心命题**：弹性现在是标配；运行时重配置才是现代流处理的核心挑战。

- **负载管理**：Load Shedding（DSMS 丢数据策略）/ Load Balancing（2nd Gen 键值再分配）
- **弹性策略**：症状检测 → 根因分析 → 扩缩容决策（目前大多要求用户自行决策）
- **重配置**：SEEP（集成式扩缩容+恢复）/ Chi（Live Reconfiguration 含 Alignment Phase）/ Megaphone（基于数据流图的拓扑重配置）
- **Flow Control**：Buffer-based Back-Pressure / Credit-based Flow Control (CFC)

**开放问题**：自动化弹性决策；运行时重配置的零停机；跨算子状态迁移的通用化

### 3.5 数据流模型与语义 (Dataflow Model & Semantics)

以 Google Dataflow Model 为标志性里程碑：

- **Relational Model**（1st Gen）：流是演变的关系，Schema 由系统定义
- **Dataflow Model**（2nd Gen）：只要求 Timestamp，语义由用户代码定义
- **时间域**：Event Time / Processing Time / Ingestion Time
- **Watermark**：事件时间的进度估计
- **Trigger & Refinement**：何时物化窗口结果 + 如何修正早期结果

### 3.6 架构演化全景

| 维度 | 1st Gen (DSMS) | 2nd Gen (Dataflow) |
|------|----------------|-------------------|
| 架构 | 单机/Scale-up | 分布式/Scale-out Shared-Nothing |
| 查询模型 | 共享查询计划 | 独立 Job/DAG |
| 数据模型 | Relational (Schema-on-System) | Dataflow (Schema-on-User) |
| 乱序处理 | Buffer + Sort → In-Order | Watermark + Trigger → Out-of-Order |
| 状态管理 | Synopsis (系统内部) | System-Managed State (显式API) |
| 容错 | Best-Effort / At-Least-Once | Exactly-Once on State |
| 弹性 | 无 | 手动 → 半自动 |
| 目标 | 低延迟 | 高吞吐 + 正确性 |

## 4. 与当前知识库的关联

### 4.1 直接关联概念

- **[[LSM-Tree]]**：多处提及 LSM-Tree 是流处理系统外存状态管理的标准选择。RocksDB (Flink) / FASTER (MSR) 均为 LSM 变体。
- **[[Fluss-架构体系综述]]** / **[[Fluss-流处理平台架构综述]]**：Fluss 作为新兴事件流平台，论文的分类框架可帮助定位其在流处理演化中的位置。
- **[[分布式数据系统事务与一致性新进展-2026综述]]**：论文的事务保障讨论（Exactly-Once on State/Output）与本综述的事务部分直接关联。
- **[[LSM-Tree-存储引擎体系综述]]**：LSM-Tree 作为流处理状态后端，Out-of-Core 架构描述与其有交叉。

### 4.2 补全知识缺口

论文覆盖了当前知识库缺少的方向：

1. **流处理的乱序语义**：Watermark / Trigger / Refinement 的完整体系
2. **流处理容错保障的分级体系**：At-Least-Once → Exactly-Once on State → Exactly-Once on Output 的递进关系
3. **Dataflow Model**：Google Dataflow 的统一批流处理模型
4. **弹性与运行时重配置**：流处理系统的动态扩缩容策略

## 5. 方法论评价

### 优点
- 覆盖全面：1992→2022，30年跨度，100+ 引用
- 功能域分类清晰：六大域 + 1st vs 2nd 对比结构
- 统一的术语命名：解决不同系统中不一致的术语
- 开放问题清单实用：每个功能域附带了待解决问题

### 局限
- 对 3rd Gen（新兴阶段）仅给出方向性判断，缺乏系统性分析
- 对 SQL 流处理的标准化工作（如 Apache Calcite 流式扩展）讨论较少
- 对 ML 推理与流处理的集成（Model Serving）仅一笔带过
- 缺少性能基准对比（各系统的延迟/吞吐量化比较）

## 6. 对 CHANG_AI_TEAM 的启示

1. **流处理是 AI Agent 基础设施的关键组成**：事件驱动架构（3rd Gen 趋势）与 Agent 系统的消息驱动模型天然匹配
2. **可观测性平台可直接受益**：Watermark / Progress Tracking 机制可应用于数据管道的延迟监控
3. **后续论文选择建议**：
   - Dataflow Model (Akidau et al., VLDB 2015) — 批流统一模型
   - Millwheel (Akidau et al., VLDB 2013) — 外部状态 + Exactly-Once
   - Naiad (Murray et al., SOSP 2013) — 定时数据流 + Pointstamps
   - Clonos (Silvestre et al., EuroSys 2021) — 非确定性计算容错
