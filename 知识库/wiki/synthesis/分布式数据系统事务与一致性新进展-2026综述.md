# 分布式数据系统事务与一致性新进展（2026 论文综述）

> 综合 2026 年 SIGMOD / CIDR / FAST 三会 5 篇论文，覆盖 CockroachDB、Amazon Aurora、Rosé、Agent-First 四大系统的事务与一致性设计。

- **战场一：共识层的规模化**——CockroachDB 提出 Leader Leases + Liveness Fabric，将 O(N_groups) 的 lease 维护降到 O(N_nodes²)
- **战场二：时间戳替代 xid 的事务实现**——Aurora Limitless 用 AWS Time Sync + HLC 实现外部一致性，同时保持 2PC 的高可用
- **战场三：事务模型的边界扩展**——Rosé 将异步复制与事务持久化解耦，Agent-First 将分支事务引入 LLM Agent 工作流

三条线看似独立，但有一个共同趋势：**把事务系统从"数据库内核"推到"基础设施层"**——共识检测变成共享服务、时钟变成云服务、事务变成应用 API。

## 二、CockroachDB：共识层的去中心化革命

### 问题：per-group lease 的 O(N) 困境

分布式数据库为强一致读引入 Lease 机制——持有 lease 的副本可以不经过共识就服务读请求，大幅降低读延迟。但当数据被分割为数十万个 Raft group 时，每个 group 独立维护自己的 lease 成为 CPU 和网络瓶颈。

CockroachDB 旧方案的双输局面：
- **Expiration Lease**：每个 Range 通过 Raft 日志周期性续约 → 10 万 Range 消耗 90%+ CPU
- **Centralized Lease**：通过集中式 liveness Range 管理 node 级 epoch → CPU 极低，但 **partial partition 时 leaseholder 能续约却不能写** → 永久不可用

### 解法：三层解耦

```
Leader Lease  ← 极薄包装：lease = fortified leader 的 [now, LSU)
Raft (修改版) ← Fortification / LSU 计算 / 关闭心跳
Liveness Fabric ← 集群 mesh：每对 store 间双向 support
```

**Liveness Fabric**：将故障检测从 Raft 中抽离为共享基座。节点间维护有向 support 关系 `(epoch, expiration)`——n₂ 承诺在 expiration 前不会撤销对 n₁ 的 support。一旦 support 被撤销，永不复原。这个设计允许关闭 Raft 心跳，所有 Raft group 共享同一套故障检测。

**Leader Fortification**：修改 Raft 协议，leader 通过 `MsgFortifyLeader` 从 follower 获得"在时间戳 X 前不发起选举"的确定性承诺，替代心跳超时的随机性。结合 Liveness Fabric 的节点级存活判断，leader 能提供可证的安全时间窗口。

**Leader Lease**：lease = fortified leader 的 `[now, LeaseStartUpperBound)`。因为 fortification 给出的是 leader 地位的确定性窗口，lease 的安全性直接继承。

### 收益
- 故障检测从 O(N_groups) 降到 O(N_nodes²)
- CPU 节省 85%+
- 消除集中化 lease 的 partial partition 永久不可用问题

### 与 Aurora 的对比
| 维度 | CockroachDB Leader Leases | Aurora Limitless |
|------|--------------------------|-----------------|
| 共识协议 | Raft (multi-group) | 无（Aurora Storage quorum writes 代劳） |
| 故障检测 | Liveness Fabric (共享 mesh) | Shard standby + write lease + zombie detection |
| 读路径优化 | Lease 免共识读 | 单 shard 查询直推 |
| 写扩展 | Raft leader per group | Hash sharding + lead shard 2PC |
| 适用场景 | 跨 AZ + 跨 region（多活） | 跨 AZ 单 region |

## 三、Aurora Limitless：云原生时间戳事务

### 为什么时间戳？

传统 PostgreSQL 的 xid-based snapshot 有两个致命问题：
1. 高并发下 snapshot 计算是 CPU 瓶颈（需要 `{xmin, xmax, xip_list}` 三元组）
2. 分布式环境下单节点不知道所有运行中的事务

Aurora Limitless 的答案：**用物理时钟时间戳替代 xid**。依托 AWS Time Sync 服务（卫星同步原子钟集群，CEB < 1ms，部分区域低至双位微秒），事务的 startTs 和 commitTs 都是绝对时间。

### Clock-SI + HLC 混合方案

这是 Aurora Limitless 最具创新的设计选择：**不是纯 Clock-SI，也不是纯 HLC，而是两者混合**。

- **纯 Clock-SI 的问题**：读事务的 snapshot timestamp 可能在 shard 本地时钟之前 → 需要等待时钟追上（延迟）
- **纯 HLC 的问题**：不保证外部一致性（CockroachDB 已知的 tradeoff）
- **Aurora 的混合方案**：用 AWS Time Sync 的 `now().latest` 作为 startTs，用 HLC 的 `C = max{C, T.startTs + 1}` 消除读延迟，用 commit wait（`now().earliest > commitTs`）保证外部一致性

结果是：**读不等待 + 写外部一致性 + 无需保守时钟偏移假设**（不像 YugabyteDB 那样设置数百毫秒的保守偏移）。

### Lead Shard 2PC：无状态 Router 下的高可用妥协

Router 无 standby（省钱），所以不能当 2PC coordinator（故障恢复需数分钟→事务长时间阻塞）。解法：**把事务状态持久化到 Lead Shard**（有 standby，秒级恢复）。

这个设计在"省钱"和"高可用"之间取得了好的平衡：
- Router 故障 → 事务由 Lead Shard 裁决，最多秒级恢复
- Lead Shard 故障 → standby 接管
- Router + Lead Shard 同时故障 → 降级为 abort（安全侧）

### 二维扩缩容：独一份

| 维度 | 机制 | 粒度 | 自动 |
|------|------|------|------|
| 垂直 | ACU (Serverless V2) | 单节点 | ✅ 动态比例分配 |
| 水平 | Shard split (table slice) | 512 slices/table | ✅ heat-management 触发 |

竞品中无人同时做到两者：CockroachDB 只有水平（auto rebalance），Aurora Serverless 只有垂直，PolarDB 只有水平。

## 四、Rosé：异步复制的持久化解耦

### 核心思想

传统主备复制的假设是"WAL 先落地，副本再同步"——复制和持久化耦合在一起。Rosé 提出 **Coordinated Apply**：主节点持久化 WAL，备节点持久化 KV 存储，两者解耦，在不同的存储层上分别处理。

好处：
- 备节点可以选择性地持久化（对不重要的表跳过高代价写入）
- 主备的持久化策略可以独立配置（主节点全量 WAL，备节点只存热表）
- 灵活的一致性保证：可以为不同分区配置不同的复制语义

### 与知识库上下文的关系

Rosé 的异步复制 + 分区灵活度，和 CockroachDB 的多 Raft group 在不同维度解决了"数据分布的灵活性"问题——但 Rosé 面向的是**读扩展 + 灾备**场景（异步、单主），而不是多活写的场景。它的 tradeoff 是：牺牲写扩展换取更灵活的持久化策略和更低成本的副本。

## 五、Agent-First：事务模型遇上 LLM Agent

### 核心命题

传统事务模型假设"一个请求 = 一个事务 = 一个线性执行计划"。LLM Agent 颠覆了这个假设：
- Agent 的执行路径是**非确定性的**——它可能在中间任何一步重新规划
- Agent 需要**探索多个备选路径**——哪个更优不可预知
- Agent 的"提交"可能是**逐步的、部分可逆的**——不像传统事务要么全提交要么全回滚

### 分支事务（Branch Transactions）

关键区别：
- **分支是轻量级的**：基于 MVCC 快照 fork，而非完整数据复制
- **合并是有策略的**：可以选最优分支、合并互补分支、或回滚所有分支
- **可观测性内建**：每个分支有独立 trace，Agent 可以在运行时"自我调试"

### 语义缓存与 Agentic Speculation

Agent-First 的另外两个关键技术：
- **语义缓存**：基于含义而非精确 key 匹配的缓存——"查上周的订单总额"和"查 7 天内的 sales"可能命中同一缓存
- **Agentic Speculation**：事务开始前预估后续操作，提前预取数据——用概率模型而非规则来驱动

### 与知识库的关系

Agent-First 的事务模型是对 [[事务模型深度调研]] 中 ACID/MVCC/2PC/Spanner 体系的**语义层扩展**——它不是替换现有的并发控制，而是在其之上加了 Agent 语义层。分支事务本质上是 MVCC snapshot 的应用，语义缓存可以看作 Redis/向量检索之上的语义层。

## 六、跨论文洞察

### 共同的趋势线

1. **从"系统内"走向"基础设施层"**——CockroachDB 把故障检测做成共享服务，Aurora 把时钟做成云服务，Agent-First 把事务做成应用 API。事务不再是数据库内核的专属。
2. **时间戳正在取代传统方案**——从 Aurora 的 Clock-SI 到 CockroachDB 的 fortification timestamp 到 Agent-First 的 MVCC snapshot fork——时间戳作为分布式一致性的通用语言越来越普遍。
3. **"解耦"是 2026 的主题词**——Liveness Fabric 与 Raft 解耦、Rosé 的 WAL 与 KV 解耦、Aurora 的 Router 与 Shard 解耦。
4. **AI Infra 的数据库需求开始成形**——Agent-First 是这轮五篇里唯一面向 AI 场景的，但它提出的分支事务、语义缓存、Agentic Speculation 很可能是下一个 3-5 年的主流范式。

### 技术选型决策矩阵

| 需求场景 | 推荐参考 | 竞品对比 |
|---------|---------|---------|
| 大规模 Raft 集群扩容 | CockroachDB Leader Leases | TiKV PD-based scheduling |
| 云原生 PG 水平扩展 | Aurora Limitless | Citus, YugabyteDB, PolarDB |
| 异步读扩展 + 灾备 | Rosé Coordinated Apply | MySQL async replication, Kafka MirrorMaker |
| LLM Agent 数据存储 | Agent-First Branch Transactions | 尚无直接竞品 |
