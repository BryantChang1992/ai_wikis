# Rosé: Flexible Replication With Strong Semantics for Partitioned Databases
## 论文全文翻译

### 摘要（Abstract）

异步主备数据库复制因其在写延迟与持久性之间的良好平衡而广受欢迎，但也存在显著缺陷。在分区数据库中，每个分区通常独立复制，这意味着故障切换中的数据丢失可能使数据库处于开发人员难以推理的未定义状态。此外，复制延迟可能随时间增长，导致用户接触过期数据并产生持久性问题。最后，如果备份分区进度不一致，恢复时间和故障后的性能都会受影响。

Rosé 是一种新颖的复制方案，通过在完全同步和完全异步之间取得平衡，解决分区数据库中异步主备复制的局限性。首先，数据库将其现有的快照机制（如实时或 epoch）与异步复制相结合，在备份端提供单调前缀一致性语义。其次，为限制复制延迟，Rosé 提出基于推送的复制机制，可跟踪延迟并对主节点施加回压，同时保持高可用性。第三，Rosé 通过将写操作的复制与其在备份分区 KV 存储中的应用解耦，确保快速恢复和故障后的完整性能。我们将 Rosé 集成到 Chablis（一个地理分布式、多版本事务 KV 存储）中，在保留单数据中心快速事务优势的同时，实现多数据中心持久性。

### 1. 引言（Introduction）

**异步主备复制**：主副本在本地执行和提交所有事务，然后异步将写操作发送到备份副本，备份副本独立应用这些写入来重建主副本的状态。优势是优秀的写延迟（只需主副本确认），劣势是可能丢失最近数据。

**单调前缀一致性（Monotonic Prefix Consistency）**：备份应暴露主副本最近状态的一个渐进序列。在非分区数据库中这相对直观。但在**分区数据库**中，每个 partition 有自己的日志并独立复制——即使每个 partition 的复制都保持单调前缀一致性，备份的整体状态也可能不一致。

**例子**：事务 T 写 K1 到 P1、K2 到 P2。由于复制速度差异，K1 的写入到达了备份 P1，但 K2 的写入尚未到达备份 P2。在备份上同时读 K1 和 K2 → 只能看到 T 的部分效果 → 破坏原子性。

**纸面贡献**：
1. 利用数据库已有的全局一致性快照机制，将单调前缀一致性扩展至分区数据库
2. 基于回压的机制有效限制复制延迟，同时保持高可用性
3. 分离 WAL 复制与 KV 存储应用，解决故障后性能退化问题

### 2. 背景与相关工作（Background and Related Work）

#### 2.1 复制

- **共识复制**：Paxos/Raft → 需多数派确认 → 容 f 个副本故障需 2f+1 个副本 → 写延迟较高
- **主备复制**：primary 本地提交后异步发送 log → backup 应用 log → 低写延迟但可能丢数据
- **单调前缀一致性**（M-P Consistency）：备份的日志必须是有进展的、连续的前缀

#### 2.2 Chablis

Chablis 是 Columbia 开发的地理分布式多版本事务 KV 存储：
- **RangeServer**：管理 key 空间分区，实现 2PL + 2PC，持久化在 WAL 中，异步应用到 KV Service
- **Warden**：健康监控 + Range 分配
- **Epochs**：全局顺序机制——事务在 epoch eᵢ 中一定发生在 epoch eⱼ (eᵢ < eⱼ) 中的事务之前。Regional epoch publisher 提供当前 epoch，global epoch service 定期推进 epoch

### 3. 挑战（Challenges）

Rosé 解决分区数据库异步复制的三个核心挑战：

1. **限制复制延迟**：异步复制不可避免地有延迟；延迟过大会使用户看到过期数据并削弱故障切换能力
2. **最小化恢复时间**：分区进度不一致导致备份有大量"无用"数据（超过最快共同 epoch），故障后需清理
3. **保证恢复后性能**：传统方案（如 Yugabyte xCluster）可在 2s 内故障切换但恢复后性能退化

---

### 4. Rosé 协议

#### 4.1 维护单调前缀一致性

**核心观察**：许多分布式数据库已经提供跨分区的全局一致性快照读。这些快照机制（epoch/HLC/real-time）可以作为分区数据库单调前缀一致性的自然基础。

**方案**：
- 每个 primary partition 按 epoch 递增序发送事务 write-set 到对应 backup partition
- backup 集群的 cluster manager 跟踪每个 partition 的最新完整应用 epoch eᵢ
- backup 对外暴露的快照 epoch = min(eᵢ)
- snapshot epoch 单调递增 → 单调前缀一致性

#### 4.2 限制复制延迟

##### 4.2.1 回压机制

replication_lagᵢ = e_primary,ᵢ − e_backup,ᵢ
effective_lag = min(replication_lagᵢ) —— 一个掉队 partition 影响整个数据库

**Rosé 方案**：
- 每个 partition 维护有界队列（size L）
- Push-based：primary 主动推送变更到 backup，同时监控所有 partition 的复制进度
- 任一 partition 队列满 → **仅对该 partition 限流**
- **Struggler isolation**：掉队 partition 只影响自己，不扩散

**关键参数 L**：需要足够大以利用全网络带宽并吸收正常负载，又要匹配用户对数据新鲜度的要求。若备份区域长时间不可达，管理员/监控系统手动解除回压。

##### 4.2.2 可用性证明

**命题**：Rosé 的可用性不劣于同步复制。

**单 partition 证明**：
- 初始状态：Rosé 接受同步复制的所有事务（因为队列尚未满）
- 队列满后出现新事务：
  - (1) 若同步复制拒绝 → Rosé 也拒绝
  - (2) 若同步复制接受 → 说明备份链路可达，队列可清出空位
- 多 partition 同理

**单 partition 完全不可达的概率**：
- 假设单节点 99.9% 可用性，R=3 复制
- Pr[partition 完全不可达] = (0.001)³ ≈ 10⁻⁹——极低

因此，仅需回压机制覆盖的拥塞场景才是瓶颈。

#### 4.3 最小化故障恢复时间

**问题**：由于 snapshot epoch = min(replicated_epoch)，故障恢复时必须将所有 partition 回滚到该 snapshot epoch。恢复后需删除超过该时间戳的所有数据 → 膨胀恢复时间，或影响恢复后性能。

**Yugabyte 的例子**（4.3.1）：
Yugabyte 在每个 SST 文件的 metadata block 中存储 max_ts。故障恢复时：
- 若 file.max_ts ≤ desired_snapshot_ts → 保持原样
- 若 file.max_ts > desired_snapshot_ts → 在 metadata block 中写入 keep_ts=desired_snapshot_ts
- 读操作需过滤掉超过 keep_ts 的条目，compaction 慢慢清理
- **代价**：即时恢复确实做到了，但恢复后性能退化 22%（吞吐）/ 15%（P99 延迟）

**Rosé Coordinated Apply**（4.3.2）：
- 将 WAL 复制和 KV 存储应用分离
- 备份集群持续跟踪 min(replicated_epoch)，通知各 partition 仅 apply 其 WAL 到该 epoch
- 故障恢复时只需 **trim WAL**（极快操作），KV store 中无脏数据 → **恢复后满性能**

**关键分析**：
- dead_timeᵢ = max(RT) − RTᵢ（慢 partition 应用完成后快 partition 的等待时间）
- 缓解：Rosé 使用毫秒级 epoch → dead time 最小化
- 没有极端 hot partition 时 dead time ≈ 0
- snapshot epoch 推进不受影响——快 partition 的 apply 虽延迟开始，但应在慢 partition 传输完成前完成

### 5. 评估（Evaluation）

设置：单机 Cloudlab c6525-25g 模拟多节点 + 网络故障。对比 Yugabyte 2.25.2.0-b359。

#### 5.2 回压效果（Q1）

- 有回压：lag 被有效限制
- 无回压：lag 不受限增长
- 仅过载 partition 受影响

#### 5.3 故障恢复性能（Q2）

- 两者均在 2s 内完成故障切换
- **恢复后性能关键差异**：
  - Yugabyte：**吞吐下降 22%，P99 延迟上升 15%**
  - Rosé：**0% 退化**（KV store 中无脏数据）

> 注意：作者坦言不比较绝对性能——Yugabyte 是生产级系统（全 SQL 支持），Rosé 是 KV store 原型。相对性能对比才有意义。

### 6. 相关工作

- **共识复制**：Spanner、CockroachDB、Yugabyte → 多数派确认，高延迟但强一致
- **异步主备**：Socrates (SQL Server)、Aurora、Azure Storage → 低延迟但可能丢数据
- **单调前缀一致性**：C5（cloned concurrency control）用于非分区系统，Rosé 将其推广到分区场景
- **Yugabyte xCluster**：pull-based 异步复制 + 即时故障切换（通过 SST 文件 keep_ts），但恢复后性能退化
- **Chardonnay / Chablis**：同组之前的工作，epoch-based 事务系统

---

### 7. 结论

Rosé 是一种新的复制协议，在保留异步复制的低延迟优势的同时，通过三件套解决其典型痛点：
1. **全局快照 → 单调前缀一致性**（跨分区一致备份状态）
2. **回压限制复制 lag**（可用性不低于同步复制）
3. **Coordinated Apply 分离 WAL 和 KV 应用**（故障恢复即时 + 满性能）
