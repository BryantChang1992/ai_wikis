# 论文概览

- **论文全名**：Rosé: Flexible Replication With Strong Semantics For Partitioned Databases
- **作者**：Ioannis Zarkadas, Kelly Kostopoulou, Thomas Graham, Junfeng Yang, Philip A. Bernstein, Asaf Cidon, Tamer Eldeeb（Columbia University & Microsoft Research）
- **会议**：CIDR 2026, Chaminade, USA
- **页数**：7 页
- **核心洞见**：**利用数据库已有的横跨分区的全局快照机制（epoch/HLC/real-time），将单调前缀一致性（monotonic prefix consistency）自然扩展到分区数据库的异步主备复制中**，通过三件套（bound lag + coordinated apply + 回压）解决异步复制在分区场景的三个核心痛点。

---

## 上下文（Background）

### 为什么这个方向重要？

异步主备复制（Asynchronous Primary-Backup）是实现低写延迟 + 跨地域容灾的主流方案（AWS Aurora、Yugabyte、Azure Storage、PostgreSQL）。但对**分区数据库**（每个 partition 独立复制日志），异步复制有三个痛点：

1. **分区各自的复制进度不同** → 整个备份集群可能处于未定义状态（跨分区事务的写入只到达部分分区的备份）
2. **复制 lag 无上限** → 可能导致大量数据在故障时丢失
3. **故障恢复后性能退化** → 备份上调的部分分区需要清理掉已复制但尚未全局一致的数据

### 现有方案的不足

| 方案 | 延迟 | 一致性 | 故障恢复 | Lag 控制 |
|------|------|--------|----------|----------|
| 同步共识（Raft/Paxos） | 高（等多数派） | 强 | 即时 | N/A |
| 异步主备（朴素） | 低 | 弱/未定义 | 复杂修复 | 无限制 |
| Yugabyte xCluster | 低 | 单调前缀 | 即时（后清理） | 无限制 |
| **Rosé** | 低 | 单调前缀 | 即时 + 满性能 | 回压控制 |

---

## 贡献（Contributions）

1. **将全局快照扩展为分区数据库单调前缀一致性**：利用已有的 epoch/snapshot 机制，备份集群始终暴露一个跨分区一致的全局快照（以最小已复制 epoch 为界）
2. **回压（Backpressure）机制限制复制 lag**：每个 partition 维护一个有界队列（size L），队列满时对该 partition 写进行限流。理论证明可用性不低于同步复制
3. **Coordinated Apply（协调应用）**：将 WAL 复制与 KV store 应用分离，备份分区只在所有分区的一致 epoch 内统一 apply，从而故障恢复仅需 trim WAL（极快），恢复后立即满性能

## 方法（Approach）

### 1. 单调前缀一致性（Partitioned M-P Consistency）

**关键观察**：现代分布式数据库（CockroachDB HLC / Yugabyte HLC / Chablis epochs / Spanner TrueTime）已经支持跨分区的全局一致性快照读。这个基础设施可以直接用来支持分区级的单调前缀一致性。

**协议**：
- 每个 primary partition 按 epoch 递增序将事务 write-set 发送到对应的 backup partition
- backup 集群的 cluster manager 跟踪每个 partition 的最新完整应用 epoch eᵢ
- backup 集群对外暴露的最近快照 epoch = min(eᵢ)
- 随着每个 partition 的进度前进，snapshot epoch 单调递增 → 单调前缀一致性

### 2. 回压限制复制 Lag

**复制 lag 定义**：replication_lagᵢ = e_primary,ᵢ − e_backup,ᵢ

**系统有效 lag** = min(replication_lagᵢ) —— 一个 straggler partition 拖慢整个备份集群的可用快照 epoch。

**回压机制**：
- 每个 partition 维护一个有界队列（size L）
- Push-based：primary 主动推送写操作到 backup
- 当某 partition 队列满 → **仅对该 partition 限流**（不拖累其他 partition）
- 队列大小 L 是关键调参——需要足够大以吸收正常抖动，但足够小以满足用户对数据新鲜度的要求

**可用性证明**（4.2.2）：
> Rosé 的可用性不劣于同步复制。
> 逻辑：初始状态 Rosé 接受同步复制的所有事务。队列满后只有两种情况：
> （1）事务不被同步复制接受 → Rosé 也不接受
> （2）事务被同步复制接受（意味着网络可达）→ 队列有空位

**容灾降级**：如果备份区域长时间不可用，管理员/外部监控系统可手动解除回压，恢复可用性。

### 3. Coordinated Apply（协调应用）

**核心问题**：异步复制中，不同 partition 的备份可能处于不同 epoch。在故障恢复时：
- 最简单方案：删除所有高于目标快照时间的数据 → 膨胀恢复时间
- Yugabyte 方案：在 SST 文件 metadata block 中标记 keep_ts，读时跳过，后台 compaction 清理 → **恢复后性能退化**（22% 吞吐下降，15% L99 延迟上升）

**Rosé 方案**：分离 WAL 复制和 KV store 应用
- 备份集群跟踪 min(replicated_epoch) 作为 snapshot epoch
- 只有当某 partition 的所有写操作在 snapshot epoch 内时，才通知该 partition 从 WAL apply 到 KV store
- 故障恢复时只需 trim WAL（极快操作），KV store 中无脏数据 → 恢复后满性能

**存储引擎集成**：Rosé 不做具体的 KV 存储设计——只描述协议层。文中以 Chablis 作为集成宿主。

### 协调套件的 Dead Time 分析

不同 partition 的写入速度不同（hot vs cold partition），hot partition 复制时间长，cold partition 复制时间短。若等慢的传输完才一起 apply，会产生 dead time。

**dead_timeᵢ = max(RT) − RTᵢ**，其中 RT 是复制时间。

**缓解策略**：
- Rosé 使用短 epoch（毫秒级）— 这直接可以减少 dead_time 的量级
- 在没有极端 hot partition 的场景下，dead time 接近于零
- **snapshot epoch 的推进不会因 coordinated apply 而变慢**——cold 的 apply 虽然延迟开始，但应能在 hot partition 传完前完成

## 评估（Evaluation）

### 实验设置
- 单机 Cloudlab c6525-25g 模拟多节点 + 网络故障
- Rosé 集成到 Chablis（Columbia 的 geo-distributed KV store）
- 对比 Yugabyte 2.25.2.0-b359

### 关键结果

**Q1: 回压是否有效？**
- 有回压：lag 被有效控制在设定上限内
- 无回压：lag 随不受限增长
- 代价：仅过载 partition 受影响，不扩散

**Q2: 故障恢复性能？**
- 备份节点之一不可达 → Rosé coordinated apply 使剩余节点也不再应用不可达节点之后的 epoch
- 两者都可在 2s 内完成故障切换
- **关键差异**（恢复后）：
  - Yugabyte：**22% 吞吐下降，15% P99 延迟上升**（因为有 keep_ts 标记的 dirty SST 文件需要读取时过滤）
  - Rosé：**0% 退化**（KV store 中无脏数据）

## 批判性思考（Critical Analysis）

### 前提假设合理吗？

1. **"现有数据库都有全局快照"** ✅：论文列举了 HLC (CockroachDB/Yugabyte)、TrueTime (Spanner)、epochs (Chablis/Chardonnay) —— 覆盖面广，假设成立
2. **"partition 完全不可达极罕见"** ✅：R=3 复制因子下，单 partition 不可用概率 ≈ 10⁻⁹ —— 合理假设
3. **"短 epoch (毫秒级) 能最小化 dead time"** ⚠️：对硬件时钟和网络延迟要求较高。极端 hot partition（如社交网络中的热门话题 shard）可能仍然会有显著的 dead time
4. **"只评估了 Yugabyte 对比"** ⚠️：不应只对比一个系统。像 CockroachDB 的 follower read、Spanner 的 read-only replica 等方案也应该比较

### 与现有知识库的关系

1. **[[事务模型深度调研]]**：Rosé 利用全球一致性快照来实现单调前缀——这跟 MVCC 机制天然契合。全局快照本身就是 MVCC 的副产品
2. **[[LSM-Tree]] / [[LSM-Tree-合并优化]]**：Yugabyte 的 failover 性能退化根源在于 LSM 树结构——write-heavy 记录的 SST 文件中有大量旧版本数据，标记 keep_ts 后 compaction 负担巨大。这解释了 Rosé 为何选择分离 WAL 和 KV application
3. **[[分布式数据系统一致性体系综述]]**：Rosé 填补了"异步复制 + 强语义"这个象限——它在同步复制（强一致+高延迟）和淳异步复制（低延迟+弱一致）之间找到了一个新平衡点

### 可借鉴 vs 潜在坑

**可借鉴**：
- ✅ 全局快照 → 单调前缀一致性 这个映射非常简洁优雅
- ✅ WAL/KV 分离思想可以推广到任何有 WAL + 结构化存储的系统
- ✅ 回压仅影响 straggler partition 的设计原则（Struggler Isolation）
- ✅ 协调 apply 消除故障恢复后的性能退化

**潜在坑**：
- ⚠️ 回压参数 L 的调优不是自动的——不同 workload 需要不同的 L
- ⚠️ 备份区域长时间不可达时需要人工介入解除回压 → 不是全自动
- ⚠️ 只做了单机模拟 + 对比仅 Yugabyte → 生产规模验证缺位
- ⚠️ 论文是 CIDR（vision/experience track），7 页短论文，设计细节不够完整。协议的形式化验证和安全证明缺失
