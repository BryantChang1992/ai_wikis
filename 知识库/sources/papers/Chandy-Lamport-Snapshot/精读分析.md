# Chandy-Lamport 分布式快照算法 精读分析

## 一句话总结

Chandy-Lamport 算法是**首个分布式系统全局状态快照算法**，通过进程状态记录 + 通道状态记录的协同，在不停止系统运行的情况下捕获一个一致的全局状态，是 Flink Checkpoint 机制的理论基石。

## 论文背景

- **问题定义**：分布式系统中没有全局时钟，如何在系统持续运行期间记录一个"一致"的全局状态（global state）？
- **历史地位**：这是分布式快照问题的开山论文，1985 年发表，2013 年获 ACM SIGOPS Hall of Fame Award，被引用超过 5000 次。

## 核心贡献

### 1. 系统模型

- 分布式系统建模为有向图 G = (V, E)
- V = 进程集合（processes），E = 单向通道集合（channels）
- 每个进程有自己的状态（state），每条通道有正在传输的消息序列
- **全局状态** = 所有进程状态 + 所有通道状态

### 2. 快照算法的关键性质

- **一致性**（Consistency）：记录的快照是一个"可能发生"的全局状态——存在某个合法的执行序列使系统确实经过该状态
- **一致性切割**（Consistent Cut）：快照对应一个一致性的切割，对于任意进程 p 和 q，如果 p 在快照中记录了收到来自 q 的消息 m，则 q 在快照中也记录了发送 m 的事件

### 3. 算法步骤（两阶段）

**Phase 1: 启动（Initiator）**

1. 任意进程 p 启动快照
2. p 记录自己的本地状态
3. p 在记录完状态后，向每个出边通道发送一个 **marker（标记）**消息
4. p 开始记录每个入边通道上到达的消息（通道状态）

**Phase 2: 传播（Non-Initiator）**

当进程 q 首次从通道 c 收到 marker：
1. q 记录自己的本地状态
2. q 将通道 c 的状态记录为空（因为 marker 之前没有消息）
3. q 向每个出边通道发送 marker
4. q 开始记录所有其他入边通道上的消息（直到收到对应通道的 marker）

当进程 q 后续从其他通道 c' 收到 marker：
- q 停止记录通道 c'，并将期间接收到的消息序列作为 c' 的状态

### 4. 算法终止条件

当所有进程都记录了状态，且所有通道的状态都被记录了，快照完成。

### 5. 稳定属性检测（Stable Property Detection）

论文证明了快照算法可以用于检测**稳定属性**（Stable Property）——一旦为真就永远为真的属性，例如：
- 死锁检测
- 终止检测（termination detection）
- 资源泄漏

## 关键洞察

1. **Marker 是核心创新**：marker 消息充当逻辑时钟，将全局状态切割点传播到所有进程
2. **通道状态记录**：通过"记录 marker 之前收到但尚未处理的消息"来捕捉 in-flight 消息，这是保证一致性的关键
3. **首次收到 marker 的语义**：首次收到 marker 意味着该进程位于"切割边界"上，此时必须快照状态
4. **FIFO 通道假设**：原始算法的正确性依赖 FIFO 通道。后续扩展（如 Lai-Yang 算法）放宽了这一假设

## 与 Flink 的关系

Flink 的 Checkpoint 机制是 Chandy-Lamport 算法的**工程化实现**：

| Chandy-Lamport | Flink Checkpoint |
|---|---|
| Marker 消息 | Checkpoint Barrier |
| 进程状态快照 | Task State Snapshot（StateBackend）|
| 通道状态 | In-flight records 在 Channel 确认后持久化 |
| 稳定属性 | Exactly-Once 语义下的恢复点 |
| 多快照并发 | 分布式快照 + Checkpoint ID 对齐 |

## 影响力

- **2013 SIGOPS Hall of Fame Award**
- 被 Apache Flink、Apache Storm、Google Dataflow/MillWheel 等流处理系统广泛采用
- 启发了分布式系统中的 checkpoint/restart、debugging、distributed garbage collection 等领域

## 论文局限性

1. 依赖 FIFO 通道（通过加序号可以缓解）
2. 快照期间的 in-flight 消息可能导致快照较大
3. marker 传播开销在大型系统中不可忽略
4. 不考虑进程崩溃场景（后续工作 "Distributed Snapshots in Spite of Failures" 解决了这个问题）

## 关键引用

> "The global state recorded by the algorithm is a possible global state of the system during the computation."

这一定理是整个分布式快照理论的基石——证明了一致性切割总是存在且可达。
