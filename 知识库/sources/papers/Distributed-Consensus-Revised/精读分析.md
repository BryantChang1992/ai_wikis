## 1. 一句话总结

## 2. 核心贡献（4 层递进泛化）

### Layer 1: Quorum Intersection Revised（Ch 4）
- **Classic Paxos** 要求：所有 Quorum 两两相交（∀Q, Q' ∈ Q: Q ∩ Q' ≠ ∅）
- **Revision A (Flexible Paxos)**：Phase-1 Quorums (Q₁) 只需和 Phase-2 Quorums (Q₂) 相交，Q₁ 内部、Q₂ 内部不需要相交
  - 例如：4 个 acceptor，Q₁ = {{a₁,a₂}, {a₃,a₄}}，Q₂ = {{a₁,a₃}, {a₂,a₄}}
- **Revision B**：Phase-1 Quorum 只需和**前一 epoch** 的 Phase-2 Quorum 相交，无需和后续 epoch 相交
  - epoch e 的 Q₁ 只需 intersect epoch < e 的 Q₂
  - 推论：最小 epoch e_min 的 proposer 可以跳过 Phase-1，直接进入 Phase-2

### Layer 2: Promises Revised — Quorum Transitivity（Ch 5）
- 如果一个 proposer 收到了 promise(e, v)，那么它**不需要**再和其他更小 epoch 的 phase-2 quorum 做 intersection
- quorum intersection 具有**传递性**：收到 (e, v) 就能覆盖所有 ≤e 的 epoch
- 实现效果：phase-1 可以比 quorum requirement 更早完成 → 减少等待

### Layer 3: Value Selection Revised（Ch 6）
- Classic Paxos 的 value selection：选最高 epoch 关联的 value → **保守**（可能被 low epoch 的 candidate value）
- **Quorum-based Value Selection**：按 quorum 粒度判断哪些 value 可能已被 commit
  - 如果某 quorum 的所有 acceptor 都返回 nil → 该 quorum 未 commit
  - 如果同一 quorum 内有两个不同 epoch 的不同 value → 该 quorum 未 commit
  - 只有当所有 quorum 都指向同一 value → 必须 propose 该 value
- 结果：prop 可以有更多机会 propose **自己的 candidate value**，而非被迫选先前 proposal

### Layer 4: Epochs Revised（Ch 7）
- 传统 epoch 分配：Unique to proposer（pre-allocated）或 Allocated（allocator 分配）
- **Epochs by Recovery**：允许任意 proposer 使用任意 epoch，不强求 epoch 全局唯一
  - 多个 proposer 提出**相同 value** → 不会 dueling，可以一轮达成
  - 泛化 Fast Paxos：弱化 Fast Paxos 的 quorum intersection 要求
- 附加方案：
  - Epochs by Value Mapping：value 映射到 epoch
  - Epochs by Allocator：单点分配 minimum epoch
  - Multi-path Paxos：Hybrid 混合方案

**贡献汇总**：
```
Classic Paxos Q1 ∩ Q2 all-to-all
  → Revision A: Q1 ∩ Q2 across phases only     [Flexible Paxos/FPaxos]
  → Revision B: Q1_e ∩ Q2_f where f < e        [Per-epoch weakening]
  → Ch 5: Transitive quorum reuse               [Received (e,v) → covers ≤e]
  → Ch 6: Quorum-based value selection          [Escape from "highest epoch" rule]
  → Ch 7: Epochs by recovery + Multi-path       [Generalised Fast Paxos]
```

## 5. 方法论评估

- **问题定位精准**：博士论文第一章就明确 Paxos 的局限不是共识问题本身的局限，而是 Paxos 算法设计的局限
- **递进结构清晰**：4 个维度递进式泛化，每层都保持 proof of safety 的完整
- **工程导向**：Ch 4.3 的 "Implications" 一节直接给出 Multi-Paxos 优化方案（even nodes、phase-2 quorum 减半），论文不仅是理论也有直接工程启示
- **关联广度**：引用并泛化 Paxos、Fast Paxos、Flexible Paxos、Mencius、Multi-Ring Paxos 等 15+ 变种

---

## 6. 局限性

- **不涉及 Byzantine Fault Tolerance**（作者在 §1.6 明确排除）
- **所有 safety proofs 是数学演绎，未包含 formal verification（TLA+/PlusCal）内的完整实现**
- **Epochs by Recovery 的工程可行性尚未在大型生产系统验证**（相比 Raft 被 etcd/TiKV 广泛验证）
- **thesis 提出的 Multi-path Paxos 是概念级原型，缺少性能 benchmark**

## 7. 后续研究方向

基于 Howard 的广义共识框架，以下方向值得追踪：
1. **Generalised Solution 在 Geo-distributed Consensus（WAN 跨地域）中的应用**：FPaxos 已有 WPaxos/DPaxos 的 follow-up，是否可进一步在跨数据中心部署中使用 Epochs by Recovery
2. 与 Raft 的对比：Howard 方案能否简化为类似 Raft 的 leader-based 模型，保留 flexible quorum 的好处？
3. 与 CRDT/Eventual Consistency 的桥接：广义共识框架能否为强一致性 + 最终一致性之间的 trade-off 提供 formal 指导？
