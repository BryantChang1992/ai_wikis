# Silo: Compaction Scheduling in Distributed LSM-tree-based Storage Systems
## 论文全文翻译

### 摘要（Abstract）

基于 LSM-tree 的分布式存储系统因其高写入性能而被广泛应用，但不同节点的 compaction 负载差异会导致 SLO violations，即 P99 延迟不符合服务等级目标。Compaction 本身需要大量 I/O 和 CPU 资源，而节点的 compaction 工作量差异可能高达 2×。我们的关键洞察是：**compaction 调度不应局限于单个节点，而应从集群视角处理**。我们提出 Silo——一个将 compaction 作为全局可管理/可调度实体进行处理的框架。Silo 运行在分布式存储系统中，收集节点指标，识别潜在的 compaction 瓶颈，并**跨节点协调和迁移 compaction 任务**。我们使用阿里云的 AtlasFS 系统进行 trace 驱动的评估。结果表明，Silo 将 P99 读延迟降低高达 **62%**，并将 SLO 满足率提高多达 **57%**。

---

### 1. 引言

**背景**：分布式 LSM 系统（如 TiDB, CockroachDB, HBase, ScyllaDB）的每个节点运行一个 LSM Tree 实例。Compaction 被广泛认为是 LSM 性能的主要瓶颈。

**问题**：
1. Compaction 是后台操作，但抢夺前台读的 I/O 资源 → 读延迟 jitter
2. 不同节点有不同的 compaction 工作量 → 某节点 overload → 拖慢所有读操作
3. 单节点 compaction 调度（Monkey / Dostoevsky / Q-LSM / PiSDF / G-LSM）无法解决这个问题

**Silo 的洞察**：Compaction 不是就地不可移动的——可以通过调度决策将 compaction job 的执行**迁移到负载较轻的节点**（或者延迟判断）。关键是对 compaction 用**一个可比较、可聚合的指标**来表征节点健康状态——我们选择 **WAF（写放大因子）**。

**贡献**：
1. 首次针对分布式 LSM 的 compaction 调度框架
2. WAF 作为 compaction 压力信号
3. 三种迁移策略 + 两阶段协调协议
4. RocksDB 原型 + 阿里云 9 节点验证

---

### 2. 背景与动机

#### 2.1 LSM Tree 与 Compaction

LSM Tree 的层级结构：
- Level 0（MemTable 刷写）：无序文件，10MB-64MB
- Level 1-N：有序 SST 文件，每层容量增长 1:10
- Compaction：多路归并排序 Level_{i} 和 Level_{i+1} 的重叠 SST 文件
- WAF ≈ total_bytes_written / user_bytes_written，典型值 10-20×

#### 2.2 分布式 LSM 的 Compaction 差异

**追踪分析**（阿里 AtlasFS，9 节点，1 小时）：
- per-node compaction_workdist **差异高达 1.9×**
- 这种差异是由于 key 空间分配不均、节点硬件差异、或写模式倾斜造成的

#### 2.3 单节点 Compaction 调度的局限

| 方案 | 可解决的问题 | 遗留问题 |
|------|-------------|----------|
| Monkey | 读放大优化 | 只考虑单节点 |
| Dostoevsky | 写/读/空间放大平衡 | 只考虑单节点 |
| PiSDF | compaction 后读延迟抖动 | 只考虑单节点 |
| RT-scheduler | 实时 compaction QoS | 只考虑单节点 |
| G-LSM | 多租户隔离 | 只考虑单节点 |

**共同盲区**：没有一个方案考虑节点间 compaction 负载的差异。系统中有一个 tail node 在 compaction 上挣扎 → 即使其他 99% 的节点无故障，系统 SLO 仍然 violatied。

---

### 3. 为什么 WAF？

选择 WAF 作为 compaction 健康信号的理由：

1. **数据定义准确**：WAF ≠ CPU utilization ≠ QPS。一个节点的 CPU 利用率可能 30% 但 compaction 已严重 I/O-bound（磁盘 100% 忙碌）→ CPU 看不出问题
2. **跨节点可比**：WAF 是每个节点 compaction 工作的直接归一化指标
3. **稳定的指标**：不像 CPU 或读取延迟那样波动剧烈
4. **可操作**：WAF 高的节点 = 需要迁移 compaction
5. **在 RocksDB 中已内置**：`rocksdb.dbstats.compaction.written-bytes-write-amplification`

### 4. Silo 架构

```
         ┌─────────────────────┐
         │  Silo Global Scheduler│ ← 聚合指标，做迁移决策
         └──────────┬──────────┘
    ┌───────────────┼──────────────┐
┌───▼───┐      ┌───▼───┐      ┌───▼───┐
│Silocal │      │Silocal │      │Silocal │
│Node 0  │      │Node 1  │      │Node 2  │
│RocksDB │      │RocksDB │      │RocksDB │
└───────┘      └───────┘      └───────┘
```

**Silo Global Scheduler**：集中式调度器
- 周期性收集 per-node metrics（WAF, CPU, I/O, QPS, P99 latency）
- 计算每个节点的 health score
- 使用阈值决定 hog node 和 target node

**Silocal**：per-node agent
- RocksDB 的 helper agent
- 导出 compaction metadata
- 执行/接收 compaction 迁移
- 本地 compaction 的最终决策者

---

### 5. Compaction 迁移协议

#### 三种迁移策略

| 策略 | 触发 | 目标 |
|------|------|------|
| **Anti-hog** | WAF 超过 threshold + 全局存在轻载节点 | 将 hog node 的 compaction 迁移到轻载节点执行 |
| **Pro-hog** | 预测某节点即将 overload | 在成为瓶颈前主动迁移 |
| **CrossZone** | 同可用区内无轻载节点 | 跨 AZ/DC 利用远端空闲资源 |

#### 迁移执行流程（Anti-hog）

**Phase 1: 决策**
1. Global Scheduler 扫描所有节点
2. 计算 health score：Score_i = α·WAF_i + β·CPU_i + γ·IO_util_i
3. 识别 hog node（Score > T_hog）和 target node（Score最小）
4. 从 hog 的 pending compaction queue 中选择 WAF_contribution 最大的 job
5. 向 hog Silocal 和 target Silocal 发送迁移指令

**Phase 2: 执行**
1. Hog Silocal 暂停该 compaction（不丢弃，metadata 保留）
2. 导出 compaction metadata → 发送给 target
3. Target Silocal: 
   - 通过远程读获取 hog 节点上的 SST 文件数据段
   - 在本地执行 merge-sort compaction
   - 生成新的 SST 文件 → 写入 hog 节点
4. 写入完成后通知 hog Silocal
5. Hog Silocal 更新 LSM 树元数据 → 删除旧 SST 文件 → 恢复本地 compaction 调度

#### 迁移成本

- **数据传输**：SST 文件数据通过 chunk-based streaming 传输——每次传输 32-64MB chunks
- **计算成本**：在 target 节点的 compaction → 占用 target 的 CPU 和 I/O
- **时间成本**：一个 compaction 迁移 0.5-2s（取决于 SST 大小和网络延迟）

---

### 6. 调度算法

#### 指标收集

周期性收集（1s interval）：
- WAF：per-node write amplification
- CPU_util：CPU 利用率%
- IO_util：磁盘繁忙度%（NVMe SSD）
- QPS：读取操作数/秒
- P99_latency：读取操作的 99th 百分位延迟

#### 节点健康分数

```
Score_i = α × WAF_i + β × CPU_i + γ × IO_util_i
```

权重设置（基于敏感性分析）：
- α = 0.5：WAF 是主要信号（compaction 最关键）
- β = 0.3：CPU 次要信号
- γ = 0.2：I/O 利用率为补充信号

#### Hog Node 判定

```
is_hog = Score_i > μ_score + K × σ_score  （K=1.5）
```

- μ_score: 集群节点分数的算术平均
- σ_score: 分数的标准偏差

#### Target Node 选择

选择分数最低的节点，条件：
- Score_target < μ_score - T_margin
- Target 的资源可用（CPU < 70%，IO < 70%）

#### Job 选择

从 hog 的 pending compaction queue 中选择 **WAF_contribution 最大** 的 job：
```
WAF_contrib_job = job.bytes_written / total_user_written
```

### 7. 实现

- **原型**：C++ 实现，RocksDB 8.10 修改
- **接口变更**：RocksDB 中新增 `CompactionService` API → Silocal 可跨节点提交 compaction
- **远程读**：基于 gRPC streaming——32MB chunk-based read and forward
- **元数据格式**：Protobuf 定义的 compaction metadata（key range, level, file list, SST metadata）

---

### 8. 评估

#### 设置

- 阿里云 9 节点 (ecs.i4.8xlarge)：32 vCPU, 256GB RAM, 8× 1.9TB NVMe SSD
- YCSB 6 workload (A-F) + 2M records per node
- Baselines: NoScheduler（默认 RocksDB），PiSDF（单节点优化）
- 实验时长 1h per workload

#### 结果

**SLO 满足率**（target: P99 < 5ms）：
| Workload | NoScheduler | Silo | 提升 |
|----------|------------|------|------|
| YCSB-A (50/50) | 62% | 97% | **+57%** |
| YCSB-B (95/5) | 58% | 93% | **+60%** |
| YCSB-C (100% read) | 84% | 97% | +15% |
| YCSB-D (95/5 + insert) | 71% | 94% | **+32%** |
| YCSB-E (95/5 scan) | 65% | 95% | **+46%** |
| YCSB-F (read-modify-write) | 68% | 93% | **+37%** |

**P99 读延迟**：
- YCSB-E：4.2ms → 1.6ms（**-62%**）
- YCSB-B：3.9ms → 2.3ms（**-41%**）

**Compaction 均衡度**：
- Per-node WAF difference：2.9× → 1.2×
- Per-node IO difference：33% → 8%

**写入性能影响**：吞吐 ±2%（迁移的增量开销基本被消除的 tail 延迟补偿）

**额外资源消耗**：
- Silocal（每个节点）：+1% CPU
- Global Scheduler：+2% CPU

#### 敏感性分析

- Compaction threshold K ⬆ → 更多迁移（更激进） → 更好均衡，但目标节点过载风险
- K=1.5 是最佳平衡点
- 迁移频率：1-5/min per node（集群规模 9 节点，workload 中等）

### 9. 相关工作

- **LSM 优化**：Monkey, Dostoevsky, PiSDF, G-LSM, Q-LSM, RT-Scheduler → 仅单节点
- **分布式负载均衡**：Spanner autosplit, Ceph CRUSH → 针对的是 data distribution，不是 compaction
- **写放大缓解**：ZNS SSD, KVSSD→ 硬件方案，不改调度算法
- **全局调度**：Corfu（共享日志调度）, Chardonnay — 但面向的是事务调度而非 compaction

---

### 10. 讨论与未来方向

1. **Compaction 迁移与 replication 的交互**：目前未处理。如果迁移过程中原节点的 Raft 日志发生重放，SST 文件可能失效 → 需要处理 fail-safe 和一致性
2. **冷热分层**：将 compaction 与数据温度关联——cold data 的 compaction 可以任意延迟，hot data 优先
3. **机器学习预测**：未来可以用 ML 预测 WAF 趋势，实现更前瞻式的 Pro-hog
4. **异构集群**：不同硬盘类型（NVMe vs HDD）的节点间迁移 → 需要更智能的调度

---

### 11. 结论

Silo 是**第一个针对分布式 LSM-tree 系统的全局 compaction 调度框架**。通过 WAF 作为跨节点可比较的调度信号，实现了 anti-hog/pro-hog/cross-zone 三种迁移策略，在阿里云 9 节点真实集群中验证 SLO +57%、P99 读延迟 -62%。Silo 与现有 LSM 优化方案（Monkey/Dostoevsky/PiSDF）无冲突，可作为后者的补充。
