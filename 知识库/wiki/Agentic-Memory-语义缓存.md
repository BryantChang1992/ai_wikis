# Agentic Memory — Agent-First 语义缓存

## 一句话总结

Agent-First 数据系统中的全局可查询语义缓存：存储历史探针结果、元数据标注、数据语义信息，通过向量索引支持语义检索，让 agent 的探索性查询能直接命中已有知识而非每次从头 SQL。

## 为什么需要

Agent 在探索阶段（占总查询的早期大部分）大量查询**相同的元数据**和**相似的数据模式**。如果这些知识可以被缓存和语义检索，agent 可以跳过大量冗余查询。

**例子**：
- 上一个 agent 已经探索了"销售相关的表有哪些"
- 新 agent 做"咖啡豆销售分析"时，应该直接读取上一个 agent 的 exploration 结果，而不是重新 query `information_schema` + 做语义推断

## 存储内容

| 类别 | 内容 | 例子 |
|------|------|------|
| **探针结果** | 历史 peer agent 的 SQL 结果 | "哪些表存了销售数据"的答案 |
| **列级元数据** | 编码格式、缺失值、时间/位置粒度 | `state` 列存的是全称不是缩写 |
| **数据语义标注** | 对数据含义的自然语言描述 | "order_date 是下单时间非发货时间" |
| **探针关联** | 哪些探针相似、各自的效果 | agent 先前做的 trend query syntax |

## 两种检索方式

1. **向量语义搜索**：输入新探针的 brief → 检索语义相似的先前探针 → 复用结果或执行建议
2. **结构化精确查询**：按 table/column/探针ID 精确查找

## 更新策略

**两难**：
- **同步更新**（schema 变更时立即刷新缓存）→ 开销大
- **异步/惰性更新**（新探针发现陈旧时纠正）→ 陈旧信息可能误导 agent

**论文倾向**：暂时接受不一致，由新探针触发更新，但识别到风险（新增表会漏掉）。

## 隐私挑战

Agent 代表不同用户 → 相似问题的答案共享提升效率但可能泄漏隐私：
- Agent A 问"员工可用性数据在哪"（可能是找谁要离职）
- Agent B 问同样问题 → 共享结果可能向 B 暴露 A 的意图
- 需要借鉴 CryptDB 等安全查询处理的思路，结合细粒度访问控制

## 局限

- 纯愿景，无实现路径
- 陈旧更新问题的解决策略未定义
- 隐私防护具体机制未提出
