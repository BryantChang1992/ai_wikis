# 05b — 客户端写入流程深度分析

> **定位**：本文聚焦 **Fluss 纯客户端（Java Client）写入全链路**，从 `ConnectionFactory` 创建连接到消息收到 Servier Ack 的完整生命周期。
> **与 05 的区别**：[[Fluss源码分析/05-客户端与计算集成|05]] 以架构概览和 Flink/Spark 计算集成视角为主；本文深入客户端内部的写入引擎管线，逐文件分析。
> **源码版本**：Fluss trunk，模块 `fluss-client`。

---

## 1. 写入 Demo：最小化 E2E 示例

从单测 `FlussTableITCase.testAppendOnly()` 提取出的最小写入代码：

```java
// 1. 创建连接
Configuration conf = new Configuration();
conf.setString("bootstrap.servers", "localhost:9092");
Connection conn = ConnectionFactory.createConnection(conf);

// 2. 获取 Table 句柄
TablePath tablePath = TablePath.of("test_db_1", "test_table_1");
Table table = conn.getTable(tablePath);

// 3. 创建 AppendWriter + 写入
AppendWriter writer = table.newAppend().createWriter();
writer.append(GenericRow.of(1, BinaryString.fromString("hello")));
writer.flush();  // 阻塞直到发送完成

// 4. 释放资源
table.close();
conn.close();
```

**三个关键点**：
- `Connection` 是重量级（long-lived），整个应用只创建一个
- `Table` 和 `AppendWriter` 是轻量级（per-thread），不需缓存
- `append()` 异步返回 `CompletableFuture<AppendResult>`，在 `flush()` 时阻塞等待

## 2. 客户端生命周期

### 2.1 创建：`ConnectionFactory → FlussConnection → Table → Writer`

```
ConnectionFactory.createConnection(conf)
  └─ new FlussConnection(conf)
       ├─ FileSystem.initialize(conf)              // 初始化文件系统（S3/HDFS/本地）
       ├─ setupClientMetricsConfiguration()         // 配置指标（默认 JMX）
       ├─ new ClientMetricGroup(registry, clientId) // 创建指标组
       ├─ RpcClient.create(conf, clientMetricGroup) // 创建 Netty RPC 客户端
       └─ new MetadataUpdater(conf, rpcClient)     // 创建元数据更新器
            └─ initializeCluster(conf, rpcClient)
                 ├─ parseAndValidateAddresses(bootstrap.servers)
                 ├─ 逐个 bootstrap 地址尝试
                 ├─ 若是单地址: tryToInitializeClusterWithRetries (指数退避, 重试3次)
                 └─ 通过 AdminReadOnlyGateway 获取 Coordinator 地址 + TabletServer 列表
```

**`Connection.getTable(tablePath)`（创建 Table）**：
```java
// FlussConnection.getTable():
metadataUpdater.updateTableOrPartitionMetadata(tablePath, null);  // 强制刷新元数据
Admin admin = getOrCreateAdmin();
return new FlussTable(this, tablePath, admin.getTableInfo(tablePath).join());
//                      ↑              ↑               ↑
//                    连接引用      表路径         从 Server 拉取 TableInfo（schema/bucket/format）
```

**`table.newAppend().createWriter()`（创建 Writer）**：
```java
// Builder 模式:
table.newAppend()         // 返回 AppendBuilder
     .createWriter()      // new AppendWriterImpl(tablePath, tableInfo, writerClient)
                          // 其中 writerClient = conn.getOrCreateWriterClient()
```

**`conn.getOrCreateWriterClient()` 即 WriterClient 初始化**：
```java
new WriterClient(conf, metadataUpdater, clientMetricGroup, admin)
  ├─ buildIdempotenceManager()     // 幂等写入管理器（如果 enable.idempotence=true）
  │    └─ new IdempotenceManager(idempotenceEnabled, maxInflightRequestsPerBucket,
  │                              coordinatorGateway, metadataUpdater)
  ├─ configureAcks() / configureRetries()  // 幂等模式强制 acks=all, retries>0
  ├─ new RecordAccumulator(conf, idempotenceManager, writerMetricGroup, clock)
  │    ├─ LazyMemorySegmentPool.createWriterBufferPool(conf)  // 缓冲池（Arrow 内存段）
  │    ├─ BufferAllocator (Arrow shaded allocator)             // Arrow 内存分配器
  │    └─ ArrowWriterPool                                      // Arrow Writer 复用池
  ├─ new Sender(accumulator, maxRequestTimeoutMs, maxRequestSize, acks, retries,
  │             metadataUpdater, idempotenceManager, writerMetricGroup)
  └─ ioThreadPool = Executors.newFixedThreadPool(1, "fluss-write-sender")
       └─ ioThreadPool.submit(sender)  // 启动单线程 Sender 后台循环
```

### 2.2 运行：Sender 主循环

`Sender.run()` 是唯一的后台线程，循环执行 `runOnce()`：

```
while (running):
  runOnce():
    1. 幂等写入需确保 writerId 已初始化（await InitWriter RPC）
    2. sendWriteData():
       a. metadataUpdater.getCluster() 获取集群拓扑快照
       b. accumulator.ready(cluster) 判断哪些 node 的 batch 可发送
          - 条件: batch 满 OR linger.ms 超时 OR 缓冲耗尽 OR 已关闭
       c. 若 unknownLeaderTables 非空 → metadataUpdater.updatePhysicalTableMetadata()
       d. accumulator.drain(cluster, readyNodes, maxRequestSize)
          - 从 WriteBatch 队列取出就绪 batch，按目标 TabletServer 分组
       e. sendWriteRequests(batches):
          - Log 表: TabletServerGateway.produceLog(ProduceLogRequest)  // Arrow/Indexed/Compacted
          - KV 表 : TabletServerGateway.putKv(PutKvRequest)            // Indexed-KV/Compacted-KV
          - 响应处理: completeBatch() / failBatch()
       f. updateWriterMetrics()  // recordsSendTotal, bytesSendTotal, recordPerBatch 等
    3. 异常: 记录日志，继续循环

优雅关闭循环:
while (!forceClose && accumulator.hasUnDrained()):
  runOnce()  // 继续消费尚未 drain 的 batch
```

### 2.3 回收

```java
// table.close() → FlussTable.close() (空实现，不释放 Writer)
// writer 的显式调用:
writer.flush();  // 等待所有 pending 写入完成

// conn.close():
writerClient.close(Duration)  // 先关 WriterClient
  ├─ sender.initiateClose()   // running = false, accumulator.close()
  ├─ ioThreadPool.shutdown()  // 等待 Sender 线程优雅退出
  │    └─ 若超时 → shutdownNow()
  └─ sender.forceClose()      // 强制关闭
lookupClient.close(Duration)  // 关 LookupClient
remoteFileDownloader.close()  // 关远程文件下载
securityTokenManager.stop()   // 关安全令牌管理器
clientMetricGroup.close()     // 关指标组
rpcClient.close()             // 关 Netty 连接池
metricRegistry.closeAsync()   // 关指标注册中心
```

**WriterClient.close() 的优雅关闭逻辑**：
- `sender.initiateClose()` 先设置 `running=false`，触发 Sender 主循环退出
- Sender 在退出后执行 drain loop（处理剩余未发送 batch），然后 `destroyResources()`
- 再关闭 `ioThreadPool`，给 `timeout` 时间等 Sender 线程结束
- 超时后 `shutdownNow()` 强行终止

## 3. 消息写入生命周期

以 `append() → Server Ack → Future 完成` 为主线，逐层分析。

### 3.1 API 层：`AppendWriterImpl.append(row)`

```java
public CompletableFuture<AppendResult> append(InternalRow row) {
    checkFieldCount(row);                               // 字段数校验
    PhysicalTablePath physicalPath = getPhysicalPath(row); // 分区路由
    byte[] bucketKey = bucketKeyEncoder.encodeKey(row);    // bucket key 编码（有 key 时）

    // 根据表 LogFormat 构造 WriteRecord
    WriteRecord record;
    if (logFormat == LogFormat.INDEXED)
        record = WriteRecord.forIndexedAppend(tableInfo, physicalPath, indexedRow, bucketKey);
    else if (logFormat == LogFormat.COMPACTED)
        record = WriteRecord.forCompactedAppend(tableInfo, physicalPath, compactedRow, bucketKey);
    else // ARROW
        record = WriteRecord.forArrowAppend(tableInfo, physicalPath, row, bucketKey);

    return send(record).thenApply(ignored -> APPEND_SUCCESS);
}
```

**5 种写入格式**（`WriteFormat` 枚举）：
| 格式 | isLog | 请求类型 | 说明 |
|------|-------|---------|------|
| `ARROW_LOG` | ✅ | ProduceLog | Arrow 列式日志（默认） |
| `INDEXED_LOG` | ✅ | ProduceLog | IndexedRow 行式日志 |
| `COMPACTED_LOG` | ✅ | ProduceLog | CompactedRow 行式日志 |
| `INDEXED_KV` | ❌ | PutKv | KV 表 IndexedRow |
| `COMPACTED_KV` | ❌ | PutKv | KV 表 CompactedRow |

其中 Log 表走 `ProduceLog` RPC，KV 表走 `PutKv` RPC。

### 3.2 基类层：`AbstractTableWriter.send(record)`

```java
protected CompletableFuture<Void> send(WriteRecord record) {
    CompletableFuture<Void> future = new CompletableFuture<>();
    writerClient.send(record, (bucket, offset, exception) -> {
        if (exception == null) future.complete(null);
        else future.completeExceptionally(exception);
    });
    return future;
}
```

将 `WriteRecord` 交给 `WriterClient`，并用 `WriteCallback` 桥接到 `CompletableFuture`。

### 3.3 引擎层：`WriterClient.doSend(record, callback)`

```
doSend(WriteRecord, WriteCallback):
  │
  ├─ throwIfWriterClosed()  // sender.isRunning()?
  │
  ├─ dynamicPartitionCreator.checkAndCreatePartitionAsync()
  │     └─ 若分区不存在 → admin.createPartition() + inflight 去重
  │
  ├─ bucketAssigner = bucketAssignerMap.computeIfAbsent(physicalTablePath, ...)
  │     ├─ 有 bucket key: new HashBucketAssigner(bucketNumber, bucketingFunction)
  │     └─ 无 bucket key: new RoundRobinBucketAssigner() 或 StickyBucketAssigner()
  │
  ├─ bucketId = bucketAssigner.assignBucket(record.getBucketKey(), cluster)
  │
  ├─ result = accumulator.append(record, callback, cluster, bucketId, abortIfBatchFull)
  │     ├─ 若 result.abortRecordForNewBatch → 重新 assign bucket + 重试 append
  │     └─ 若 result.batchIsFull || result.newBatchCreated → 唤醒 Sender
  │
  └─ return (异步, 立即返回)
```

**两种 BucketAssigner**：
| 策略 | 有 Key | 无 Key |
|------|--------|--------|
| `HashBucketAssigner` | 对 `bucketKey` 做哈希取模 | N/A |
| `RoundRobinBucketAssigner` | N/A | `AtomicInteger` 轮询 |
| `StickyBucketAssigner` | N/A | 固定一个 bucket 直到 batch 满 |

### 3.4 缓冲层：`RecordAccumulator.append()`

```
RecordAccumulator.append(WriteRecord, callback, cluster, bucketId, abortIfBatchFull):
  │
  ├─ partitionIdOpt = cluster.getPartitionId(physicalTablePath)
  ├─ bucketAndWriteBatches = writeBatches.computeIfAbsent(physicalTablePath, ...)
  │     // CopyOnWriteMap: TablePath → BucketAndWriteBatches
  │     //                               ├─ partitionId
  │     //                               └─ batches: Map<bucketId, Deque<WriteBatch>>
  │
  ├─ 同步块 (dq) 内 tryAppend(writeRecord, callback, dq)
  │     └─ 取尾部 WriteBatch 尝试 tryAppend(), 成功返回
  │
  ├─ 若 tryAppend 失败（batch 满或不匹配）:
  │     ├─ 若 abortIfBatchFull → 返回 abortRecordForNewBatch=true
  │     │     // 由 WriterClient 重新 assign bucket 后重试
  │     └─ 否则 allocateMemorySegments() → appendNewBatch()
  │
  └─ 内存管理: writerBufferPool (LazyMemorySegmentPool, 按需分配)
       └─ unused segments 在 finally 中 returnAll()
```

**内存分配**（`allocateMemorySegments`）：
- Log batch（Arrow）：从 `arrowWriterPool` 获取 `ArrowWriter`，利用 `BufferAllocator` 管理 Arrow 内存
- 非 Arrow batch：从 `writerBufferPool` 获取 `MemorySegment` 页（默认 page size 由 `client.writer.buffer.page.size` 控制）

### 3.5 发送层：`Sender.sendWriteData()`

```
sendWriteData():
  cluster = metadataUpdater.getCluster()              // 获取集群快照

  readyCheckResult = accumulator.ready(cluster)       // 扫描所有 (table, bucket) 的 batch
    ├─ readyNodes: 就绪的 TabletServer node ID 集合
    ├─ nextReadyCheckDelayMs: batchTimeoutMs（默认 linger.ms）
    └─ unknownLeaderTables: 未知 leader 的表（触发 metadata 更新）

  if (unknownLeaderTables 非空):
    metadataUpdater.updatePhysicalTableMetadata(unknownLeaderTables)

  if (readyNodes 为空):
    Thread.sleep(readyCheckResult.nextReadyCheckDelayMs)  // 避免忙等
    return

  batches = accumulator.drain(cluster, readyNodes, maxRequestSize)
    // 按 node 分组: Map<serverId, List<ReadyWriteBatch>>

  sendWriteRequests(batches):
    for each (serverId, batchList):
      tabletServerGateway = metadataUpdater.newTabletServerClientForNode(serverId)

      // 分离 Log batch 和 KV batch
      若 batch.isLogBatch():
        produceLogRequest = makeProduceLogRequest(serverId, logBatches)
        tabletServerGateway.produceLog(produceLogRequest).whenComplete(...)
      若 !batch.isLogBatch():
        putKvRequest = makePutKvRequest(serverId, kvBatches)
        tabletServerGateway.putKv(putKvRequest).whenComplete(...)

      响应处理:
        ├─ produceLog: 按 bucket 逐个处理 PbProduceLogRespForBucket
        ├─ putKv    : 按 bucket 逐个处理 PbPutKvRespForBucket
        ├─ 成功 → completeBatch()  → callback.onCompletion(null)
        │         ├─ idempotenceManager.handleCompletedBatch()
        │         └─ writeBatch.complete()
        │              └─ completeFutureAndFireCallbacks()
        │                   ├─ callback.onCompletion(bucket, offset, null)
        │                   └─ requestFuture.done()  // CountDownLatch 释放
        └─ 失败 → failBatch()      → callback.onCompletion(null, -1, exception)
                  ├─ 可重试错误 (NotLeaderOrFollower, LogEndOffsetMismatch 等)
                  │    → idempotenceManager.handleFailedBatch() + adjustSequences
                  └─ 不可重试错误 → 直接标记失败
```

**与 Kafka Producer 的差异**：
- Fluss 没有独立的 NetworkClient 轮询线程，Sender 直接持有 `MetadataUpdater` 和 RPC 代理
- 请求粒度上有 **两种 RPC**：`ProduceLog`（Log 表）和 `PutKv`（KV 表），而非 Kafka 的单一 `Produce` 请求
- Log 表响应不返回 offset（append-only），KV 表返回 `logEndOffset`（用于 exactly-once 语义）

### 3.6 幂等性层：`IdempotenceManager`

当 `client.writer.enable.idempotence=true` 时启用：

```
IdempotenceManager 核心职责:
  ├─ writerId 管理: 
  │    ├─ maybeWaitForWriterId(targetTables)
  │    │    └─ coordinatorGateway.initWriter(InitWriterRequest).get()
  │    │         └─ 指数退避重试, 最多 3 次
  │    └─ resetWriterId(): 遇到不可恢复错误时重置 (OutOfOrderSequenceException / UnknownWriterIdException)
  │
  ├─ batch sequence 管理 (per TableBucket):
  │    ├─ nextSequence(tableBucket)      // 分配递增 sequence
  │    ├─ incrementBatchSequence(tableBucket)
  │    ├─ lastAckedBatchSequence(tableBucket)
  │    └─ isNextSequence(tableBucket, sequence)  // 验证是否连续
  │
  ├─ inflight batch 跟踪:
  │    ├─ addInFlightBatch(batch, tableBucket)
  │    ├─ removeInFlightBatch(readyWriteBatch)
  │    ├─ canSendMoreRequests(tableBucket)  // 未超过 max.in.flight (idempotent 模式 ≤5)
  │    └─ IdempotenceBucketMap 维护 per-bucket 的 sequence + inflight 队列
  │
  └─ 错误恢复:
       ├─ handleCompletedBatch(): 更新 lastAckedSequence
       ├─ handleFailedBatch(): 区分可重试/不可重试
       ├─ canRetry(): 判断 batch 是否应重试
       │    ├─ OutOfOrderSequence + sequence 已被 reset → 重试
       │    ├─ UnknownWriterId + sequence 已被 reset → 重试
       │    └─ 其他情况 → 不重试
       └─ isAlreadyCommitted(): 通过 lastAckedSequence 判断 batch 是否已被 Server 提交
```

**幂等性保证机制**：
- 每个 `WriteBatch` 分配唯一的 `(writerId, batchSequence)` 元组
- Server 端根据 `(writerId, bucketId, batchSequence)` 去重
- 若检测到连续写失败 → `resetWriterId()` 清空整个 producer 状态

## 4. 完整调用链追踪

以 `append(row(1, "hello"))` 为例，完整栈轨迹：

```
用户线程:

AppendWriterImpl.append(row)
  │ checkFieldCount → OK
  │ getPhysicalPath → PhysicalTablePath("test_db_1", "test_table_1")
  │ logFormat == ARROW → WriteRecord.forArrowAppend(...)
  │                     → estimatedSizeInBytes = -1 (Arrow 不预估)
  └─ AbstractTableWriter.send(record)
       └─ CompletableFuture<Void> + writerClient.send(record, callback)

WriterClient.doSend(record, callback)
  │ throwIfWriterClosed → OK
  │ dynamicPartitionCreator.checkAndCreatePartitionAsync → 分区已存在, skip
  │ bucketAssigner = HashBucketAssigner(3, function)  // 3 bucket
  │ bucketId = bucketAssigner.assignBucket(bucketKey, cluster) → 1
  └─ accumulator.append(record, callback, cluster, bucketId=1, abortIfBatchFull=false)

RecordAccumulator.append(...)
  │ BucketAndWriteBatches.batches[1] → Deque<WriteBatch>
  │ dq 同步块内 tryAppend → ArrowLogWriteBatch.tryAppend(record)
  │   ├─ ArrowWriter.write(row)  // 写入 Arrow columnar buffers
  │   └─ recordCount++ → 追加成功, return

Sender 线程 (异步, 稍后触发):

Sender.sendWriteData()
  │ accumulator.ready(cluster) → readyNodes={serverIdForBucket1}
  │ accumulator.drain(...) → Map<serverId=3, [ReadyWriteBatch]>
  │ sendWriteRequests(batches):
  │   makeProduceLogRequest(serverId=3, batches)
  │     ├─ 对每个 WriteBatch 调用 writeBatch.build() 获取 BytesView
  │     ├─ 填充 PbProduceLogReqForBucket (tableId, bucketId, logData, schemaId, ...)
  │     └─ 构建 ProduceLogRequest
  │   tabletServerGateway.produceLog(request)
  │     .whenComplete((response, exception) -> {
  │       if (exception != null):
  │         failBatch(readyWriteBatch, exception, ...)
  │       else:
  │         for each PbProduceLogRespForBucket in response:
  │           if success:
  │             completeBatch(readyWriteBatch)
  │               ├─ idempotenceManager.handleCompletedBatch(readyWriteBatch)
  │               │    └─ maybeUpdateLastAckedSequence(tableBucket, batchSequence)
  │               └─ writeBatch.complete()
  │                    ├─ callbacks.forEach(cb -> cb.onCompletion(null, -1, null))
  │                    │    └─ future.complete(null)  // 用户线程的 CompletableFuture 完成
  │                    └─ requestFuture.done()  // CountDownLatch 释放
  │     })

用户线程:

writer.flush() → writerClient.flush()
  │ accumulator.beginFlush()
  └─ accumulator.awaitFlushCompletion()
       └─ 等待所有 in-flight batch 的 RequestFuture.done()
            └─ 返回 → 用户得到 AppendResult
```

## 6. 与 Kafka Producer 的架构差异

| 维度 | Fluss WriterClient | Kafka Producer |
|------|-------------------|----------------|
| 发送线程数 | 单线程 (1) | 单线程 IO (Sender) |
| 缓冲模型 | `RecordAccumulator` (借鉴 Kafka) | `RecordAccumulator` |
| 网络模型 | Sender 直接通过 RPC 代理发送 | Sender → NetworkClient 轮询 |
| 记录格式 | 5 种 WriteFormat (Arrow/Indexed/Compacted + Log/KV) | 1 种 RecordBatch (压缩) |
| Row 编码 | KeyEncoder (KV 表), IndexedRowEncoder, CompactedRowEncoder, ArrowWriter | Serializer |
| 桶分配 | BucketAssigner (Hash / RoundRobin / Sticky) | Partitioner |
| 幂等性 | IdempotenceManager + Server 去重 | Idempotent Producer (transactional.id) |
| 请求类型 | ProduceLog / PutKv 两种 RPC | Produce 请求 |
| 内存管理 | Arrow BufferAllocator + LazyMemorySegmentPool | BufferPool |
| 分区创建 | DynamicPartitionCreator (运行时异步创建) | 需显式管理 |

---

## 7. 总结

Fluss 客户端写入引擎在架构上与 Kafka Producer 有明确的借鉴关系（`RecordAccumulator` + `Sender` 模式），但在具体实现上有几个鲜明的差异化特征：

1. **多格式写入管线**：Arrow / Indexed / Compacted 三种 Log 格式 + Indexed KV / Compacted KV 两种 KV 格式，每种格式有独立的记录编码器和 RPC 请求路径
2. **单线程 Sender**：相比 Kafka 的 NetworkClient 轮询模型，Fluss 的 Sender 更轻量化，直接通过 RPC 代理与 TabletServer 通信
3. **双层内存管理**：Arrow 使用独立的 `BufferAllocator`（列式内存），非 Arrow 使用 `LazyMemorySegmentPool`（页式内存），在 `destroyResources()` 时需按序释放
4. **动态分区**：内置 `DynamicPartitionCreator`，运行时按需异步创建分区，对写入线程透明
5. **幂等性**：借鉴 Kafka 的 `(writerId, sequence)` 模型，但通过 Coordinator 的 `initWriter` RPC 分配 writerId，而非 TransactionCoordinator
