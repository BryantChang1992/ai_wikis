## 1. 核心论点

本论文提出**三条核心主张 (Three Claims)**：

### Claim 1: Binding-Constraint Thesis（绑定约束论）
> Agent 可靠性的天花板不是模型能力，而是基础设施质量（Harness Quality）。AI 从工具进化为同事的关键一步，不在于更好的模型，而在于构建可操作、可审计、可恢复的 Agent 基础设施。

### Claim 2: ETCLOVG 七层分类法
将现有的"Lifecycle Hooks"概念拆分为七层独立架构：**E**xecution Environment, **T**ool Interface, **C**ontext Management, **L**ifecycle/Orchestration, **O**bservability, **V**erification, **G**overnance。其中 O 和 V 从 Lifecycle 中独立为第一级层。

### Claim 3: 生态系统映射
将 170+ 开源项目映射到 ETCLOVG 分类体系，形成迄今为止最完整的 Agent 基础设施快照，并揭示采纳模式、覆盖缺口和新兴设计原则。

## 2. 三阶段演化模型

论文提出 Agent 工程的三阶段演化：

| 阶段 | 时间范围 | 核心关注点 | 典型技术 |
|------|----------|------------|----------|
| **Prompt Engineering** | 2022-2023 | 如何通过提示词驱动模型完成单次任务 | Few-shot, Chain-of-Thought, ReAct |
| **Context Engineering** | 2023-2024 | 如何管理模型的上下文窗口，注入检索信息 | RAG, Memory Stores, Compaction |
| **Harness Engineering** | 2024-now | 如何构建 Agent 运行的完整基础设施层 | 沙箱、工具协议、编排、可观测、评估、治理 |

关键洞察：**Harness 不是 Agent 应用的附属品，而是一个独立的工程层面**——就像操作系统之于应用程序。

## 3. 七层 ETCLOVG 分类法深度分析

### 3.1 E — Execution Environment & Sandbox (§3)

**核心问题**：Agent 在哪里执行？如何隔离？如何重置？

**7 大沙箱类别**：

| 类别 | 代表项目 | 安全边界 | 典型用途 |
|------|----------|----------|----------|
| Computer-Use Agent Infra | Claude Computer Use, OpenAI Operator | VM 级隔离 | 桌面自动化 |
| Code-Specialized Sandboxes | SWE-bench Docker, R2E, OpenHands | 容器隔离 | 代码生成与修复 |
| Framework-Integrated Runtimes | LangChain, AutoGen (内置沙箱) | 框架级 | 框架内 Agent 运行 |
| Browser Evaluation Environments | BrowserGym, WebArena | 浏览器级 | Web Agent 评估 |
| OS-Level Permission Sandboxes | Progent (进程级), IsolateGPT | OS 内核级 | 高风险 Agent 部署 |
| Managed Sandbox Platforms | E2B, Daytona, CodeSandbox | 托管云 | 生产级沙箱服务 |
| Sandbox Abstraction Layers | Composio, AgentConnect | 抽象层 | 统一沙箱接口 |

**核心挑战**：
- **SandboxEscapeBench**（Marchand et al., 2026）：前沿模型可在现实配置下利用沙箱弱点进行逃逸
- **可复制性 vs. 真实性**：真实环境提高任务成功率但降低可复制性（镜像差异、网络延迟、浏览器版本）
- **规模化**：大规模训练/评估需要成千上万并行轨迹，一台容器一个任务成本过高
- **Docker 的局限性**：Docker 源自 Linux 内核，macOS/Windows/浏览器/混合云环境需要不同的隔离原语

**设计原则**：
1. 沙箱选择应由威胁模型和任务保真度共同决定，而非单一隔离原语
2. 评估环境的设计直接塑造 Agent 行为——环境即评估
3. 托管沙箱和抽象层代表了从"框架集成"到"平台服务"的架构迁移

### 3.2 T — Tool Interface (§4)

**核心问题**：Agent 如何调用外部能力？工具接口怎样标准化？

**两大协议竞赛**：

#### MCP（Model Context Protocol, Anthropic 2024）
- 客户端-服务器架构，JSON-RPC 通信
- 三个原语：`tools/list`（发现）、`tools/call`（执行）、`resources/read`（数据访问）
- 提供 Resources（数据）、Prompts（模板）、Tools（执行）三种能力
- 生态覆盖：文件系统、数据库、API、浏览器、代码执行器等
- 业内最接近事实标准的工具协议

#### A2A（Agent-to-Agent, Google 2025）
- 基于 HTTP/JSON 的 Agent 间通信协议
- 核心抽象：`Task`（任务对象，含状态机和 Artifacts）
- 支持流式、长轮询、推送通知
- 定位与 MCP 互补：MCP 连接 Agent ↔ Tool，A2A 连接 Agent ↔ Agent

**关键工具设计问题**：
- **Schema 粒度**：描述过简 → 模型误解工具语义；过繁 → 消耗上下文预算
- **工具选择错误**：更多工具 → 更多选择错误（selection error），极端情况下表现不如少工具
- **Prompt Injection 表面**：每个工具都是一个攻击面——恶意文档可能通过文件读取工具注入指令
- **信息流控制**：工具间数据流动需要输入/输出护栏（guardrails）

**设计原则**：
1. 工具描述应当精确到参数级别，并包含错误反馈语义
2. 工具数量和权限应当按需授予（最小权限原则），而非全部暴露
3. 工具响应应结构化（成功/失败/部分结果），便于 Agent 确定性处理

### 3.3 C — Context Management & Memory (§5)

**核心问题**：Agent 如何在长期任务中维持状态一致性？

**三层记忆架构**：

| 记忆层级 | 跨度 | 存储方式 | 代表技术 | 核心风险 |
|----------|------|----------|----------|----------|
| **Short-term** | 单次推理 | Context Window | Prompt Assembly, System Messages | Token 消耗 |
| **Mid-term** | 多轮对话 | Vector Store + RAG | Mem0, Memobase, Letta | 检索精度 |
| **Long-term** | 跨 Session | 结构化记忆 + 配置文件 | Zep, cognee, SuperMemory | Context Drift |

**上下文漂移（Context Drift）**：论文的核心概念——Agent 在长时间运行中，其内部状态逐渐偏离真实任务状态的不可逆过程。

**漂移来源**：
1. **Compaction（压缩）**：每步压缩丢失信息，且无法保证压缩保留了所有关键约束
2. **Retrieval（检索）**：漂移中的 Agent 不知道需要检索什么来纠正自己
3. **Sub-agent Isolation（子 Agent 隔离）**：子任务上下文不污染编排器，但编排器自身仍积累漂移
4. **幻觉沉淀**（QSAF）：幻觉输出被存入持久记忆，在后续 Session 中复用，形成自强化退化

**关键基准测试**：
- **MemoryArena**（He et al., 2026）：专门针对相互依赖的多 Session 任务——上下文漂移最具破坏性的场景
- **MemBench**（Tan et al., 2025）：跨 Session 的时序推理、知识更新、跨 Session 聚合

**设计原则**：
1. 上下文管理应重新定义为状态估计问题（State Estimation）
2. Agent 需要不确定性感知摘要、事实溯源、矛盾处理、显式过期标记
3. **记忆应与评估绑定**——不应仅靠召回准确性判断，而要看是否阻止了下游行动错误
4. Bowne-Anderson & Huber (2026) 的边界论：Context Engineering 本身无法解决长期可靠性——需要完整的 Harness 层

### 3.4 L — Lifecycle & Orchestration (§6)

**核心问题**：Agent 如何跨多轮推理、工具调用、失败恢复、交付物交接来完成任务？

**三层编排架构**：

#### Level 1: Single-Agent Inner Loop（单 Agent 内循环）
- 基础执行单元：观察 → 决策 → 行动 → 反馈
- 遵循 ReAct 范式（Yao et al., 2023）
- **Stateless vs. Stateful 之争**：
  - Codex CLI：纯无状态重播（从交互历史重建执行）
  - Claude Code / OpenCode / Aider：混合模式（可重播历史 + 持久化工件）
- 关键项目：OpenCode (159k ⭐), Claude Code (123k ⭐), Codex CLI (82k ⭐)

#### Level 2: Multi-Agent Orchestration（多 Agent 编排）
- **5 种编排模式**：
  | 模式 | 代表项目 | 特点 |
  |------|----------|------|
  | Hierarchical（层级） | AutoGen, OpenAI Agents SDK, DeerFlow | 高级控制器分配工作 |
  | Team（团队） | oh-my-claudecode | 具名角色的专业化 Agent |
  | Workflow（工作流） | Semantic Kernel | Agent/工具组成显式阶段 |
  | Fan-out（扇出） | Emdash | 多 Agent 并行探索 |
  | Graph Composition（图组合） | LangGraph, Hive | Agent/工具为节点，多种协作模式共存 |

#### Level 3: Full Lifecycle Pipeline（完整生命周期管道）
- 从 Issue → 规划 → 编码 → 测试 → Review → PR
- **Task Runner** 作为核心抽象：管理调度、状态持久化、重试、验证、迭代
- 代表项目：Symphony (OpenAI), Vibe Kanban, GitHub Agentic Workflows
- 人类在循环中的角色：**steering（指导）** 而非 **executing（执行）**

**生命周期状态管理**：论文强调这是**独立于 Context/Memory 和 Observability 的第三类状态**——它是 Harness 自身用来继续执行的操作状态（待处理子任务、检查点、重试、共享工件）。

**设计原则**：
1. 编排的可靠性不取决于模型能否生成"好的下一步"，而取决于 Harness 能否记住已发生的事、决定下一步、从错误中恢复
2. 长期任务中，Harness 需要"离开时保持环境整洁"的交接纪律
3. Anthropic 的演化论证：模型升级（Opus 4.5 → 4.6）后，可以移除 sprint 构造和上下文重置，成本从 $200 → $125

### 3.5 O — Observability & Operations (§7)

**核心问题**：如何监控、调试、优化 Agent 的生产行为？

**论文主张**：Observability 应**从 Lifecycle Hooks 中独立**为第一级层——因为它催生了一个独立的平台和规范生态。

#### 可观测堆栈

| 层次 | 代表平台/规范 | 核心能力 |
|------|--------------|----------|
| **Tracing & Monitoring** | Langfuse, Opik, Arize Phoenix, MLflow | 链路追踪、Token 细目、成本归因、Prompt 版本管理 |
| **底层规范** | OpenTelemetry (OTel) + OpenLLMetry | 标准化 Span 属性、自动 Instrumentation |
| **Agent-Specific Platforms** | AgentOps SDK, RagaAI Catalyst, Laminar | Agent 生命周期视角（Session → Agent → Task → Tool Call） |
| **Cognitive Observability** | Watson (代理推理恢复), AgentLens (可视化分析) | "Agent 为什么这么做"而非"Agent 做了什么" |
| **System-Level Monitoring** | AgentSight (eBPF), AgentTrace | 框架无关、不可篡改的系统级监控 (< 3% CPU) |

#### 成本追踪与优化

- **TensorZero**：统一 Gateway + 可观测 + 实验 + 优化的 LLMOps 栈
- **FrugalGPT**（Chen et al., 2023）：自适应级联路由可匹配 GPT-4 性能并降低 98% 成本
- **QC-Opt**（Shekhar et al., 2024）：质量感知的模型选择，联合优化模型/Token/延迟
- **TALE 现象**（Han et al., 2025）：Token 预算设得过低反而增加消耗——模型溢出预算生成更多 Token

#### 可靠性工程

- **Anthropic 四大故障模式**：One-Shot 尝试 → 过早宣布完成 → Session 间环境破碎 → 标记完成但无测试
- **Managed Agents 架构**（Anthropic）：Brain（Harness+LLM）、Hands（沙箱+工具）、Session（持久事件日志）三者独立恢复
- **MAST 故障分类**（Cemri et al., 2025）：14 种多 Agent 故障模式（κ=0.88）
- **AgentErrorTaxonomy**（Zhu et al., 2025）：错误传播是核心可靠性瓶颈——单一根因故障级联

**关键洞察**：LangChain 2026 调查：89% 团队使用可观测性，但仅 52.4% 运行离线评估——存在结构性"可观测-评估鸿沟"。

**设计原则**：
1. 可观测系统应包含**元监控层**——追踪哪些 Harness 干预仍有效，哪些已成为不必要的开销
2. 基础设施配置本身可偏移基准分数 6 个百分点（p < 0.01, Anthropic 2026a）
3. 每个 Harness 组件编码了一个关于"模型不能做什么"的假设——这些假设随模型进化而过时

### 3.6 V — Verification & Evaluation (§8)

**核心问题**：如何评估 Agent？如何将评估从"终局打分"转化为"质量闭环"？

**论文主张**：Evaluation 应**从 Lifecycle Hooks 中独立**为第一级层。

**五阶段评估生命周期（Task-to-Feedback Lifecycle）**：

```
Task & Benchmark Grounding
    ↓
Pre-Execution Readiness Validation
    ↓
Controlled Execution & Trace Capture
    ↓
Multi-Level Judgement & Failure Attribution
    ↓
Continuous Regression & Deployment Feedback
    ↑___________________________________________↓ (反馈循环)
```

| 阶段 | 核心问题 | 代表工作 |
|------|----------|----------|
| **Stage 1: Grounding** | 评估什么？什么环境？什么工具？什么成功标准？ | SWE-bench, Terminal-Bench, WebArena, OSWorld, GAIA, TheAgentCompany |
| **Stage 2: Pre-Execution** | 沙箱、依赖、工具、权限、评估器初始化是否正确？ | Repo2Run, HAL, SWE-ReX |
| **Stage 3: Controlled Execution** | 在可复制条件下运行 Agent，记录完整 Trace | SWE-agent, OpenHands, R2E-Gym |
| **Stage 4: Multi-Level Judgement** | 结果层面 + 轨迹层面 + 评估器层面 + 失败归因 | LLM-as-Judge, SWE-bench, Terminal-Bench |
| **Stage 5: Continuous Regression** | 评估结果 → 回归测试 → 部署反馈 → Harness 改进 | promptfoo, DeepEval, RAGAS, lm-eval-harness |

**关键洞察**：
- 评估噪声可能伪装成模型失败——失败可能源于模型（推理错误）、工具（schema 误导）、沙箱（配置错误）、上下文（过时记忆）、评估器（不稳定评分）、编排（循环故障）
- 单次运行 pass rate 可能隐藏巨大方差（Bjarnason et al., 2026）
- 评估应将 Agent 行为转化为结构化判断、失败归因和回归反馈——而不仅是报告最终分数

**设计原则**：
1. 报告分数应被理解为一个 **model–harness pair** 的属性，而非模型单独属性
2. 评估应锁定 Harness 跨模型比较，或将 Harness 配置作为显式实验因子
3. 长期运行 Agent 的核心评估问题不是"Agent 是否成功"，而是"为什么成功/失败，路径是否可接受，评估器是否可信，哪个 Harness 组件需要改进"

### 3.7 G — Governance (§9)

**核心问题**：Agent 在部署中的安全、权限、审计和合规如何保障？

**六大治理机制**：

#### 9.1 Identity & Permissions（身份与权限）
- 从本地用户身份到组织级 OAuth 委派
- 跨系统身份联合：Agent 需要代表用户跨多个服务行动
- 当前状态：身份管理在大多数 Agent 系统中缺失或不完整

#### 9.2 Guardrails & Information Flow Control（护栏与信息流控制）
- **Input Guardrails**：检测 Prompt Injection、越狱尝试、不安全指令
- **Output Guardrails**：过滤 PII 泄露、有害内容、不安全代码
- **信息流控制**：追踪数据在工具间的流动，防止敏感数据通过工具链泄露
- 护栏组合问题：独立护栏层可能相互干扰

#### 9.3 Lifecycle Hooks（生命周期钩子）
- 在 Agent 执行的关键节点嵌入检查点：工具调用前/后、上下文更新前/后
- 支持人类审批、策略验证、状态检查

#### 9.4 Constitutions（章程）
- **训练时章程**（Training-time）：RLHF 对齐阶段内化行为准则，难以部署后修改
- **部署时章程**（Deployment-time YAML）：外部化治理规则到 YAML 文件（AutoHarness），安全团队无需触碰 Agent 代码
- **可编程策略语言**（Progent, Formal-LLM, VeriSafeAgent）：逻辑谓词、量词、形式化约束

#### 9.5 Audit Infrastructure（审计基础设施）
- **结构化审计日志**（AutoHarness JSONL）：工具名、参数、风险分类、权限决策、执行结果、Token 成本、延迟
- **跨 Agent 审计**（SAGA）：加密 Token 交换 → 跨信任边界的溯源追踪
- **异常检测分层**：
  - Per-Action 检测（轻量级、低延迟、但无法检测分布式攻击）
  - Trajectory-Level 检测（可检测多步攻击、但延迟高、归因模糊）
- **三级治理管道**（AutoHarness）：Core → Standard → Enhanced 渐进式治理深度

#### 9.6 Situating Governance in Agent Security Landscape
- **Kim et al. (2026)**：7 个设计维度 × 51 攻击方法 × 60 防御方法的系统性安全调查
- **Chen et al. (2026)**：50 篇论文的软件工程视角，六维分类法
- 治理机制与安全风险的映射（Table 3）：Permissions 应对 R1/R5/R6，Input Guardrails 应对 R1/R2，Output Guardrails 应对 R2/R4/R5/R6...
- **真实案例审计**（Codex / Gemini CLI / OpenHands / Browser Use / Nanobrowser / Skyvern）：信息流控制、身份管理、形式化验证在全部系统中缺失

**关键治理缺口**（Table 4）：

| 系统 | Permissions | Hooks | Hardening | Constitution | Audit | Multi-Agent |
|------|------------|-------|-----------|-------------|-------|-------------|
| Codex | ❌ | ✅ | ❌ | ❌ | ✅ | ❌ |
| Gemini CLI | ❌ | ✅ | ❌ | ❌ | ✅ | ❌ |
| OpenHands | ✅ | ❌ | ❌ | ❌ | ✅ | ✅ |
| AutoHarness | ❌ | ❌ | ✅ | ✅ | ✅ | ❌ |

**设计原则**：
1. 从"提示安全"到"Agent 安全"的范式转变——威胁面已扩展到工具调用、沙箱逃逸、多 Agent 社会工程
2. Contextual Security 应作为 AI 安全第四目标（补充 CIA 三元组）——但尚未被 NIST 等标准机构采纳
3. 治理不应该是安全补丁的叠加——需要**可组合的治理模块**和**可移植的策略语言**

## 4. 跨层综合分析 (§11)

### 4.1 Cost-Quality-Speed Trilemma（成本-质量-速度三元悖论）
- 更强的沙箱 → 更高的启动延迟和基础设施成本
- 更丰富的上下文 → Token 消耗和检索开销
- 更深的评估和可观测 → 更慢的迭代和更高的存储/标注成本
- **生产系统不能将质量视为标量目标**——必须决定哪些风险值得昂贵控制

### 4.2 Capability-Control Tradeoff（能力-控制权衡）
- 更大的工具菜单 → 更广的任务覆盖 → 更多的选择错误和 Prompt Injection 面
- 持久记忆 → 更好的连续性 → 更大的溯源、过期和隐私风险
- **这不是安全附加组件——而是连接工具 Schema、上下文策略、运行时权限、身份、可审计性和人类审批的设计轴**

### 4.3 Harness Coupling Problem（Harness 耦合问题）
- 执行环境影响评估结果（包可用性、重置语义、延迟）
- 工具描述消耗上下文预算并塑造模型行为
- 可观测 Trace 仅在身份和权限状态以相同粒度捕获时才能成为治理证据
- 评估设计通过奖励/惩罚特定恢复循环反馈到编排
- **含义**：Harness 变更应作为系统变更进行测试——孤立优化的组件可能在整体中表现更差

### 4.4 From Agent Frameworks → Agent Platforms
- **框架**：本地抽象（Agent, Tool, Memory Store, Execution Loop）
- **平台**：持久工作空间、托管沙箱、身份、计费、可观测、评估、治理、人工交接
- **核心转变**："如何构建 Agent？" → "如何运营一个行动可检查、可逆转的 Agent 集群？"

## 5. 五大开放问题 (§12)

### 5.1 Hardening & Scaling Execution Environments
- 统一安全评估（Prompt Injection、目标不对齐、组合放大）
- 成本模型指导隔离选择（Container / MicroVM / OS 权限 / 完整 VM / 浏览器 / 学习替代）
- 跨自我托管、云端、混合部署的可移植性

### 5.2 Maintaining Reliable State in Long-Running Agents
- **重新定义为状态估计问题**：每次压缩/检索/遗忘步骤损失多少任务相关信息？
- 需要：不确定性感知摘要、事实溯源、矛盾处理、显式过期标记、从外部工件重建状态的恢复程序

### 5.3 Diagnosing Failures from Agent Traces
- **Trace-Native Evaluation**：Trace 应成为计算结果分数、轨迹质量、失败归因、回归测试的主要对象
- 关闭可观测 → 评估循环：将生产异常 Trace 转化为回归 Case

### 5.4 Standard Handoffs Across Agents, Tools & Humans
- 当前状态：MCP（Agent↔Tool）、A2A（Agent↔Agent）、OpenTelemetry（Trace）各自局部标准化
- **缺少跨层交接契约**：当 Plan → Execute、Agent → Tool、Subagent → Parent、System → Human 时应转移什么？
- 需要的状态：意图、约束、权限、工件、溯源、预算、风险级别、Trace 历史、未解决决策

### 5.5 Keeping Harnesses Useful as Models Improve
- **Meta-Engineering**：随着模型变强，Harness 应优化和简化自身——而非单向增加脚手架
- 每个 Wrapper / Reset / Verifier / Planner / Memory Rule / Permission Gate 编码了"模型不能独自做什么"的假设
- Anthropic 实证：Opus 4.5 → 4.6 时移除 context resets，成本降 $200 → $125，质量不降
- **风险**：Benchmark Overfitting——Harness 仅针对狭窄测试套件优化会变脆
- **目标**：Adaptive Simplification——持续追问哪些控制仍必要

## 6. 方法论评估

### 优势
1. **系统性**：170+ 项目映射是迄今最全面的 Agent 基础设施生态快照
2. **概念创新**：ETCLOVG 分类法填补了社区缺失的统一概念框架
3. **实践导向**：大量引用 Anthropic / OpenAI / LangChain 的生产部署经验
4. **跨层视野**：Harness Coupling Problem 和 Capability-Control Tradeoff 的正确识别
5. **安全视角**：将 Agent 安全问题从"Prompt 安全"扩展到"系统安全"

### 局限
1. **快照性**：170+ 项目映射是 2026 年 5 月的时间切片，生态快速演化
2. **覆盖面偏差**：对中文 Agent 生态（Dify, FastGPT, Coze）覆盖有限
3. **深度不均**：E/T/C/L 层讨论较深，O/V/G 层部分项目分析较浅
4. **实证不足**：跨层耦合的量化证据有限——大部分仍是概念性和案例驱动的

## 7. 对我们（CHANG_AI_TEAM）的启示

### 直接可用的设计原则
1. **Agent 可靠性 = Harness Quality × Model Capability**——不能只关注模型选型
2. **沙箱选择遵循威胁模型驱动**——评估/训练/部署场景需要不同的隔离策略
3. **Tool 最小权限原则**——不要给 Agent 它不需要的工具
4. **Context Drift 是长期 Agent 的头号杀手**——需要不确定性感知状态管理
5. **可观测和评估必须闭环**——否则就是"知道 Agent 做了什么，但不知道对不对"

### 可以跟进的方向
- **A2A / MCP**：工具和 Agent 间协议的标准演进值得持续追踪
- **Governance 缺口**：身份管理 + 信息流控制 + 形式化验证几乎在全部系统中缺失——这是差异化机会
- **Harness Simplification**：随着模型变强，我们有意识地移除不需要的脚手架

### 与现有知识库的关联
- 与 [[wiki/Agent-Harness]] / [[wiki/MCP协议]] / [[wiki/Agent可观测性]] 方向强相关
- 建议在合成综述时考虑建立"Agent Infrastructure"方向的 synthesis 页面
