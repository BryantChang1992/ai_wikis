# 流处理弹性与重配置

## 定义

**弹性**（Elasticity）是流处理系统动态调整计算和存储资源以适应负载变化的能力。**重配置**（Reconfiguration）是在运行时修改查询拓扑、并行度或状态分布的能力。

论文的核心判断：

> 弹性现在是标配；**运行时重配置**才是现代流处理系统的核心挑战。

## 负载管理（§6.1）

### Load Shedding（降载）— 1st Gen 策略

DSMS 时代面对过载的唯一手段——选择性丢弃低价值元组：

- **Aurora**：基于 QoS 图指定每个端点的延迟和"价值"函数，过载时丢弃价值最低路径的数据
- **STREAM**：基于算子选择度和资源需求做降载决策
- **核心困境**：丢什么、丢多少、何时丢——没有通用答案；丢失的数据不可恢复

### Load Balancing（负载均衡）— 2nd Gen 策略

- 基于 Key 粒度的状态再分配
- 检测热点 → 重新哈希 → 状态迁移
- 挑战：状态迁移期间的双写/读写一致性

> **当前状态**：大多数系统只完成了"症状检测+告警"，决策仍需人工。形成从检测到执行的**闭环自动化弹性**是主要目标。

## 三种运行时重配置方案（§6.2）

| 方案 | 机制 | 核心创新 | 状态处理 | 局限性 |
|------|------|---------|---------|--------|
| **SEEP** | 扩缩容 + 恢复一体化方案 | 统一故障恢复和扩容的状态迁移流程 | 全局一致性快照 | 面向 Storm，未考虑对齐优化 |
| **Chi** | Live Reconfiguration 含 Alignment Phase | 在 Flink 对齐快照阶段实现状态迁移→零停机 | Aligned Snapshot | 需额外 Alignment 延迟 |
| **Megaphone** | 基于数据流图的拓扑重配置 | 状态迁移路径优化：最小化待迁移数据量 | 增量迁移 + Changelog | 仅适用无环数据流图 |

### Chi 的 Alignment Phase 详解
- 利用 Flink 的 Aligned Snapshot 机制
- 在对齐阶段（所有 input channel 都已收到 marker）执行状态重新分区
- 优势：状态迁移在一致性边界完成，不停机
- 代价：对齐阶段阻塞部分输入 → 吞吐暂时下降

## Flow Control 流控（§6.3）

| 策略 | 原理 | 反馈机制 | 代表系统 |
|------|------|---------|---------|
| **Buffer-based Back-Pressure** | 下游缓冲区满 → 阻塞上游发送 | 被动：下游缓冲区 occupancy | Flink, Storm |
| **Credit-based Flow Control (CFC)** | 下游主动发放 credit → 上游有 credit 才能发送 | 主动：credit 令牌 | Naiad, Timely Dataflow |

![流处理弹性与重配置 - 图2](../diagram/流处理弹性与重配置-fig2.svg)

### 两种策略的对比

| | Buffer-based | Credit-based |
|---|---|---|
| **延迟** | 高（背压逐级反向传播） | 低（credit 提前调度） |
| **公平性** | 任意（先到先得，饥饿可能） | 可配置（按 credit 分配策略） |
| **缓冲区需求** | 大（需吸收传播延迟） | 小（credit 限制 inflight） |
| **实现复杂度** | 低（自然阻塞） | 高（需分布式 credit 管理） |

## 1st vs 2nd Gen 对比

| 维度 | 1st Gen (DSMS) | 2nd Gen (Dataflow) |
|------|---------------|-------------------|
| **过载策略** | Load Shedding（丢数据） | Backpressure（限流）+ Elasticity（扩容） |
| **弹性** | 无（单机架构） | 手动 → 半自动 |
| **重配置** | 不支持 | Epoch 边界重配置 (Chi/Seep) |
| **流控** | 无 | Buffer Backpressure / Credit-based |
| **决策** | 固定策略 | 症状检测 + 人工决策 |

## 开放问题（§6.4）

1. **自动化弹性决策**：闭环自动扩缩容——检测 → 诊断 → 决策 → 执行 → 验证 全自动化
2. **运行时重配置零停机**：Chi 已接近但仍有吞吐抖动
3. **跨算子状态迁移通用化**：不同算子（Window/Join/Aggregate）的状态结构不同，迁移策略应不同
4. **弹性与 SQL 流处理的结合**：声明式查询 vs 弹性执行计划的运行时适配
