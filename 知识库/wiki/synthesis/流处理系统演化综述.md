# 流处理系统演化综述


> 基于 Fragkoulis et al., "A Survey on the Evolution of Stream Processing Systems" (arXiv:2008.00842v2, 2023) 的精读，覆盖流处理系统从 1992 到 2022 的 30 年演化，涵盖乱序数据管理、状态管理、容错、弹性、Dataflow 模型五大核心域。


![[diagram/stream-processing-evolution.svg]]

## 一、流处理的三个时代

流处理经历了**三个代际**的范式跃迁，每代的架构哲学和处理模型有本质区别（参见 [[Stream-Processing-System-Generations]]）。

### 第一代：DSMS — 把数据库倒过来 (2000–2010)

代表系统：STREAM, Aurora/Borealis, TelegraphCQ, Gigascope, IBM System S

核心思想：**"流是演变的关系"**。DSMS 本质上是 DBMS + 流适配层——共享查询计划、关系模型（CQL）、单机 Scale-up。关键创新包括滑动窗口聚合、Synopsis（系统内部状态）和降载策略。

限制很明显：只能单机运行，状态是系统内部实现细节，用户不能自定义处理逻辑，容错仅限于 Best-Effort。

### 第二代：Dataflow — 分布式数据处理 (2011–2022)

代表系统：Storm, Flink, Spark Streaming, Beam/Dataflow, Kafka Streams, Naiad

核心转变三个：
1. **从单机到分布式 Shared-Nothing** — 数据并行、任务并行、流水线并行
2. **从"流即关系"到"流即事件"** — Dataflow Model 只需要 Timestamp，语义由用户定义
3. **从无状态到 System-Managed State** — 用户定义状态，系统负责持久化、扩展、容错

标志性里程碑：Google Dataflow Model (2015) 首次提出批流统一，将乱序处理形式化成 Event Time / Watermark / Trigger / Refinement 四抽象。

### 第三代：Emerging — 事件驱动 + 微服务 (2022–)

代表系统：Ray, Stateful Functions, Neptune, Ambrosia, Arcon

趋势方向：
- 从数据流 Job/DAG → Actor 模型的事件驱动微服务
- 从流处理引擎 → 通用分布式计算集成
- 硬件加速（GPU/FPGA 加速流算子）
- 事务化流处理（S-Store 开创的方向）

论文对 3rd Gen 的判断较为谨慎——仍是方向性判断，缺乏系统性分析。

## 二、乱序数据管理：从排序到追踪

流处理系统对乱序的认知经历了根本性转变（参见 [[流处理乱序数据管理]]）。

### 两种哲学

| | In-Order Processing | Out-of-Order Processing |
|---|---|---|
| **做法** | 接收元组 → 缓冲区排序 → 按序转发处理 | 元组不经排序直接处理 → 追踪进度确定迟到的范围 |
| **代价** | 等待延迟 + 内存开销 | 实现复杂 + 需保存状态到延迟上界 + 修正处理 |
| **代表** | STREAM, Aurora/Borealis | Flink, Beam/Dataflow, Millwheel |

2nd Gen 几乎全部转向 Out-of-Order——因为 Watermark + Trigger 机制让系统可以做到：低延迟处理（不等待排序）+ 高正确性（迟到数据通过修正处理纳入结果）。

### 进度追踪机制的演化

论文首次将五个机制统一成清晰对比：

| 机制 | 本质 | 1st/2nd | 关键局限 |
|------|------|---------|---------|
| **Slack** | 固定等待量（元组数/时间）| 1st | 不考虑实际延迟，固定值 |
| **Heartbeat** | 源端生成进度信号 | 1st | 只反映源端进度，不反映系统内部 |
| **Low-Watermark** | 系统内最老未处理工作 | 2nd | 需要 Punctuation 传播 |
| **Punctuation** | 嵌入流的元数据通信通道 | 1st→2nd | 本身不追踪进度，是进度的载体 |
| **Pointstamp** | (timestamp, location) 对 | 2nd | 实现复杂，每元组开销大 |

关键洞察：LWM 将"进度"从 "源端进度"推广到 "系统内处理进度"，这是 2nd Gen 能做 Out-of-Order Processing 的基础。而 Pointstamp 进一步增加了位置维度，支持循环查询的进度追踪。

### Dataflow Model 的统一

Google Dataflow Model 将上述分散的机制整合为四条抽象：
- **Event Time / Processing Time 分离** — 正确性用前者，延迟用后者
- **Watermark** — 事件时间进度的启发式估计
- **Trigger** — 何时物化窗口结果（Watermark/P-Time/Count/Composite）
- **Refinement** — 如何修正早期结果（Accumulating/Discarding/Retracting）

这个框架使批处理成为流处理的特殊情形（有界数据流），催生了 Apache Beam 并定义了 2nd Gen 的语义标准。

## 三、状态管理：区分 1st 和 2nd Gen 的决定性能力

论文将状态管理定位为**区分 1st Gen 和 2nd Gen 的决定性能力**（参见 [[流处理状态管理]]）。

### 从 Synopsis 到 System-Managed State

```
Synopsis (系统内部) → Application-Managed (用户全责) → User-Defined, System-Managed
    1st Gen                 过渡期 (Storm)                   2nd Gen
```

- **1st Gen**：状态是系统内部实现细节，用户不可见。Borealis 已认识到问题，但受限于单机架构
- **Storm**：完全不管状态，用户自己接外部 KV-Store (Redis/HBase)
- **2nd Gen**：用户通过 State API 定义语义，系统处理持久化、扩展、容错——这是正确方案

### 三种存储架构的选择

| 架构 | 适用场景 | 代表 |
|------|---------|------|
| **In-Memory** | 状态有限、低延迟要求 | 1st Gen DSMS |
| **Out-of-Core** | 大规模状态、成本可控 | **Flink + RocksDB (业界主流)** |
| **External** | 状态与计算解耦、Cloud 原生 | Millwheel + BigTable/Spanner |

Out-of-Core 是当下的主流选择——RocksDB/FASTER 等 LSM-Tree 引擎让流处理系统可以管理远超内存容量的状态。LSM-Tree 的合并策略直接影响流处理状态的写放大和读性能（参见 [[LSM-Tree]] 的合并策略分析）。

### 持久化粒度的权衡

| 粒度 | 正确性 | 延迟 | 代表 |
|------|--------|------|------|
| Record-Level | 最高 (E2E Exactly-Once) | 最高 | Millwheel |
| Epoch-Level | 高 (Exactly-Once on State) | 中 | **Flink（业界标准）** |
| Batch-Level | 中 | 低 | Spark Streaming |

Flink 的 Epoch-Level 通过 Chandy-Lamport 异步快照或其 Aligned 变体实现。Aligned Snapshot 通过对齐阶段保证 snapshot 的完整性——这同时也是运行时重配置（如 Chi 系统）的基础。

## 四、容错：Exactly-Once 不等于端到端 Exactly-Once

容错是流处理系统最容易被误解的领域（参见 [[流处理容错模型]]）。

### 语义的四级

```
At-Most-Once (已过时)  →  At-Least-Once (Storm默认)  →  Exactly-Once on State (Flink/Beam)  →  Exactly-Once on Output (极少数)
```

关键区别：**Exactly-Once on State** 保证系统内部状态可恢复到一致性快照，但不能撤销已发布给外部的输出。**Exactly-Once on Output** 需要端到端的事务协调——源端去重 + 算子状态一致 + Sink 幂等/事务提交。

这就是论文所说的 **Output Commit Problem**：分布式快照可以恢复状态，但不能撤回已发出的消息。

### 容错的四维度

论文通过四个维度对 18 个系统做了横向对比（Table 4）：

| 维度 | 选项空间 | 主流选择 |
|------|---------|---------|
| **处理语义** | At-Most / At-Least / Exactly-Once-State / Exactly-Once-Output | Exactly-Once on State |
| **复制策略** | Active / Passive / Replay-only | Passive |
| **恢复数据** | Snapshots / Output Buffers / Log Offsets | Snapshots + Log Offsets |
| **存储介质** | Local SSD / Remote DFS / In-Memory | Local + Remote 混合 |

### 高可用：一个缺失的定义

论文明确指出**流处理没有标准的高可用定义**。现有文献混用恢复时间、性能开销和资源利用率。论文提出基于**处理进度延迟**的定义方案——高可用 = 处理进度在前沿附近的时间占比。但这仍未成为业界标准。

### 非确定性计算的容错

Clonos 是唯一明确提出 Exactly-Once 包含非确定性计算的系统：保存"决定因子"到持久存储，恢复时精确重现。但这未形成通用框架——今天的流处理系统如果使用 Processing Time 窗口或随机采样，故障恢复后结果可能不同。

## 五、弹性和重配置：从丢数据到智能调度

### 从 Load Shedding 到 Elasticity

1st Gen 面对过载的答案：**丢数据**（Load Shedding）。Aurora 基于 QoS 图决定丢弃哪些路径的数据，STREAM 基于算子选择度和资源需求做决策。

2nd Gen 的答案：**Backpressure + Elasticity**。背压限流避免雪崩，弹性扩缩容主动增加资源。但目前的弹性大多数停留在——系统检测到症状（背压/延迟），**人来做决策**。

### 运行时重配置：真正的挑战

| 方案 | 机制 | 影响力 |
|------|------|--------|
| **SEEP** | 统一故障恢复和弹性扩容的状态迁移 | 概念开创 |
| **Chi** | 在 Aligned Snapshot 的对齐阶段执行状态重分区 | 零停机重配置的代表 |
| **Megaphone** | 基于 DAG 拓扑优化状态迁移路径 | 面向无环数据流图 |

Chi 是目前最接近生产可用的方案——利用 Flink 的 Aligned Snapshot 机制在对齐阶段一次性完成状态迁移，实现不停机扩容。代价是对齐阶段有短暂的吞吐下降。

### Flow Control 的选择

| | Buffer-based Back-Pressure | Credit-based Flow Control |
|---|---|---|
| **原理** | 下游缓冲满 → 阻塞上游 | 下游主动发 credit → 上游有 credit 才发 |
| **延迟** | 高（逐级反向传播）| 低（提前调度） |
| **公平性** | 差（饥饿可能）| 可配置 |
| **复杂度** | 低 | 高 |
| **代表** | Flink, Storm | Naiad, Timely Dataflow |

背压虽然简单但存在级联传播、尾延迟放大等问题——下游的一个慢 Task 会反向阻塞整条流水线。

## 六、Dataflow Model：2nd Gen 的语义基石

Google Dataflow Model (2015) 不是凭空创造——它是 1st Gen 概念的重新组合和形式化（参见 [[Dataflow-Model]]）。

| Dataflow 概念 | 1st Gen 前身 | 来源系统 |
|--------------|-------------|---------|
| Watermark | Low-Watermark + Heartbeat | STREAMS, Naiad |
| Watermark 传播 | Punctuation | Aurora, Naiad |
| Trigger | 修正处理的触发逻辑 | CEDR, Borealis |
| Refinement (Retract) | Dynamic Revision (delta messages) | Borealis |

Dataflow Model 的真正贡献是**用四条抽象统一了分散在多个系统中的机制**，并将批处理定义为流处理的特殊情形——“批就是有界数据流”。

### What / Where / When / How

```
What  →  做什么计算？    ParDo, GroupByKey, Combine
Where →  在哪个窗口？    Fixed / Sliding / Session Windows
When  →  何时物化？     Trigger (Watermark / P-Time / Count / Composite)
How   →  如何修正？     Refinement (Accumulating / Discarding / Retracting)
```

这个四问框架让 Dataflow Model 成为一个**完备的窗口计算模型**——所有窗口聚合都可以映射到这四个维度上。

## 七、对知识库的交叉关联

### 与 LSM-Tree 的关系

论文多处将 **LSM-Tree（RocksDB/FASTER）** 定位为流处理 Out-of-Core 状态管理的事实标准。这与已有 [[LSM-Tree]] / [[LSM-Tree-写放大]] / [[LSM-Tree-存储引擎体系综述]] 直接关联——LSM-Tree 的合并策略直接影响流处理状态的写放大和读性能。

### 与 Fluss 的关系

[[Fluss-流处理平台架构综述]] 中 Fluss 用 RocksDB 做存储引擎，Arrow 做列式格式，Iceberg 做湖仓融合——这些选择都可以在 SP-Survey 的分类框架中找到定位。Fluss 本质上是 2nd Gen 技术栈 + 3rd Gen 湖仓融合的混合方案。

### 与事务体系的关系

[[分布式数据系统事务与一致性新进展-2026综述]] 覆盖了 ACID 事务概念，SP-Survey 补充了流处理特有的**Exactly-Once 分级体系**和 **Output Commit Problem**——两篇综述在容错语义上形成互补。

## 八、核心知识点汇总

1. **乱序是常态**：网络延迟不均、多源汇聚、Join/Window 操作天然产生乱序。处理乱序不是 bug fix，而是流处理系统的核心能力
2. **进度追踪是乱序管理的基础**：LWM 将进度从源端推广到系统内，Pointstamp 加上位置维度支持循环查询
3. **状态管理是 2nd Gen 的决定性能力**：从 Synopsis → App-Managed → User-Defined, System-Managed 的三阶段演化
4. **Exactly-Once on State ≠ 端到端 Exactly-Once**：分布式快照可恢复状态但不能撤回外部输出（Output Commit Problem）
5. **弹性是标配，重配置是真正挑战**：Chi 的方案（在 Alignment Phase 迁移状态）是目前最接近零停机的方案
6. **Dataflow Model 是 2nd Gen 的语义基石**：四条抽象统一了乱序处理，批流统一重新定义了计算模型
7. **流处理在向事件驱动和湖仓融合演进**：3rd Gen 趋势 + Fluss 等新系统体现代际跃迁
