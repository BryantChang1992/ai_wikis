## 二、强化类（本周/下周实施）

### 2.1 标签体系规范化

**现状**：大家都写了 tags，但标签粒度和命名风格不一致。

**建议标签层级**（不强约束，但作为 Agent 写入指导）：

```yaml
# 领域标签（建议每张卡至少 1 个）
tags:
  - 存储引擎       # 或 流处理 / OLAP / 时序数据库 / 事务 / 分布式协调
  - LSM-Tree       # 核心技术名
  - 写优化         # 关注点（可选但推荐）
```

| 领域 tag | 适用卡片 |
|----------|---------|
| `存储引擎` | LSM-Tree 系、Doris 存储层、InfluxDB TSM |
| `流处理` | 流处理系、Dataflow、Fluss |
| `OLAP` | Doris 系 |
| `时序数据库` | InfluxDB 系 |
| `事务` | 事务模型调研、CockroachDB 系、Aurora 系、Rosé 系 |
| `分布式协调` | CockroachDB Leader Lease, Silo, Fluss 分布式协调 |
| `消息系统` | Fluss 系 |
| `Agent-First` | Agent-First 系列 |

> 这不是强制的，是指导性的。Agent 写入时参考此表，新领域出现后自动加入。

### 2.2 Agent 写入质量门禁

在 schema.md 中补充一条 **写入前自检清单**，Agent 写 wiki 卡片时必须确保：

```
✅ frontmatter 含 type / sources / tags / status / created / related
✅ sources 指向 sources/ 下实际存在的文件
✅ related 中所有 [[wikilink]] 指向 wiki/ 或 wiki/synthesis/ 下实际存在的 .md
✅ status 非空（draft / reviewed / final）
✅ 内容非纯摘要——包含论文章节号、系统对比数据、架构图
```

这条不依赖 Lint 扫描——是 Agent 写入时的自律规则。可以减少今天 SP-Survey 第一版那种"二次摘要退化"。

### 2.3 前端 [[wikilink]] 使用规范（补充）

正文中**首次引用**其他概念时用 `[[概念名]]` 包裹，但不要在同一个段落里用三次——只在首次出现时链一次。

```markdown
❌ LSM-Tree 的 Compaction 策略影响 [[LSM-Tree]] 的写放大，[[LSM-Tree]] 的读性能，以及 [[LSM-Tree]] 的空间效率。
✅ LSM-Tree 的 Compaction 策略影响写放大、读性能和空间效率（详见 [[LSM-Tree]]）。
```

### 3.2 每季度做一次"知识库回顾"（新增建议）

不是全面扫描，而是问三个问题：
1. 哪些卡片从未被引用过（孤儿）——要不要删或建连接？
2. 哪些 synthesis 的 `updated` 超过 3 个月——要刷新吗？
3. 哪些领域我们感兴趣但 wiki 里是空白——下季度重点补什么？

产出：一页 Markdown 回顾，写到 `wiki/synthesis/季度回顾-YYYY-QX.md`。

### 3.3 未来新增 AI Infra 领域时

新增卡片命名直接沿用现有模式：`AI-Infra-XXX.md` 或 `推理加速-XXX.md`。synthesis 综述照例生成到 `wiki/synthesis/`。
