# 流处理状态管理

## 定义

**状态**（State）是所有连续流计算内部副作用的载体——包括活跃窗口、记录桶、部分/增量聚合、以及用户自定义的变量。论文将其定位为**区分 1st Gen 和 2nd Gen 流处理系统的决定性能力**，并追踪了从 "synopsis" → "application-managed" → "user-defined, system-managed" 的完整演化路径。

## 状态类型演化（§4.2）

| 阶段 | 类型 | 特征 | 代表系统 |
|------|------|------|----------|
| **1st Gen (DSMS)** | System-Defined Synopsis | 系统内部算子状态，用户不可见。Borealis 已考虑故障恢复协议设计。STREAM 支持组合式 synopsis（用已有 synopsis 定义新 synopsis） | STREAM, Aurora/Borealis, TelegraphCQ |
| **过渡期** | Application-Managed | 用户自行在内存或外部 KV-Store 中管理状态。无系统持久化保证。灵活性高但用户需自己处理扩展和恢复 | Storm, S4 |
| **2nd Gen** | User-Defined, System-Managed | 用户通过 Stateful API 定义状态，系统负责持久化、扩展、容错。限制：用户不能直接控制底层数据结构 | Flink, Beam, Kafka Streams, Samza, Spark Streaming |

### 核心转折

> 早期 DSMS（Aurora/STREAM）已对状态管理做了开创性工作——Borealis 甚至在 2000 年代初期就提出了嵌入式状态、持久存储访问、故障恢复协议等概念。但那个时代受限于单机架构和内存容量，状态的作用被限制为"支撑系统操作符的内部实现"。

## 三种状态存储架构（§4.3 + Figure 4）

![[diagram/流处理状态管理-fig1.svg]]



| 架构 | 原理 | 代表系统/技术 | 优势 | 代价 |
|------|------|------------|------|------|
| **In-Memory** | 状态全部在内存数据结构中 | 1st Gen DSMS | 零 I/O 延迟，实现简单 | 内存容量硬限制；故障后全丢 |
| **Out-of-Core** | 热数据内存 + 冷数据 LSM-Tree 外存 | Flink(RocksDB), FASTER(MSR) | 支撑大规模状态；利用 NVMe 带宽 | 写/读放大（LSM-Tree 合并开销） |
| **External** | 状态完全由外部 DB/KV-Store 管理；算子本身无状态 | Millwheel(BigTable+Spanner) | 状态与计算解耦(Cloud-friendly)；可复用 DB 的 ACID/弹性 | 外部调用延迟；增加运维依赖 |

### Out-of-Core 详解：为什么是 LSM-Tree？

论文明确将 **LSM-Tree 变体（RocksDB/LevelDB/FASTER）** 定位为 Out-of-Core 架构的标准选择。原因：
- 顺序写入与流处理的追加模式匹配
- LSM-Tree 的合并可将冷数据下沉到 SSD
- RocksDB 的合并策略直接影响流处理状态的**写放大**和**读性能** → 参见 [[LSM-Tree-写放大]]

### External 详解：Millwheel 模式

Millwheel (Google Cloud Dataflow 的运行时引擎) 是 External 架构的代表：
- 每个 Task 本身**无状态**——只保留最近变更在内存
- 每条输出 + 状态更新作为一个原子事务写入 BigTable
- 利用 BigTable 的盲写入、Bloom Filter 和 Spanner 的 ACID 特性
- 代价：每个记录级操作都要涉及 BigTable/Spanner 的 I/O，延迟较高
- 优势：状态天然持久化、可弹性扩缩、故障恢复无需快照

## 持久化粒度（§4.4）

![[diagram/流处理状态管理-fig2.svg]]



| 粒度 | 机制 | 代表系统 | 原理 |
|------|------|---------|------|
| **No Persistence** | 状态不持久化，依赖复制操作符恢复 | Aurora/Borealis, Storm(1.0) | 通过 Standby 操作符或全量重算恢复 |
| **Batch-Level** | 每微批次结束持久化 | Spark Streaming, Trident/Storm, S-Store | 严格微批次处理范式；S-Store 以 DB 事务为批次 |
| **Epoch-Level** | 周期性地（时间/记录数）做分布式一致性快照 | Flink, Samza, Kafka Streams, Naiad, IBM Streams | Chandy-Lamport 异步快照或其对齐变体 |
| **Record-Level** | 每条输出 + 状态更新原子提交 | Millwheel | 利用 BigTable 事务保证每条记录的一致性 |

### Epoch-Level 的两种提交策略（§4.6.2）

![[diagram/流处理状态管理-fig3.svg]]

- **Unaligned (Chandy-Lamport)**：最佳运行时性能，但恢复时需重放 in-flight events（类似 DB recovery 的 fuzzy checkpoint）
- **Aligned (Flink)**：通过对齐阶段（Alignment Phase）让所有通道同步 marker，产生完整 epoch 快照。延迟略高，但恢复更快，且是对齐重配置（如 Chi 系统）的基础

## 可扩展状态操作（§4.5）

| 类型 | 语义 | API 示例 | 分布方式 |
|------|------|---------|----------|
| **Partitioned (Keyed) State** | 每个 key 独立管理状态分片 | `keyBy()` → 有状态算子 | 按 key 哈希分布到不同物理 Task |
| **Non-Partitioned (Global/Broadcast) State** | 单例状态或广播到所有实例 | 全局聚合、物理算子级计数器 | 作为单例映射到物理 Task，不拆分 |

## 1st vs 2nd Gen 对比（§4.7）

| 维度 | 1st Gen (DSMS) | 过渡期 (Storm) | 2nd Gen (Flink/Beam) |
|------|---------------|----------------|---------------------|
| 状态定义权 | 系统内部 (Synopsis) | 用户自定义 | 用户定义 API + 系统管理 |
| 持久化保证 | 无/依赖复制 | 无（需用户接外部 DB） | Epoch-Level 分布式快照 |
| 存储规模 | 仅内存容量 | 外部 DB 容量 | Out-of-Core（TB 级） |
| 可配置性 | 不开放 | 完全开放（无管理） | API 级开放 + 系统内优化 |
| 数据独立性 | 无 | 无 | 状态与持久化解耦（类比数据库的数据独立性） |

## 开放问题（§4.8）

1. **负载感知的状态管理**[78]：不同算子有不同访问模式——有的写密集（窗口聚合），有的读密集（Join）。自动选择存储后端和数据结构是活跃方向
2. **可插拔状态后端**：FASTER、RocksDB 等多后端供选择，但自动化选择+运行时切换仍未解决
3. **新负载**：ML Serving、图分析、Cloud Apps 引入新的状态访问模式，现有系统未优化
4. **状态独立性的形式化**：类比数据库的"数据独立性"概念，状态编程与状态持久化的分离正成为趋势
