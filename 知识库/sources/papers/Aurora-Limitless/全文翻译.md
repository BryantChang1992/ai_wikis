# Aurora PostgreSQL Limitless Database: 构建高可扩展 OLTP 数据库（全文翻译）

> 原文：Dmitry Arkhangelskiy et al., SIGMOD Companion '26
> DOI: 10.1145/3788853.3803089

## 摘要

我们介绍 Aurora Limitless Database，一个云原生分布式数据库系统，在保持强一致性保证的同时为 Amazon Aurora PostgreSQL 扩展水平扩展能力。该系统通过 Router 层实现查询分发，通过 PostgreSQL shard 存储层实现透明扩展，消除了应用层分片的需求。我们的关键技术贡献包括：(1) 一个将基于时间的多版本并发控制（MVCC）与两阶段提交（2PC）相结合的分布式事务协议；(2) 一个结合垂直和水平扩展的自适应扩缩容框架；(3) 一个在 DML 和 DDL 操作中均保持强一致性的分布式查询处理引擎。

Aurora Limitless Database 在各种生产工作负载下实现了近线性扩展，达到每秒数百万事务，同时保持毫秒级延迟。系统通过无服务器能力和动态分片自动优化资源分配，相比静态配置显著降低了运维成本。该方法使可扩展 OLTP 应用能够利用 PostgreSQL 的丰富功能集，而无需牺牲一致性，也无需管理和配置多个系统的运维负担。

## 1. 引言

现代云原生应用需要数据库能够无缝扩展，处理每秒数百万事务和 PB 级数据，而无需复杂的应用层分片或基础设施管理。然而，运营高吞吐工作负载的客户——从处理实时交易的金融服务平台到 SaaS 应用——面临一个根本性约束：传统单主数据库架构最终会达到写吞吐、存储容量和并发连接的上限。

为解决这一问题，我们提出 Aurora Limitless Database，一个云原生分布式数据库系统，通过水平扩展能力扩展 Amazon Aurora PostgreSQL [36, 37]。系统通过数据分区实现写扩展，同时保持强一致性保证——Read Committed 和 Repeatable Read 隔离级别，并增强为外部一致性 [10]。它还利用 Aurora 现有的计算存储分离架构和分布式存储来提供持久性、跨可用区高可用性、I/O 吞吐扩展和弹性磁盘容量。

Aurora Limitless 旨在实现以下关键目标：(1) 不依赖集中式事务管理器的写扩展；(2) 即使是跨 shard 事务也保留 PostgreSQL 事务语义；(3) 对单 shard 访问模式达到类单系统性能；(4) 无缝水平扩展；(5) 类单系统管理体验。

系统通过以下技术实现这些目标：

- 数据模型允许客户指定哪些表需要分片及如何分片（§2）。使用 hash 分布对分片表进行水平分区。
- 分离 Shard（存储数据分区并处理查询）和 Router（服务应用流量并编排跨 shard 查询处理）（§3）。两者均利用 Aurora 存储系统。
- Router 和 Shard 集群可独立扩展，以适配变化的工作负载条件。它们还可同时在垂直和水平两个方向上扩展（§4）。
- 支持 ACID 分布式事务，涉及多个 shard 同时符合 PostgreSQL 隔离级别语义（§5）。通过 Amazon Time Sync 服务 [21] 计算事务快照和提交时间戳，实现基于时间的 MVCC。
- 利用基于时间戳的事务处理允许客户在特定时间点创建整个集群的一致性快照（§6）。
- Router 的查询处理器通过将计算下推至数据所在的 shard 来优化复杂跨 shard 查询（§7）。

Aurora Limitless 已在 AWS 生产环境中运行超过一年，客户使用的集群包含多达 64 个 shard 和 32 个 router。

## 2. 数据模型

Aurora Limitless 支持三种表类型，每种针对不同的访问模式优化——sharded、reference 和 standard。以示例 schema 说明：`customers` 表按 `cust_id` 分片（sharded），`orders` 表也按 `cust_id` 分片且与 `customers` co-located，`tax_rates` 表在所有 shard 上全量复制（reference）。`CREATE TABLE` 语法不变，通过 session 参数声明表类型。

- **Sharded 表**：按 hash(shard_key) 水平分区到各 shard。共享相同 shard key 的表可通过 `limitless_create_table_collocate_with` 共置。
- **Reference 表**：在每个 shard 全量复制。适合频繁 JOIN 但不适合分片的小表。
- **Standard 表**：位于单个 shard。用于无需分片的数据，后续可转换为 sharded/reference。

## 3. 系统架构

Aurora Limitless 实现了多层架构，将控制操作与数据处理分离。

### 3.1 Aurora 分布式存储

Aurora 分布式存储通过跨 3 个 AZ 的数据复制提供高可用性和持久性（6 副本，每 AZ 2 个）。数据库卷被分割为更小的段（segment），每个段复制到 6 个存储节点。即使在单个 AZ 完全故障加上另一个 AZ 的一个副本同时故障的情况下，数据仍保持持久。存储层还自动检测并修复段故障，动态调整数据容量。Aurora Limitless 利用该存储层管理分布在多个 shard 上的 PB 级数据。

### 3.2 数据平面

数据平面由两个独立扩展的组件组成：Router 集群和 Shard 集群。每个 Router 或 Shard 实现为一个带有自己存储卷的 PostgreSQL 集群。

- **Shard**：拥有实际数据分片，执行 Router 下发的子计划。
- **Router**：服务所有应用流量。不存储实际数据，但维护集群拓扑、数据库 schema 和分片键范围到 shard 的映射。Router 接收客户端查询，规划执行并发送到 shard。还负责设置事务快照和驱动提交协议。Router 连接管理器以事务粒度在 shard 连接上复用客户端会话。

### 3.3 控制平面

控制平面包含以下组件：
- **集群管理服务**：管理 shard/router 生命周期，维护 shard group 拓扑的权威注册表，传播拓扑变更。
- **监控服务**：每个节点本地 agent 监控健康状态并触发 failover，跨节点监控检测网络分区。
- **热管理服务**：用于自动扩缩容（§4）。
- **备份恢复服务**（§6）。

### 3.4 高可用性

Router 和 Shard 分布在 3 个 AZ 上：
- **Router 故障**：Routers 同构且通过 DNS 单端点访问。一个 Router 不可用时，DNS 将流量解析到其他 Router，无需专用 standby。
- **Shard 故障**：Shard 可配置 0-2 个 standby（由客户选择）。Shard 故障时，standby 接管其存储卷和工作负载。

## 4. 自适应扩缩容

Aurora Limitless 实现二维自动扩缩容。客户设置 shard group 的总最小/最大计算容量（ACU 为单位）。初始 Router 和 Shard 数量由最大容量决定。

### 4.1 垂直扩缩容

每个节点有自己的 min/max ACU（基于 Aurora Serverless V2）。初始时 ACU 预算均分给所有 shard group 成员，随后按各节点实际消耗 ACU 比例动态调整：

```
dynamicMinACU_i = shardGroupMinACU × (consumedACU_i / Σ consumedACU)
dynamicMaxACU_i = shardGroupMaxACU × (consumedACU_i / Σ consumedACU)
```

Shard 和 Router 的扩缩容触发器不同：Shard 侧重于 compute + buffer cache；Router 侧重于 heap memory。采用非对称速率：scale-up 快速响应负载尖峰，scale-down 保守防 thrashing。

### 4.2 水平扩缩容

客户可手动或自动触发 shard split 和 router 添加。

**Table Slice**：每个 sharded table 最多 512 个 slice。Slice 是 shard split 的最小迁移粒度。Co-located 表的对应 slice 一起迁移。

**Shard Split 工作流**：
1. **Phase 1: 存储层克隆**：使用 Aurora copy-on-write 克隆源 shard volume，避免给已高负载的源 shard 增加读负担。
2. **Phase 2: Redo 日志回放**：克隆完成后，新 shard 追赶上源 shard 的 redo log。
3. **Phase 3: Switchover**：短暂全局锁表→更新 slice→shard 映射→源/新 shard 更新 schema（DETACH PARTITION）→释放锁。此阶段对客户有短暂影响（DDL/更新被阻塞，某些进行中的事务被终止）。
4. **Phase 4: Cleanup**：后台清理不再属于各 shard 的 slices。

**Router 添加**：克隆现有 Router，更新拓扑，注册 DNS。

**垂直 vs 水平**：两种扩展维度相互补充，客户可根据工作负载特性结合使用以最大化性能或优化性价比。

## 5. 并发控制与提交协议

Aurora Limitless 在分布式数据库中通过将基于时间的并发控制与 Aurora 存储精心集成，保持 PostgreSQL 的事务语义。支持 Repeatable Read（在 PG 中等同于 Snapshot Isolation）和 Read Committed。

核心可见性属性：**Property 1**——对于两个不同的事务 T 和 T'，T' 的修改被 T 的快照包含，当且仅当 `T'.commitTs ≤ T.startTs`。

### 5.1 时间戳快照隔离

传统 PostgreSQL 的 xid-based snapshot 在高并发下是已知的 CPU 瓶颈，且不适合分布式环境（单节点不知道所有运行中的事务）。Aurora Limitless 通过 Amazon Time Sync 服务 [5] 生成的时间戳定义快照。

Time Sync 提供 `now()` 函数返回当前时间 `t` 和时钟误差界（CEB），真实时间保证在 `[earliest = t - CEB, latest = t + CEB]` 内。典型 CEB 低于 1ms，部分 AWS 区域已降至低双位微秒级。

事务开始时 Router 分配 `startTs = now().latest`。行版本可见性判定：
```
xmin.commitTs ≤ T.startTs < xmax.commitTs
```
Shard 维护 xid → commitTs 映射（嵌入 PG commit log 结构）。

### 5.2 写冲突检测

Shard 层使用与 PG 相同机制：修改行时获取排他锁直至提交，同时检查已有行版本是否仍在事务的快照覆盖范围内。如果有最新版本，事务中止。

### 5.3 时间感知的两阶段提交 (2PC)

由于 Router 无 standby 节点（节省成本），Router 故障恢复可能需要数分钟，不适合作为 2PC coordinator。Aurora Limitless 将分布式事务状态持久化到指定的 **Lead Shard**，Lead Shard 可以有 standby。

**无故障 2PC 流程**：
1. Router 选取一个参与 shard 作为 Lead Shard，向其他参与 shard 发送 `PREPARE TRANSACTION`（含 Lead Shard ID）
2. 各 shard 计算 prepare timestamp（使用 `now().latest`），持久化 prepare 信息 + lead shard ID，返回 timestamp
3. Router 取所有 prepare timestamp 的最大值发送给 Lead Shard
4. Lead Shard 取 `max(router_value, own_proposal)` 为 commitTs，本地持久化并提交，回复 Router
5. Router 通知客户端，异步向其他 shard 发送 `COMMIT PREPARED`

**Router 故障恢复**：已 prepare 但未 commit 的事务，其他 shard 查询 Lead Shard 决定：Lead 已 commit 则全部 commit，否则全部 abort。

**优化**：只读事务直接 commit；单 shard 更新事务跳过 2PC，目标 shard 本地管理 commit。

### 5.4 从 Shard 读取数据

**时钟偏移处理**：使用 HLC（混合逻辑时钟）。每个 shard 维护逻辑时钟 C，每次收到读请求 T 时推进 `C = max{C, T.startTs + 1}`。Shard 的 prepare timestamp = `max{C, now().latest}`。确保 T 读取后提交的任何事务都有 `commitTs > T.startTs`，无需等待时钟同步。

**Prepare 状态事务处理**：读到 prepare 状态的行时，Shard 向 Lead Shard 查询该事务状态。如果已提交则根据 commitTs 决定可见性；否则 Lead Shard 推进其 C ≥ `T.startTs + 1`，确保该事务最终的 commitTs > T.startTs。

**COMMITTING 状态处理**：事务已获得 commit time 但写入存储未完成时，标记为 COMMITTING。读取该事务修改的行的 T 需等待其完成 commit 或 abort。

### 5.5 保证实时顺序（外部一致性）

Aurora Limitless 实现了比 SI 更强的外部一致性：如果 T2 在 T1（修改后）返回客户端后开始，则 `T2.startTs > T1.commitTs`。

通过 **commit wait** 实现：Lead Shard 确定 commitTs 后不立即回复 Router，而是等待 `now().earliest > commitTs`。由于 commit wait 与 storage write 并行进行，而 storage write 延迟通常超过 CEB，因此很少增加关键路径延迟。

### 5.6 DDL 语句

DDL 操作遵循 PostgreSQL 语义。DDL 的提交涉及相关 shard 和所有 Router 的 2PC 变体，确保整个集群中 DDL 原子可见。DDL 操作的锁需在相关 shard 和 Router 上获取。全局 sequence 通过预分配范围优化以避免跨节点通信。DDL 死锁预防：所有 DDL 先获取一个 distinguished node 的锁，成功后再并行获取其他节点锁。

### 5.7 Read Committed 实现

Read Committed 的区别：
- 每个语句计算新的 snapshot timestamp（而非整个事务同一个快照）
- 关闭写冲突检测——修改行时获取排他锁但不检查冲突，即使有其他事务在 T 开始后修改了该行

### 5.8 分布式死锁管理

超时后，选择一个 Router 从各 shard 收集 waits-for 图，检查全局环，随机选择 victim 中止。

### 5.9 故障切换、备份与恢复

**Shard 故障切换与读租约**：利用 Aurora 单写者属性避免脑裂。通过写操作关联 lease（TTL 参数），shard 的读请求只能服务 snapshot time ≤ last_lease_end 的数据。新实例恢复时等待 `now().latest > t_last + TTL` 确保前一 lease 过期。

**备份与恢复**：所有备份关联 Time Sync 时间戳 t，精确反映 commitTs ≤ t 的所有事务。所有 WAL 记录带 timestamp 标签。恢复时从备份加载所有卷，prepared 状态事务通过查询 Lead Shard 解决。

## 6. 查询处理与优化

**表表示**：Router 将 sharded 表表示为 PG partitioned table，每个 partition 是通过 FDW 连接对应 shard 的 foreign table。利用 PG partition pruning + 自定义 FDW 实现分布式查询优化。

**关键优化技术**：
- **Predicate Pushdown**：内置操作 + IMMUTABLE 函数的谓词推至 shard
- **Join Pushdown**：co-located 表内 join 在 shard 本地并行执行，结果 append 到 Router
- **Reference Table Join**：支持与 sharded 表的特定 join 类型（内连接、outer join with reference as null-padded side）下推
- **Partial Aggregate/Sort Pushdown**：减少 Router 间数据传输
- **Function 分发**：声明函数关联的 shard key 参数，执行推至对应 shard
- **单 Shard 查询优化**：partition pruning 识别单 shard 查询，直推到底（核心性能路径）

## 7. 实验评估

**环境**：us-east-1，Serverless V2 最大 256 ACU/node（≈512 GB RAM），HammerDB (TPC-C 派生)，12000 warehouses，1000 并发客户端，~10% 分布式事务。

**五种配置**：

| 配置 | Routers | Shards | Max ACU | NOPM | Avg Latency (NEWORD) |
|------|---------|--------|---------|------|----------------------|
| r1 | 2 | 4 | 1536 | 1,268,350 | 29.70ms |
| r2 | 4 | 8 | 1536 | 2,012,763 | 21.06ms |
| r3 | 8 | 16 | 1536 | 2,042,201 | 16.42ms |
| r4 | 4 | 8 | 3072 | 2,107,539 | 13.13ms |
| r5 | 8 | 16 | 3072 | 2,891,718 | 9.72ms |

**关键发现**：
- **水平扩展对比**（r1→r2）：吞吐 +58.7%，延迟 -29.1%
- **垂直扩展对比**（r3→r5）：吞吐 +41.6%，延迟 -40.8%
- **叠加效果**（r1→r5）：吞吐 +128%，延迟 -67%
- ACU 分配图显示更多 shard 带来更均衡的负载分布

系统在非饱和状态下，增加容量可显著改善性能。水平扩展在垂直扩展达上限时发挥最大价值。

## 8. 相关工作

**分布式 PostgreSQL 系统**：
- **Citus** [12]：支持分片但不提供 SI 保障，暴露大量事务异常
- **Greenplum** [24]：支持 SI 但依赖集中式 coordinator，易成为瓶颈

**地理分布式 SQL 数据库**：
- **Spanner** [11]：TrueTime + 硬件同步时钟，是时间戳事务的先驱
- **CockroachDB** [35]：HLC 但不保证外部一致性
- **YugabyteDB** [40]：HLC + 保守最大时钟偏移等待
- **Aurora DSQL** [2]：乐观并发控制

**扩缩容与架构**：
- **PolarDB** [38]：RDMA 时间戳 oracle 或 HLC
- **Snowflake** [13]：存储计算分离但面向 OLAP；Aurora Limitless 面向 OLTP

Aurora Limitless 的独特之处：完全 PG 兼容性 + 二维扩缩容 + 物理时钟外部一致性 + Aurora 存储层省去共识协议。

## 9. 结论与未来工作

Aurora Limitless 的关键创新：混合 scale-up/scale-out 模型 + 基于时间的并发控制深度集成 + 外部一致性。

**生产运行经验与开放问题**：
1. **迁移难度**：从单机 PG 或自建分片迁移是最大的客户挑战。选择 shard key 是单向门（需数据迁移才能改）。需要更好的自动化迁移工具
2. **Schema 演进**：分布式环境需 schema versioning、canary 部署、DDL 的 MVCC（如 Azure SQL MD-MVCC）
3. **Commit 与 Visibility 顺序**：社区 PG 的 commit order ≠ visibility order；Aurora Limitless 用 timestamp 强制对齐。未支持 Serializable——原因：(a) 客户需求有限；(b) PG 的 SSI 中 serialization order ≠ commit order
4. **二维扩缩容调优**：何时 scale-up vs scale-out 的参数选择需要精细调试以最大化性能和成本效益
5. **PostgreSQL 现代化的机会**：众多 PG 行为需要适配分布式环境
