# Aurora Limitless 精读分析

| 项目 | 内容 |
|------|------|
| 标题 | Aurora PostgreSQL Limitless Database: Building a Highly Scalable OLTP Database |

## 问题背景

传统单主 PostgreSQL 架构面临写吞吐、存储容量、并发连接的上限。虽然 AWS 已有 Aurora 的存储计算分离架构解决了 I/O 和可用性问题，但写扩展仍然受限于单写节点。已有的分库分表方案（如 Citus、Vitess、ShardingSphere）需要应用层感知分片逻辑，导致运维复杂。

Aurora Limitless 的目标是将 Aurora PostgreSQL 从单写节点平滑扩展为分布式数据库，同时保持 **完整的 PostgreSQL 兼容性**（PostgreSQL 生态的全套功能）和 **强一致性**（Read Committed + Repeatable Read，增强为外部一致性）。核心差异在于：用户只需声明表的分片键，其余全部自动化。

## 核心贡献

1. **基于物理时钟的时间戳快照隔离 (Clock-SI + HLC hybrid)**：利用 AWS Time Sync 服务（卫星同步原子钟，CEB < 1ms，部分区域已达双位微秒级）提供时间戳，替代 PostgreSQL 的 xid-based snapshot。结合 HLC 避免了读延迟等待，实现外部一致性。

2. **以 Lead Shard 为中心的 2PC 协议**：分布式事务提交不在 Router（Routers 是无状态 + 无 standby 以节省成本）而是在一个 designated Lead Shard 上持久化事务状态，Lead Shard 有 standby，failover 秒级恢复。commit timestamp 取各 shard prepare timestamp 的最大值。

3. **二维自适应扩缩容**：垂直扩缩容基于 Aurora Serverless V2 (ACU)，动态按比例分配 capacity 预算；水平扩缩容通过 table slice 粒度的 shard split（512 slices/table），利用 Aurora 的 copy-on-write 克隆 + redo log 回放实现零负载迁移，switchover 阶段短暂锁表。

4. **分布式查询优化器**：基于 PostgreSQL partition pruning + FDW 机制，将 sharded table 表示为 partitioned foreign table，实现 predicate pushdown、co-located join pushdown、partial aggregation/sorting pushdown。单 shard 查询直推到底，避免跨 shard 往返。

5. **时间点一致备份与恢复**：所有 WAL 记录带 commit timestamp 标签，备份通过对齐 timestamp 边界保证所有 shard/routers 恢复到同一逻辑时间点。

## 架构设计

### 三层架构

```
Client App → DNS → Router Fleet → Shard Fleet → Aurora Distributed Storage (3-AZ)
                        ↕
                   Control Plane
```

- **Router Fleet**：无状态（无 standby），通过 DNS LB 分发连接。持有 schema + topology + shard-key-range 映射元数据。负责快照分配、查询规划、2PC 协调。连接以 transaction 粒度复用到 shards。
- **Shard Fleet**：拥有数据分片，每个 shard 是一个独立 Aurora PostgreSQL 集群（有自己的 storage volume）。支持 0-2 个 standby 跨 AZ。
- **Control Plane**：集群管理（拓扑注册/变更传播）、监控（CloudWatch）、热管理（自动 split 决策）、备份恢复。
- **Aurora Storage**：6 副本 / 3 AZ（2 per AZ），自动段修复，弹性容量。

### 三种表类型

| 表类型 | 分布策略 | 典型场景 |
|--------|---------|---------|
| **Sharded** | Hash 分布，支持 co-locate（相同 shard key 同 shard 共置） | 大表，如 customers, orders |
| **Reference** | 全量复制到每个 shard | 小表常 JOIN，如 tax_rates |
| **Standard** | 单 shard 存放 | 不便分片的表，导入过渡 |

用户通过 session 参数 `rds_aurora.limitless_create_table_mode` 声明类型，`CREATE TABLE` 语法不变。

## 关键技术深挖

### 1. 时间戳快照隔离 (Clock-SI + HLC)

**传统 PostgreSQL snapshot 的问题**：基于 xid（全局自增事务ID） + xip_list（所有活跃事务列表），高并发下 snapshot 计算是 CPU 瓶颈，且在分布式环境下各节点无法感知所有运行中的事务。

**Aurora Limitless 方案**：
- 每事务开始时，Router 调用 `now().latest`（Time Sync 上界）作为 `startTs`
- 每行版本的可见性判定：`xmin.commitTs ≤ T.startTs < xmax.commitTs`
- Shard 维护 xid → commitTs 映射（嵌入 PostgreSQL commit log 结构）
- **HLC 消除读延迟**：传统 Clock-SI 要求等到 shard 本地时间超过 snapshot timestamp。Aurora Limitless 用 HLC（`C = max{C, T.startTs + 1}`）推进 shard 逻辑时钟，确保后续提交的事务 timestamp > T.startTs，T 无需等待立即可读
- **Prepare 处理**：当 T 读到 prepare 状态的行时，shard 向 lead shard 查询该 prepare 事务的状态——要么已 commit（获得 commitTs 决定可见性），要么未 commit（lead shard 推进其 HLC 确保 commitTs > T.startTs）

**外部一致性**（比 SI 更强）：如果事务 T2 在 T1 返回客户端后开始，则 `T2.startTs > T1.commitTs`。实现方式：commit wait——lead shard 确定 commitTs 后等待 `now().earliest > commitTs` 再回复 Router。实践中 storage write latency > CEB，wait 很少增加延迟。

### 2. 分布式事务提交协议 (Lead Shard 2PC)

**为什么不是 Router 做 coordinator**：Router 无 standby → 故障恢复需数分钟 → 2PC 长时间阻塞。

**Lead Shard 2PC 流程**（图 3）：
1. Router 选一个参与 shard 为 lead，向其他 shard 发 `PREPARE TRANSACTION`（带 lead shard ID）
2. 各 shard 计算 prepare timestamp（`max{C, now().latest}`），持久化 prepare 信息 + lead shard ID，返回 timestamp
3. Router 取所有 prepare timestamp 的最大值发给 lead shard
4. Lead shard 取 `max(router_value, own_prepare_ts)` 为 commitTs，本地 commit，回复 Router
5. Router 通知客户端，异步向其他 shard 发 `COMMIT PREPARED`
6. Router 故障恢复：其他 shard 查询 lead shard 决定 commit/abort

**优化**：只读事务直接 commit；单 shard 更新事务跳过 2PC，由目标 shard 本地管理 commit。

### 3. 自适应扩缩容

**垂直扩缩容 (ACU-based)**：
- 基于 Aurora Serverless V2，每个 node 有 min/max ACU
- 总 capacity 预算按比例动态分配：`dynamicACU_i = totalACU × (consumedACU_i / ΣconsumedACU)`
- Shard 侧重 compute + buffer cache；Router 侧重 heap memory
- 非对称速率：scale-up 快速响应，scale-down 保守防 thrashing

**水平扩缩容 (Shard Split)**：
- **Table Slice**：最少 512 slices/table，slice 是迁移的最小粒度
- **Split 流程**：
  1. Clone：Aurora copy-on-write 克隆源 shard volume（零负载）
  2. Redo Replay：新 shard 追赶源 shard 的 redo log
  3. Switchover：短暂锁表，更新 slice→shard 映射，`DETACH PARTITION`，释放锁
  4. Cleanup：后台清理不再属于自己的 slices
- **自动化**：heat-management service 监控 ACU + storage，超阈值自动触发 split
- **Router 添加**：类似流程（clone → topology 更新 → DNS 注册）

### 4. 查询优化

**实现机制**：
- Sharded table 在 Router 被表示为 PostgreSQL **partitioned table**，每个 partition 是对应 shard 的 **foreign table**
- 利用 PG partition pruning 判断单 shard / 跨 shard
- **异步 Foreign Scan** 并行向各 shard 发起子计划

**关键优化**：
- **Predicate pushdown**：内置操作 + IMMUTABLE 函数的谓词推至 shard
- **Co-located join pushdown**：相同 shard key 的表 join 在 shard 本地执行，Router 仅 append 结果
- **Reference table join pushdown**：reference table 在所有 shard 有完整副本，支持部分 join 下推
- **Partial aggregation/sorting pushdown**：减少 Router 间数据传输
- **Function 分发**：声明 SQL 函数的 shard key 参数，自动推至对应 shard
- **单 shard 直推**：最优路径，单次往返

## 与竞品对比

| 维度 | Aurora Limitless | CockroachDB | Spanner | YugabyteDB | Citus | PolarDB |
|------|-----------------|-------------|---------|------------|-------|---------|
| 隔离级别 | SI (RR) + RC + External Consistency | Serializable + RC | Serializable + SI | Serializable + SI | 无 SI (分布式事务) | SI (RDMA oracle 或 HLC) |
| 时钟方案 | AWS Time Sync + HLC | HLC | TrueTime | HLC | N/A | RDMA TO / HLC |
| 外部一致性 | ✅ (commit wait) | ❌ | ✅ (TrueTime) | ❌ | ❌ | 部分 |
| 扩缩容 | 垂直 (ACU) + 水平 (auto split) | 水平 (auto rebalance) | 水平 (split) | 水平 | 水平 (手动) | 水平 |
| Coordinator | Lead Shard (有 standby) | 无中心 (lease-based) | 中心式 (Paxos leader) | Tablet leader (Raft) | 中心式 | 中心式 |
| 存储层 | Aurora Storage (3 AZ, 6 copies) | 本地 RocksDB | Colossus (GFS) | DocDB (RocksDB) | 本地 PG | PolarFS (RDMA) |
| PG 兼容性 | 完全兼容（原生PG内核） | PG 兼容（自有内核） | 无 PG 兼容 | PG 兼容 | 完全兼容 | PG 兼容 |

**Aurora Limitless 的独特定位**：
- 完全 PostgreSQL 兼容（不改 CREATE TABLE 语法、不改应用代码——只声明 shard key）
- 垂直+水平**二维**扩缩容是竞品中独一无二的
- 针对跨 AZ（非跨 region）场景优化，省去 Paxos/Raft consensus 层（借助 Aurora Storage 的 quorum writes）
- 时间戳事务 + Lead Shard 2PC 在不牺牲一致性的前提下减少中心依赖

## 实验评估总结

**环境**：us-east-1，Serverless V2 最大 256 ACU/node（≈512 GB RAM），HammerDB (TPC-C 派生)，12000 warehouses，1000 并发客户端，约 10% 分布式事务。

**配置矩阵**（表 2）：

| 配置 | Routers | Shards | Max ACU | NOPM | Avg Latency |
|------|---------|--------|---------|------|-------------|
| r1 | 2 | 4 | 1536 | 1,268,350 | 29.70ms |
| r2 | 4 | 8 | 1536 | 2,012,763 | 21.06ms |
| r3 | 8 | 16 | 1536 | 2,042,201 | 16.42ms |
| r4 | 4 | 8 | 3072 | 2,107,539 | 13.13ms |
| r5 | 8 | 16 | 3072 | 2,891,718 | 9.72ms |

**关键发现**：
- **水平扩展**（r1→r2）：吞吐 +58.7%，延迟 -29.1%
- **垂直扩展**（r3→r5）：吞吐 +41.6%，延迟 -40.8%
- **水平+垂直叠加**（r1→r5）：吞吐 +128%，延迟 -67%
- ACU 分配图显示：更多 shard 带来更均衡的负载分布（峰值 ACU 方差更小）
- 近线性扩展直至饱和点

## 局限与未来方向 (§10)

1. **迁移难度**：从单机 PG 或自建分片迁移到分布式 PG 需要重新设计 schema（选择 shard key 是不可逆的单向门），缺乏自动化工具
2. **Schema 演进**：分布式环境需要 schema versioning、canary 部署、DDL 的 MVCC（类似 Azure SQL 的 MD-MVCC）
3. **Commit 与 Visibility 顺序**：社区 PG 中 commit order ≠ visibility order，Aurora Limitless 用 timestamp 强制对齐；但未支持 Serializable——原因有两个：(a) 客户需求有限；(b) PG 的 SSI 实现中 serialization order ≠ commit order
4. **Hybrid scaling 调优**：二维扩缩容的参数（何时 scale-up vs scale-out）需要仔细调优才能最大化性能/成本比
5. **Standard/Reference table join 限制**：与 sharded table 的某些 join 类型（outer join with sharded as null-padded side, anti-join）尚未优化，回退到 Router 执行
