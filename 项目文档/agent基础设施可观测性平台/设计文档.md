# Agent 基础设施可观测性平台

> [!info] 设计文档
> CHANG_AI_TEAM 从单 Agent 走向多 Agent 协作后的可观测性方案。
> 外部发布版本：[[GitHub Pages 版设计文档]]
>
> 完整 HTML 版（含 SVG 架构图）：`ai_memory_chang_ai_team/tech_designs/agent-infra/observability-dashboard.html`

## 1. 背景与动机

CHANG_AI_TEAM 从单 Agent 走向多 Agent 协作后，面临三个关键挑战：

- **看不见** — 不知道各 Agent 在干什么
- **管不了** — 任务状态不透明
- **算不清** — Token 消耗不可控

### 现状痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| 🔮 任务黑盒 | CTO 下发任务后不知道子 Agent 是否启动、进度如何、是否阻塞 | 需要手动确认，无法自动化编排 |
| 📊 无消耗统计 | 不知道每个 Agent、每个任务、每天消耗了多少 Token | 成本失控风险 |
| 🔀 协作混乱 | Agent 之间调用没有规范，fork/isolated 选择随意 | 上下文污染或信息丢失 |
| 📋 无状态追踪 | 任务完成/失败后没有持久化记录 | 无法回溯、无法审计 |
| 🔄 冷启动脆弱 | 所有状态在内存中，Gateway 重启全丢 | 不可靠 |

### 为什么现在做

- CHANG_AI_TEAM 已有 CTO → 专家 → 执行层三级结构，协作复杂度在增长
- 技术调研周报已经跑了 2 期，自动化调度需要可靠的监控
- OpenClaw 提供了 Hooks、Webhooks、结构化日志等基础设施，集成成本可控
- 在 Agent 数量膨胀之前把基建做好，比事后补救成本低得多

## 2. 设计目标

### P0 · 必须实现
- **任务看板**：实时展示所有 Agent 任务状态（pending → in_progress → done/failed），支持按 agent、时间、状态筛选
- **Token 统计**：按 session、agent、日期维度的 Token 消耗统计，支持趋势图

### P1 · 应该实现
- **冷启动恢复**：SQLite + Git 备份，重启后 < 30s 恢复全部状态
- **协作规范**：定义 Agent 角色权限矩阵、通信协议、状态变更规范，写入 prompt

### P2 · 可以实现
- **告警通知**：任务超时、Token 超限、Agent 异常状态 → 飞书通知
- **历史回溯**：任意历史时间点的 Dashboard 快照，支持按周/月聚合

### 非目标
- 不做 Agent 调度引擎（调度由 OpenClaw 本身负责）
- 不做实时流式日志采集（日志解析 + 轮询够用）
- 不做用户权限系统（单用户场景）
- 不做分布式（单机部署，所有 Agent 在同一 Gateway）

## 3. 系统架构

> [!important] 架构原则
> 数据采集与展示完全解耦。采集层负责"写"——Hook 监听事件 + Log Parser 解析日志 → 写入 SQLite。展示层负责"读"——Dashboard HTML 通过 API 读取数据渲染。两层独立演进，互不影响。

![[architecture-overview.svg]]

### SQLite + Git 备份 对比

| 对比维度 | SQLite + Git | 飞书 Bitable | 纯 Git JSON |
|----------|-------------|-------------|------------|
| 查询能力 | ✅ SQL 聚合/过滤/排序 | ⚠️ API 查询受限 | ❌ 无查询能力 |
| 冷启动 | ✅ Git clone 恢复 | ⚠️ 依赖飞书 API | ✅ Git clone |
| 写性能 | ✅ 本地毫秒级 | ❌ 限流 100/s | ⚠️ 需 fsync |
| 离线可用 | ✅ 完全本地 | ❌ 需网络 | ✅ 本地 |
| 可视化 | ⚠️ 需自建前端 | ✅ 飞书原生 | ⚠️ 需自建前端 |

## 4. 数据模型

### agents 表
```sql
CREATE TABLE agents (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    role        TEXT NOT NULL,     -- 'cto' | 'vp_arch' | 'expert_infra' | 'rd' | 'sre' | 'qa'
    level       TEXT NOT NULL,     -- 'vp' | 'expert' | 'worker'
    model       TEXT,
    status      TEXT DEFAULT 'idle',
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now'))
);
```

### tasks 表
```sql
CREATE TABLE tasks (
    id              TEXT PRIMARY KEY,
    agent_id        TEXT NOT NULL REFERENCES agents(id),
    parent_task_id  TEXT REFERENCES tasks(id),
    title           TEXT NOT NULL,
    status          TEXT DEFAULT 'pending',  -- pending|in_progress|blocked|done|failed
    priority        TEXT DEFAULT 'medium',
    session_key     TEXT,
    started_at      TEXT,
    completed_at    TEXT,
    error_message   TEXT,
    tags            TEXT,                    -- JSON array
    created_at      TEXT DEFAULT (datetime('now')),
    updated_at      TEXT DEFAULT (datetime('now'))
);
```

### sessions 表
```sql
CREATE TABLE sessions (
    id              TEXT PRIMARY KEY,
    agent_id        TEXT REFERENCES agents(id),
    task_id         TEXT REFERENCES tasks(id),
    status          TEXT DEFAULT 'active',
    token_in        INTEGER DEFAULT 0,
    token_out       INTEGER DEFAULT 0,
    model           TEXT,
    started_at      TEXT,
    ended_at        TEXT,
    created_at      TEXT DEFAULT (datetime('now'))
);
```

### tool_calls 表
```sql
CREATE TABLE tool_calls (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id      TEXT NOT NULL REFERENCES sessions(id),
    tool_name       TEXT NOT NULL,
    params_json     TEXT,
    result_summary  TEXT,
    status          TEXT DEFAULT 'success',
    duration_ms     INTEGER,
    token_cost      INTEGER,
    created_at      TEXT DEFAULT (datetime('now'))
);
```

## 5. 数据采集层

### 采集策略总览

| 数据 | 采集方式 | 触发时机 | 优先级 |
|------|---------|---------|-------|
| Agent 注册 | Hook: `agent:bootstrap` | Agent 启动前 | P0 |
| 任务创建 | Hook: `message:received` | 用户下发任务 | P0 |
| 任务状态变更 | Agent 调 `sessions_send` 给 Supervisor | 关键节点 | P0 |
| Session 生命周期 | Log Parser 匹配 lifecycle 事件 | start / end / error | P0 |
| Tool Call 记录 | Log Parser 匹配 tool_call 事件 | 每次 tool 调用 | P1 |
| Token 消耗 | Log Parser 提取 usage metadata | Session 结束 | P0 |
| 对话摘要 | Hook: `session:compact:after` | Compaction 后 | P2 |

### 采集方案

**Hook 采集（实时事件）**：利用 OpenClaw Hook 系统监听 `agent:bootstrap`，开发自定义 Hook `agent-telemetry`。

**Log Parser 采集（轮询守护进程）**：驻留守护进程，每 2 秒轮询 OpenClaw 日志文件的增量部分，解析 JSON 行日志并写入 SQLite。

> [!warning] 为什么不用 Agent 自己上报
> Agent 自己调用工具写 SQLite 有天然的问题——如果任务超时/崩溃，就不会上报；如果 Agent prompt 不稳定，上报格式也容易变形。Log Parser 作为外部守护进程，能可靠地捕获所有事件。

**Agent 自主状态上报（Hybrid 补充）**：Log Parser 捕获"发生了什么事"，但不知道"语义"。Agent 在关键节点通过 `taskboard` CLI 主动上报状态。

### Supervisor Agent 对账

- Agent 上报 `done` 但 session 未结束 → 标记 `stale`
- Session 已结束但任务仍 `in_progress` → 自动标记 `done` 或 `failed`
- Agent 30 分钟无 tool call 且状态 `in_progress` → 标记 `stuck`

## 6. Dashboard 前端设计

### 技术选型

- **渲染方式**：纯静态 HTML + Vanilla JS，无框架依赖
- **图表库**：Chart.js (CDN)，~60KB gzip
- **自动刷新**：轮询 10s
- **数据 API 部署**：本地 FastAPI + ngrok 隧道

### 关键视图

| 视图 | 数据源 | 刷新 | 交互 |
|------|--------|------|------|
| 任务看板 | `GET /api/tasks?status=active` | 10s | 筛选、排序、点击展开 |
| Token 汇总 | `GET /api/tokens?period=today\|week` | 30s | 按 Agent 细分 |
| 趋势图 | `GET /api/tokens?granularity=hour&days=7` | 60s | hover 查看数值 |
| Agent 列表 | `GET /api/agents` | 10s | 点击进入详情 |
| Activity 流 | `GET /api/activity?limit=20` | 10s | 实时滚动 |

## 7. Agent 协作规范

### 角色权限矩阵

| 操作 | CTO | VP 层 | 专家层 | 执行层 |
|------|-----|-------|--------|--------|
| 创建子 Agent | ✅ any | ✅ 同领域 | ❌ 仅建议 | ❌ |
| 任命 VP/专家 | ✅ | ❌ | ❌ | ❌ |
| 技术决策 | ✅ 最终 | 🔶 领域内 | 🔶 建议 | ❌ |
| 写入 Dashboard | ✅ | ✅ | ✅ | ✅ |
| 修改规范 | ✅ | ❌ | ❌ | ❌ |
| 直接操作 Git | ✅ | ✅ | 🔶 受限 | ❌ |

### 通信协议

| 场景 | 方式 | context 模式 | 说明 |
|------|------|-------------|------|
| CTO → 专家 | `sessions_spawn` | `isolated` | 不需要 CTO 对话上下文 |
| CTO → 执行层 | `sessions_spawn` | `isolated` | 独立执行，仅收结果 |
| 专家 → 执行层 | `sessions_spawn` | `isolated` | 独立执行 |
| 同级协作 | `sessions_send` | — | 不创建新 session |
| 需要上下文传递 | `sessions_spawn` | `fork` | ⚠️ 谨慎使用 |

### 任务通信格式

```json
{
  "task_id": "uuid",
  "type": "research | development | review | deploy",
  "title": "简短标题",
  "description": "详细描述",
  "priority": "low | medium | high | critical",
  "deadline": "ISO8601 或 null",
  "dependencies": ["task_id_1"],
  "output": {
    "format": "html | json | markdown | file",
    "path": "输出路径",
    "schema": "可选 JSON Schema"
  }
}
```

## 8. Agent 任务状态机

![[agent-state-machine.svg]]

### 状态变更规则

| 当前状态 | 允许的目标状态 | 触发者 |
|---------|--------------|--------|
| pending | in_progress, cancelled | Agent / 用户 |
| in_progress | done, failed, blocked | Agent |
| blocked | in_progress, cancelled | Agent / 用户 |
| done | （终态） | — |
| failed | pending（重试） | 用户 |
| stuck / stale | done, failed | Supervisor |

## 9. 数据 API 设计

### API 清单

| Endpoint | 方法 | 说明 |
|----------|------|------|
| `/api/agents` | GET | 所有 Agent 及其状态 |
| `/api/agents/:id` | GET | Agent 信息 + 历史任务 + token 趋势 |
| `/api/tasks` | GET | 任务列表（支持筛选） |
| `/api/tasks/:id` | GET | 任务详情 + 子任务 + session |

## 10. 实施路径

- [x] **Phase 0 · 设计文档** — 架构设计、数据模型、API 定义、Agent 规范草案（本文档）
- [ ] **Phase 1 · 数据采集 MVP**（1 周） — Log Parser 守护进程 + SQLite schema + `taskboard` CLI
- [ ] **Phase 2 · Dashboard 前端**（1 周） — 纯静态 HTML，部署到 GitHub Pages
- [ ] **Phase 3 · Agent 规范落地**（2 周） — 将协作规范写入各 Agent 的 AGENTS.md
- [ ] **Phase 4 · 完善 & 运营**（持续） — 告警通知、Supervisor 对账、Git 冷启动备份

### 关键风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Log Parser 遗漏事件 | 中 | 高 | Supervisor 对账兜底 |
| Agent prompt 不稳定 | 中 | 中 | taskboard CLI 幂等操作 |
| ngrok 隧道不稳定 | 中 | 中 | 静态数据兜底 + Cloudflare Tunnel 备选 |
| SQLite 并发写入冲突 | 低 | 高 | WAL 模式 + 单写入者设计 |
| Token 数据采集不全 | 中 | 低 | 仅影响统计精度，不影响功能 |
