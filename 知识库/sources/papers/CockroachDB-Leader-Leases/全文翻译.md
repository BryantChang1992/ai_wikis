# Scalable Leader Leases For Multi Consensus Groups in CockroachDB
## 论文全文翻译

### 摘要（Abstract）

CockroachDB 是一个基于共识复制的大规模分布式数据库，利用租约（lease）高效且具有强一致性保证地服务读取请求。随着部署规模扩展到数十万乃至数百万个共识组（consensus group），现有的租约策略——无论是基于共识组的心跳续约还是集中化管理——在可扩展性、可用性和正确性之间产生紧张关系。

本文介绍 **Leader Leases**，一种将 Raft 领导者角色与租约持有者角色统一的新协议。Leader Leases 建立在 **Liveness Fabric** 之上——一个共享的、去中心化的故障检测层，在所有节点之间运行，独立于各 Raft 组的心跳。我们扩展 Raft 协议，加入了 **Leader Fortification** 机制，使 Raft 领导者能够获得更强的领导稳定性保证，从而安全地承担租约持有者的职责。

我们在生产环境中评估了 Leader Leases，重点关注故障恢复、CPU 开销和吞吐量，与 CockroachDB 中现存的两种租约协议进行对比。结果表明，Leader Leases **消除了冗余的心跳流量**，使租约维护的 CPU 开销降低超过 **85%**，同时保持了与过期租约（expiration leases）相同的正确性和故障容忍性。Leader Leases 还**解决了集中化租约在部分网络分区下的长期不可用问题**。

---

### 1. 引言（Introduction）

共识复制协议（Raft/Paxos）为数据库提供了强大的复制保证，但达成共识需要多数派的通信轮次。租约机制允许指定租约持有者在不进行共识轮次的情况下服务读取请求，显著降低了读取延迟并提高了吞吐量。

在多共识组架构中——即数据被分区为许多小的共识组（CockroachDB 的 Range，通常 > 10 万个）——每个组都需要独立维护其租约。这带来了可扩展性挑战：每个组的租约续约流量（Raft 心跳、周期性 Raft 日志写入）在高组数下淹没了系统。

CockroachDB 历史上使用了两种租约类型：
1. **Expiration Lease**：每个组周期性写 Raft 日志来续约租约；CPU 随组数线性增长（10 万组下 90%+ CPU）
2. **Centralized Lease**：通过一个集中式的 liveness Range 管理节点级 epoch；节省了 CPU，但存在致命缺陷——部分网络分区（partial network partition）场景下，失去 Raft 多数派的租约持有者仍能续约租约，导致该 Range **永久不可用**

本文提出 **Leader Leases**，解决了上述两难。核心思想：
- 构建一个集群级的故障检测层（**Liveness Fabric**），节点间通过双向心跳维持支持关系
- 修改 Raft 协议（**Leader Fortification**），让领导者从跟随者获得"在某个时间戳前不会投票给他人"的承诺
- 利用 fortified leadership 安全地将 leader 和 leaseholder 统一为一个角色
- 完全关闭 per-group 的 Raft 心跳

CPU 节省超 85%，彻底消除了部分网络分区下的永久不可用问题。

### 2. 背景（Background）

#### 2.1 租约（Leases）

在多共识组数据库中，租约是一种"廉价的一致性加快照"机制。租约持有者可以在租约有效期内服务读取请求而无需共识通信。租约通常通过定期续约（renewal）维持——持有者发起共识轮次以证明自己仍然具备领导条件。

#### 2.2 多共识组架构

分布式数据库为了扩展性，将数据分区为多个 range/shard，每个由独立的共识组管理。每组拥有独立的 leader/leaseholder，独立复制。这种独立性带来了跨组协调挑战——跨多个 range 的事务需要 MVCC 和 2PC 等协议来保证一致性。

#### 2.3 CockroachDB 历史租约

- **Expiration Leases**：通过 Raft 日志记录续约时间戳。故障检测与 Raft 可用性对齐，但代价高昂。
- **Centralized Leases**：通过集中式 liveness Range 管理节点 epoch。但故障检测仅依赖点对点通信到 liveness Range，可能在部分网络分区下无法正确检测 leader 不可用。

**关于时钟**：CockroachDB 使用 HLC (Hybrid Logical Clocks) 提供单键线性化，不依赖时钟同步。不确定性区间处理在事务层完成，不在租约层。

#### 2.4 多组租约的可扩展性动机

- **去中心化方案**：每个组独立续约 → 背景负载随组数爆炸
- **集中化方案**：引入单点/依赖外部可用性 → 可用性降低
- **折中方案**：面临正确性、性能和故障容忍度之间的权衡

### 3. 架构（Architecture）

#### 3.1 概览

三层架构（自底向上）：

```
┌────────────────────────────────┐
│  Leader Lease                  │
│  极薄包装：lease = fortified   │
│  leader 的 [now, LSU)          │
├────────────────────────────────┤
│  Raft（修改版）                 │
│  Fortification / LSU 计算 /    │
│  关闭心跳 / 新增持久化字段      │
├────────────────────────────────┤
│  Liveness Fabric               │
│  集群 mesh：每对节点间双向     │
│  support（epoch + expiration） │
└────────────────────────────────┘
```

每节点运行多个 Liveness Fabric 实例（每个 store 一个），以隔离磁盘故障的爆炸半径。

**Liveness Fabric 提供的关键查询**：
- `SupportFrom(ID)`："节点 ID 承诺支持我们到哪个 (timestamp, epoch)？"
- `SupportFor(ID)`："我们承诺支持节点 ID 到哪个 (timestamp, epoch)？"

#### 3.2 Leader Leases 核心概念

Leader Lease 可以视为 fortified leadership 的一个极薄包装：
- 只有 Raft leader 能持有 Leader Lease
- Lease 有效期 = leader's fortified term 内 LSU 覆盖的时间区间

**非协作获取（Non-cooperative Acquisition）**：
新 replica 必须先赢得 Raft 选举成为 leader，才能获得 lease。Fortification 保证 lease 时间区间不相交。

**协作转移（Cooperative Transfer）**：
原 leaseholder 声明放弃剩余 lease（forfeit），接任者收到 expiration lease。等 leadership 也转移后，才转为 leader lease。

#### 3.3 Raft 修改

##### 3.3.1 Leader Fortification

新增 RPC 消息：
- `MsgFortifyLeader`：leader → follower，请求 follower 承诺在某个时间戳前不发起/参与选举
- `MsgFortifyLeaderResp`：follower → leader，包含 term、ack flag、LeadEpoch

Follower 接受条件：
1. 消息 term == 自己 current term
2. Liveness Fabric 中该 follower 的节点支持 leader 的节点

Leader fortified 条件：收到 **多数派（含自己）** 成功响应。

##### 3.3.2 LeadSupportUntil (LSU)

**定义**：
```
LSU = max_{Q ∈ Quorums} min_{r ∈ Q} τ_r
```
其中 τ_r 是 replica r 对 leader 的 support 到期时间戳。

**语义**：leader 确保在 LSU 之前不会被替换。这是 Leader Lease 结束时间的直接来源。

LSU 每 500ms（Raft tick）重新计算。

##### 3.3.3 Leader De-fortification

跟随者何时停止 fortifying leader：
- **隐式**：收到更高 term 的任何消息
- **显式**：leader step down 后发送 MsgDefortify 直到所有 follower 确认，或看到更高 term 的 committed entry

##### 3.3.4 领导权转移

Leader 可以指示 follower 竞选下一 term。follower 在拉票时附带"此竞选由原 leader 发起"的元数据，被询问的 replica 可以投票尽管它们还在 fortify 原 leader。

##### 3.3.5 配置变更（Configuration Changes）

增加了新的安全约束：leader 在提出新配置变更前，必须满足 **LSU == MaxLSU**（当前 LSU 等于历史上最大 LSU）。这防止通过两次连续成员变更构造出不包含 fortified 成员的多数派。

##### 3.3.6 新增持久化 Raft 字段

- `Lead`：重启后记住自己曾在 fortify 哪个 leader
- `LeadEpoch`：fortification 对应的 Liveness Fabric epoch

防止 rebooting follower 在 fortification 承诺未到期时发起选举。

##### 3.3.7 关闭 Raft 心跳

Fortification 确立后，leader 停止向已 fortified follower 发送 Raft 心跳。故障检测完全由 Liveness Fabric 代理。历史上 CockroachDB 为此引入了 heartbeat coalescence 和 quiescence 优化——现在这些可以移除，架构更简洁。

#### 3.4 Liveness Fabric

##### 3.4.1 历史（Epoch History）

每个节点拥有线性 epoch 历史。重启或失去对某节点的 support 导致 epoch 递增（递增事件标记旧 epoch 的结束和新 epoch 的开始）。

关键规则：
- 如果 n₁ 失去 n₂ 对 epoch e₁ 的 support，n₁ 尝试用新 epoch e₃ 请求 support
- n₁ 在旧 epoch 的 support 未到期前，不能为新 epoch 请求 support → **Support Disjointness**

##### 3.4.2 数据结构与消息

每个节点维护：
- `max_epoch`：最新的 epoch 号
- `support_from[other_node]`：(epoch, expiration) 对——非持久化
- `support_for[other_node]`：(epoch, expiration) 对——**持久化**（重启不能背弃承诺）
- `max_requested`：持久化，防止在旧 epoch support 未到期时请求新 epoch
- `max_withdrawn`：持久化，防止为已撤销 epoch 重新提供 support

两种消息类型：Heartbeat（请求 support）和 Heartbeat Response（提供 support）。

三个算法：
1. **Send Heartbeat and Receive Response**：定期向所有节点发心跳；更新 `support_from`
2. **Receive Heartbeat and Send Response**：根据收到的请求更新 `support_for`；提供相同 epoch 或更高 epoch
3. **Withdraw Support**：当时钟超过 `support_for[n']` 的 expiration 时，递增 epoch 并将 expiration 置为 0

### 4. 保证（Guarantees）

**核心正确性属性**：**Lease Disjointness**——没有两个不同的 lease 在时间上重叠。

#### 4.1 Liveness Fabric 属性

- **Support Durability**：如果 n₁ 从 n₂ 收到 (epoch=e, exp=t) 的 support，则直到 n₂ 的时钟超出 t 之前，n₂ 不能撤销这个 epoch 的 support。Epoch 递增是单向的——一旦 support 被撤销，**永不复原**。
- **Support Disjointness**：同一 epoch 的不同 support 期在时间上不重叠。

这两个属性使用 TLA+ 形式化验证。

#### 4.2 Raft 属性

- **LSU 存在性**：每个 fortified term 都存在一个非零 LSU
- **LSU 前进条件**：当且仅当多数派 follower 没有撤销 support 时，LSU 前进
- **Term 结束条件**：当且仅当 leader 失去多数派 Liveness Fabric support 时，fortified leader 的 term 结束
- **新 leader 的时间上界**：新当选 leader 的时钟读数必须大于之前任何 fortified leader 的最大 LSU

#### 4.3 Lease 属性（定理 4.7）

**如果 replica A 自认为持有 Leader Lease l₁ = [t₁, t₂)，则没有其他 replica 自认为持有与 l₁ 重叠的 lease l₂。**

证明思路：
- 不同 leader 的 lease 不相交：新 leader 的 lease start time > 旧 leader 的最大 LSU ≥ 旧 lease 的 end time
- 同一 leader 的连续 lease 不相交：由 Support Disjointness 保证

### 5. 评估（Evaluation）

配置：6 节点 GCP n2-standard-8（故障恢复测试）/ 3 节点（CPU 测试）/ 5 节点 n2-standard-16（TPC-C）。所有 lease 类型统一 3s lease 时长，1s 续约间隔。

#### 5.1 故障恢复时间

| 故障类型 | Expiration | Centralized | Leader |
|----------|-----------|-------------|--------|
| Node crash (P50/P99) | 3.0s / 4.5s | 3.0s / 4.5s | **4.0s / 6.5s** |
| Full partition | ~3.5s | ~3.5s | ~4.0-4.5s |
| **Partial partition** | ✅ 恢复 | ❌ **永不恢复** | ✅ 恢复 |
| **Disk stall** | ❌ 不可用 | ❌ 不可用 | ✅ 恢复 |

Leader Lease 在节点崩溃/全分区下慢 1-2 秒——这是因为 lease 到期后才能开始竞选。但这是可接受的权衡，换来的是 partial partition 和 disk stall 下的可靠性。

**Disk stall 的处理**：Leader Lease 通过要求 Liveness Fabric 心跳前做同步磁盘写来解决——磁盘 stall → 无法发心跳 → 失去 fortification → 触发 leadership 变更。用 3s lease 时长 + 1s 心跳间隔确保至少 2 次心跳失败才触发变更（避免瞬时抖动误报）。

#### 5.2 CPU 开销

空闲集群，10K→100K Range：
- **Expiration**：15% (10K) → 90%+ (80K) ——堆栈分析确认大部分 CPU 被 per-Range timer 和 Raft 活动消耗
- **Centralized**：始终 ≤5% ——节点级 lease + quiescence 使几乎没有背景活动
- **Leader Lease**：~0% → <15% ——介于两者之间，quiescence 让 follower 休眠但 leader 仍需 tick

论文计划为 Leader Leases 实现 leader quiescence，预期届时 CPU 开销将接近 Centralized 水平。

#### 5.3 TPC-C 基准

固定 5 节点集群，25K→100K warehouse：
- **Expiration**：25K 时 100K tpmC → 100K 时下降 ~20%（lease 维护抢 CPU）
- **Centralized / Leader Lease**：吞吐基本平稳

#### 5.4 Liveness Fabric CPU 开销

150 节点 × 12 stores = 1800 stores：
- 仅消耗 0.225 cores（~2.8% of 8 vCPU）
- 线性可扩展

### 6. 相关工作

- **多共识组系统**：Spanner、TiDB、OceanBase、Yugabyte 都使用类似架构，但均未系统性地解决 lease 可扩展性
- **Spanner 10s lease**：通过隐式续约（每个成功写都可延 lease votes），但仍有 per-group 成本。对写流量少的 group 问题依旧
- **TiDB 租约**：基于 Raft leadership 让 follower 承诺不发起选举——类似思想但未解决大规模组的可扩展性
- **OceanBase PALF**：通过聚合多个共识组到一个复制组来降低成本，但代价是放置灵活性降低和爆炸半径增大
- **经典租约工作**：Chubby、ZooKeeper 中的 lease 使用，但面向单组场景

**成本对比**：
- per-group heartbeat（k=5 replica）：N × 40K × 5 = O(N)
- Leader Leases (Liveness Fabric)：O(N²)
- 在 200K store 以下的集群中，N² 更优
- 大容量硬盘趋势（20TiB）意味着 per-node shard 数增加，进一步放大 per-group 方案的成本

### 7. 经验教训（Lessons Learned）

#### 7.1 紧耦合 leader 与 leaseholder
CockroachDB 历史上将 leader 和 leaseholder 分离。这在 partial partition 下产生难以预料的故障模式：leaseholder 可以续约租约却无法向 Raft leader 提议写入 → 永久不可用。统一二者消除了这一整类问题。

#### 7.2 Disk stall 阻塞 Liveness Fabric
磁盘既不死也不活、无限阻塞的故障模式危险——leader 可能看似健康实则无法提交任何写入。解决方案：Liveness Fabric 心跳请求和响应都要求先完成同步磁盘写。磁盘 stall → 无法心跳 → 被迫失去 leadership。

#### 7.3 Store 间心跳（而非节点间心跳）
CockroachDB 支持每节点挂多块磁盘（多 store）。心跳在 store 而非 node 粒度进行——增加消息数，但将磁盘故障的爆炸半径限制在单个 store 而非整节点。通过批量化优化将发往同一节点多个 store 的心跳合并为一条消息。

---

### 8. 结论（Conclusion）

Leader Leases 通过 **Liveness Fabric**（可扩展故障检测层）和 **Leader Fortification**（增强型 Raft 领导保证），将 leader 和 leaseholder 统一，使 CockroachDB 在保持正确性的同时消除了冗余心跳。

**关键收益**：
- CPU 开销降低 85%+
- 消除了 partial partition 永久不可用问题
- 简化了架构（移除 heartbeat coalescence、quiescence 等复杂性）
- Liveness Fabric 水平可扩展至 150+ 节点

Leader Leases 证明：**可扩展的故障检测和稳定的领导可以共存于实际的生产级百万共识组共识系统中**。
