# Agent 记忆管理规范

> [!important] 负责人
> **CTO** 负责制定和修改。所有 Agent 必须遵守。

## 一、记忆分层模型

```
L0: 会话上下文     ← context compaction 保留，不持久化
L1: 日常记忆       ← memory/YYYY-MM-DD.md，Dreaming 的原材料
L2: 长期记忆       ← MEMORY.md，Dreaming 自动晋升
L3: 共享记忆       ← memory/shared/，手动/Git 同步给需要的人
```

| 层级 | 文件 | 写入方式 | 检索方式 | 可见性 |
|------|------|----------|----------|--------|
| L0 | 无（内存） | 自动（compaction） | 不可检索 | 仅当前 Session |
| L1 | `memory/YYYY-MM-DD.md` | Agent 自动 | `memory_search` | Agent 自身 |
| L2 | `MEMORY.md` | Dreaming 自动晋升 | `memory_search` + `memory_get` | Agent 自身 |
| L3 | `memory/shared/` | Agent 手动 | `memory_search`（同 workspace） | Git 同步后对目标可见 |

## 二、可见性规则

### 2.1 默认隔离原则

每个 Agent 的 `memory/` 目录和自己的 `MEMORY.md` **仅自己可检索**。
Agent A 无法用 `memory_search` 搜到 Agent B 的记忆。

### 2.2 向上汇报（执行层 → 专家 → VP）

- 执行层完成任务后，在返回消息中附带"关键记忆片段"
- 上级收到后，如需保留，写入自己的 `MEMORY.md`
- 不共享 raw 记忆文件，只共享提炼后的摘要

### 2.3 同级共享（VP ↔ VP）

- 通过 `memory/shared/` 目录 + Git 同步
- CTO 的 `memory/shared/` 可被 CEO 拉取查看
- VP 之间通过 `sessions_send` 推送摘要

### 2.4 私有记忆

每个 Agent 可以建 `memory/private/` 目录存放敏感数据（如 CFO 的预算细节）。
此目录不应提交到 Git。

## 三、Dreaming 自动晋升

OpenClaw `memory-core` 插件提供 Dreaming 机制，自动将 L1→L2：

- **Light phase**：整理近期日常记忆，去重，评分
- **Deep phase**：高分候选写入 `MEMORY.md`
- **REM phase**：提取主题和模式

CTO 定期 review `DREAMS.md` 确认晋升质量。不需要手动干预。

## 四、冷启动流程

Agent 启动/重启后自动执行：

1. OpenClaw 自动加载 `MEMORY.md` + 当天/昨天的 `memory/YYYY-MM-DD.md`
2. AGENTS.md 冷启动流程：git pull → 加载规范 → `memory_search` 最近任务
3. （Phase 3 完成后）从 SQLite 精确恢复未完成任务状态

## 五、文件规范

### 5.1 MEMORY.md 格式

- 用标准 Markdown，分 section 组织（角色、项目、决策、技术栈）
- 写事实而非流水账——"做了什么决策"比"说了什么话"更有价值
- 每次重大决策后更新

### 5.2 memory/YYYY-MM-DD.md 格式

- 以 `# YYYY-MM-DD Memory` 开头
- 用 `## Section` 分组（如 待办事项、关键对话、进度记录）
- Dreaming 依赖结构化 section 做分类

### 5.3 记忆写入时机

| 时机 | 写入位置 | 触发者 |
|------|----------|--------|
| 日常对话 | `memory/YYYY-MM-DD.md` | Agent 自动 |
| 任务完成 | `memory/YYYY-MM-DD.md` + 向上汇报摘要 | Agent 自动 |
| 重大决策 | `MEMORY.md` | Agent 主动 |
| 发现需共享的信息 | `memory/shared/` | Agent 手动 |
| Compaction 前 | `memory/YYYY-MM-DD.md`（保存当前上下文） | Agent 自动 |

## 六、观测（支线，Phase 4 实现）

`memory_ops` 表记录所有 memory_search/memory_get 调用，供 Dashboard 可视化：

```sql
CREATE TABLE memory_ops (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    agent_id     TEXT NOT NULL,
    task_id      TEXT,
    op_type      TEXT NOT NULL,   -- 'search' | 'get'
    query        TEXT,
    path         TEXT,
    result_count INTEGER,
    created_at   TEXT DEFAULT (datetime('now'))
);
```

此表不参与记忆系统的核心运行，纯粹为观测目的。
