# 论文概览

- **论文全名**：Scalable Leader Leases For Multi Consensus Groups in CockroachDB
- **作者**：Ibrahim Kettaneh, Tsvetomira Radeva, Arul Ajmani, Sumeer Bhola, Nathan VanBenschoten, Rebecca Taft
- **机构**：Cockroach Labs
- **会议**：SIGMOD Companion '26, Bengaluru, India
- **页数**：13 页
- **核心洞见**：**将 Raft leader 和 leaseholder 统一，通过一个共享的集群级故障检测层（Liveness Fabric）替代每个 Raft group 独立的心跳/lease 续约**，使 lease 维护成本从 O(N_groups) 降到 O(N_nodes²)。

---

## 上下文（Background）

### 这个方向为什么重要？

现代分布式数据库（CockroachDB、Spanner、TiDB、OceanBase、Yugabyte）都将数据分割为数百万个 Raft consensus group（Range/Shard），每个 group 需要一个 leaseholder 来高效服务强一致读。如果每个 group 独立维护自己的 lease，在几十万个 group 的规模下，lease 续约的 CPU 和网络开销将淹没系统。

### 工业实践中的痛点

- **生产集群中 60%+ 的 Raft group 长时间只有读活动**——Raft 级的心跳和 lease 续约对这些 group 纯粹是浪费
- **Expiration Lease**：每个 Range 独立通过 Raft 日志续约 lease → CPU 随 Range 数线性增长，10 万 Range 时达 90%+ CPU
- **Centralized Lease**：通过集中式 liveness Range 统一管理节点 epoch → CPU 极低，但存在 **partial network partition 时永久不可用**的致命缺陷（leaseholder 心跳正常但无法与 quorum 通信）
- **Spanner 10s lease**：用更长 lease 降低续约频率，但增加故障恢复时间

### 现有方案的不足

| 方案 | 优势 | 劣势 |
|------|------|------|
| Expiration Lease | 简单、Raft 对齐 | CPU 随 Range 数线性增长 |
| Centralized Lease | CPU 极低 | Partial partition → 永久不可用 |
| 长 lease（如 Spanner 10s） | 减少续约 | 故障恢复慢 |
| Raft 论文原版 lease | 理论优雅 | 依赖时钟漂移上界，实际级需要 per-group heartbeat |

## 贡献（Contributions）

1. **Liveness Fabric**：一个去中心化、心跳驱动的集群级故障检测服务，跟踪节点间有向边的健康状态。通过双向心跳建立"支持"(support) 语义，使故障检测从 O(N_groups) 降到 O(N_nodes²)
2. **Leader Fortification（Raft 修改）**：通过 MsgFortifyLeader/MsgFortifyLeaderResp 消息，让 leader 从 followers 获取"在某个时间戳前不会投票给其他人"的承诺，提供**更强的领导稳定性保证**，并且允许关闭 Raft 心跳
3. **Leader Lease（统一 leaseholder + leader）**：利用 fortified leadership 保证，安全地将 leaseholder 和 Raft leader 合并为同一角色，消除之前因角色分离导致的 partial partition 不可用问题
4. **全面的实验验证**：对比 Expiration/Leader/Centralized 三种 lease 在故障恢复、CPU 开销、TPC-C 吞吐下的表现

## 方法（Approach）

### 整体架构（Layer Design）

```
┌──────────────────────────────┐
│   Leader Lease               │  ← 极薄包装：lease = fortified leader's [now, LSU)
├──────────────────────────────┤
│   Raft（修改版）              │  ← Fortification / LSU 计算 / 关闭心跳
├──────────────────────────────┤
│   Liveness Fabric            │  ← 集群 mesh 心跳：每对 store 间双向 support
└──────────────────────────────┘
```

### 关键机制

#### 1. Liveness Fabric

**核心理念**：节点 n₁ → n₂ 的有向边维护一个 support 关系。n₁ 请求 n₂ 支持它到一个时间戳，n₂ 承诺在这个时间戳前持续支持 n₁。

**关键数据结构**（每个节点）：
- `max_epoch`：节点的 epoch 计数器，任何不可用事件（重启、失去 support）都会递增
- `support_from[other_node]`：(epoch, expiration) 对——我从对方收到的支持；非持久化
- `support_for[other_node]`：(epoch, expiration) 对——我承诺给对方的支持；**持久化**（重启不能背弃承诺）
- `max_requested`：持久化，防止在旧 epoch 的 support 尚未到期时请求新 epoch 的 support
- `max_withdrawn`：持久化，防止为已撤销的 epoch 重新提供 support

**两个核心性质**：
- **Support Durability**：如果 n₁ 从 n₂ 收到(epoch=e, exp=t) 的支持，那么直到 n₂ 的时钟超过 t 之前，n₂ 不能撤销对这个 epoch 的支持
- **Support Disjointness**：同一 epoch 的不同 support 周期在时间上不重叠

**心跳间隔**：每 1 秒向所有 nodes 发送。每对 store 间独立运行 Liveness Fabric 实例（一个节点可以有多个 store）。

#### 2. Leader Fortification

**问题**：标准 Raft 中，leader 不知道自己是否被替换——只有通过心跳确认多数派才能知道。这导致 leader 无法提供"我在时间 X 之前都还是 leader"的保证。

**方案**：增加两个 RPC 消息：
- `MsgFortifyLeader`：leader → follower，请求其承诺在某个时间戳前不发起/参与选举
- `MsgFortifyLeaderResp`：follower → leader，包含自己的 term、ack flag、以及 LeadEpoch（Liveness Fabric 中对应的 epoch）

**Fortification 条件**：follower 接受请求需满足：
1. 消息中的 term == 自己的 current term
2. 在 Liveness Fabric 中该 follower 的节点支持 leader 的节点

**Fortified 条件**：leader 收到 **多数派（含自己）** 的成功响应即认为 fortified

#### 3. LeadSupportUntil (LSU)

**定义**：LSU = max_{Q∈Quorums} min_{r∈Q} τ_r，其中 τ_r 是 replica r 对 leader 的 support 过期时间。

**语义**：leader 可以保证在 LSU 之前自己的 leadership 不会被取代。这是 Leader Lease 的 lease 结束时间的直接来源。

**配置变更安全**：增加 MaxLSU 约束——leader 在提出新配置变更前必须满足 LSU == MaxLSU，防止通过两次连续变更构造出不包含 fortified 成员的多数派。

#### 4. 关闭 Raft 心跳

Fortification 确立后，leader 停止向已 fortified 的 follower 发送 Raft 心跳，完全依赖 Liveness Fabric 进行故障检测。这是 CPU 节省的核心来源——心跳从 O(N_groups × N_replicas) 降到 O(N_nodes × N_nodes)。

#### 5. Leader Lease = Fortified Leadership

- leader lease 只能由 Raft leader 持有
- Lease 有效期 = leader 的 fortified term 内 LSU 覆盖的时间
- Non-cooperative acquisition：新 leader 必须先赢得选举才能获得 lease；fortification 保证了 lease 时间区间不相交
- Cooperative transfer：leader 宣布放弃剩余 lease，接任者收到 expiration lease，等 leadership 转移后才转为 leader lease

## 评估（Evaluation）

### 实验设置
- GCP n2-standard-8/16 实例
- CockroachDB v25.4.0 pre-release
- Lease 时长统一设为 3s，续约间隔 1s
- 对比三种 lease 类型：Expiration / Centralized / Leader

### 关键发现

#### 1. 故障恢复时间
| 故障类型 | Expiration | Centralized | Leader |
|----------|-----------|-------------|--------|
| Node crash (P50) | 3.0-3.9s | 3.0-3.9s | 4.0-4.7s |
| Full partition (P50) | 3.5s | 3.5s | 4.0-4.5s |
| Partial partition | ✅ 恢复 | ❌ **永久不可用** | ✅ 恢复 |
| Disk stall | ❌ 不可用 | ❌ 不可用 | ✅ 恢复 |

Leader Lease 比 expiration lease 慢 1-2s（因为 lease 到期才能开始选举），但换来了 partial partition 和 disk stall 下的可靠性。

#### 2. CPU 开销（与 Range 数关系）
- Expiration lease：100K Range → 90%+ CPU
- Leader Lease：100K Range → ~15% CPU
- Centralized：100K Range → ~5% CPU
- **Leader Lease 比 Expiration 省 85% CPU**

#### 3. TPC-C 吞吐
- Expiration lease：100K warehouse → 吞吐下降 20%（lease 维护抢 CPU）
- Leader Lease / Centralized：吞吐基本平稳

#### 4. Liveness Fabric 可扩展性
- 150 节点 × 12 stores = 1800 stores → 仅 0.225 cores（~2.8% of 8 vCPU）
- 线性可扩展

## 批判性思考（Critical Analysis）

### 前提假设是否合理？

1. **时钟不需要同步** ✅：论文明确说依赖 HLC (Hybrid Logical Clocks)，只有 monotonic clock requirement，不依赖时钟同步。合理。
2. **Liveness Fabric 的 O(N²) 心跳** ✅：N 在几百量级，O(N²) 可接受。论文给出了清晰的计算：N × 40K × 5 (per-group heartbeat) vs N²，在 200K store 集群以下 N² 更优。
3. **重启后 follower 必须记住 fortification 承诺** ✅：论文新增持久化字段 Lead/LeadEpoch 来处理这个问题。
4. **Disk stall 检测** ⚠️：要求 Liveness Fabric 心跳发出前先做同步磁盘写。这引入了磁盘 I/O 到故障检测路径中——如果磁盘真的不稳定（但还没完全 stall），可能频繁触发误报。

### 与现有知识库的关联

1. **[[事务模型深度调研]]**：Leader Lease 是 CockroachDB 事务栈的基础层——它提供了单 key 线性化读的保证。事务层（MVCC + 2PC + HLC）构建于其上。论文明确了 HLC 不确定性区间不在 lease 层处理。
2. **[[分布式数据系统一致性体系综述]]**：Leader Lease 属于"强一致性快速路径"的经典模式——通过临时授权消除共识协议的通信开销。这篇论文的创新在于把这个问题从"单 group"扩展到了"百万 group"。
3. **[[Raft-共识协议]]**：论文对 Raft 做了实质性的修改（Fortification / 心跳关闭 / 持久化新字段），这对任何 Raft 实现者都有参考价值。

### 可借鉴点 vs 潜在新坑

**可借鉴**：
- ✅ Liveness Fabric 作为共享故障检测层的设计模式——可推广到任何需要管理大量共识组的系统
- ✅ Fortification 的思想——让 leader 通过"多数派承诺"获得确定性，而不是依赖心跳超时的随机性
- ✅ 统一 leader 和 leaseholder 消除 partial partition 不可用问题

**潜在新坑**：
- ⚠️ 故障恢复延迟略高（额外 1-2s）——对 ultra-low-latency failover 场景可能不够
- ⚠️ Leader Leases 下 leader replica 不能 quiesce（仍需 tick 保持 leadership）——论文自己承认并计划 future work
- ⚠️ 配置变更约束 (LSU == MaxLSU) 增加了变更的延迟成本——需要在生产中有足够的时间窗口
