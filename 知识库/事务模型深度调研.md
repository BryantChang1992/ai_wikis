## 1. 事务基础概念

### 1.1 ACID 特性

事务（Transaction）是数据库操作的最小逻辑单元，具备四个基本特性：

| 特性 | 全称 | 含义 | 实现机制 |
|------|------|------|----------|
| **A** | Atomicity（原子性） | 事务所有操作要么全部成功，要么全部失败回滚 | Undo Log / Rollback |
| **C** | Consistency（一致性） | 事务执行前后，数据库从一致状态变换到另一个一致状态 | 约束检查 + AID 保障 |
| **I** | Isolation（隔离性） | 并发事务之间互不干扰，每个事务看到的数据视图一致 | 锁 / MVCC / 时间戳 |
| **D** | Durability（持久性） | 事务一旦提交，其结果永久保存，即使系统崩溃也不丢失 | Redo Log / WAL |

> **注意**: C（一致性）是 A、I、D 共同保证的最终结果，数据库本身通过约束（主键、外键、CHECK 等）辅助保障。

```python
# ACID 事务的典型代码模式
try:
    BEGIN
    debit(account_a, 100)     # 扣款
    credit(account_b, 100)    # 入账
    COMMIT
except:
    ROLLBACK  # 任何失败都回滚，保证 A
```

### 1.2 隔离级别与并发问题

SQL 标准定义了四种隔离级别，用于在并发控制开销与数据正确性之间做权衡：

| 隔离级别 | 脏读 (Dirty Read) | 不可重复读 (Non-Repeatable Read) | 幻读 (Phantom Read) |
|----------|:-----------------:|:-------------------------------:|:-------------------:|
| **Read Uncommitted** | ✅ 可能 | ✅ 可能 | ✅ 可能 |
| **Read Committed** | ❌ 禁止 | ✅ 可能 | ✅ 可能 |
| **Repeatable Read** | ❌ 禁止 | ❌ 禁止 | ✅ 可能（MySQL InnoDB 通过 Gap Lock 解决） |
| **Serializable** | ❌ 禁止 | ❌ 禁止 | ❌ 禁止 |

**三种典型并发问题**：

```
-- 脏读：事务 A 读到事务 B 未提交的数据
Time  T1: UPDATE accounts SET balance=500 WHERE id=1  -- 未提交
      T2: SELECT balance FROM accounts WHERE id=1      -- 读到 500
      T1: ROLLBACK                                     -- 实际回滚了

-- 不可重复读：同一事务内两次读到的值不同（其他事务 UPDATE 导致）
Time  T1: SELECT balance FROM accounts WHERE id=1      -- 读到 100
      T2: UPDATE accounts SET balance=200 WHERE id=1; COMMIT
      T1: SELECT balance FROM accounts WHERE id=1      -- 读到 200

-- 幻读：同一事务内两次查询的行数不同（其他事务 INSERT/DELETE 导致）
Time  T1: SELECT * FROM accounts WHERE balance > 100   -- 返回 5 行
      T2: INSERT INTO accounts VALUES (6, 150); COMMIT
      T1: SELECT * FROM accounts WHERE balance > 100   -- 返回 6 行
```

## 2. 单机事务与并发控制

### 2.1 两阶段锁（2PL: Two-Phase Locking）

2PL 是经典的基于锁的并发控制协议。核心规则：

1. **Growing Phase（增长阶段）**: 只能获取锁，不能释放锁
2. **Shrinking Phase（收缩阶段）**: 只能释放锁，不能获取锁

```
时序图（Strict 2PL，实际使用最多的变体）:

获取锁阶段          释放锁阶段
   ↓                  ↓
  Lock(A)  Lock(B)    Unlock(A) Unlock(B)
    |--------|-----------|---------|
              ↑ 事务在此持有所有锁
```

**锁类型与兼容矩阵**：

|  | S (共享锁) | X (排他锁) |
|--|:---------:|:---------:|
| **S** | ✅ 兼容 | ❌ 冲突 |
| **X** | ❌ 冲突 | ❌ 冲突 |

**Strict 2PL**（严格两阶段锁）额外要求：排他锁必须持有到事务 COMMIT 之后才释放，保证了恢复时的可串行化。

**核心缺点**：
- **死锁风险**: 事务间互相等待对方持有的锁
- **读写互斥**: 读操作也需要获取共享锁，与写操作冲突
- **长事务阻塞**: 一个长事务会阻塞所有等待者

### 2.2 MVCC（Multi-Version Concurrency Control）

MVCC 通过保存数据的多个版本来实现读写互不阻塞，是当前主流数据库的标配。

**核心思想**：
- 写操作创建新版本，而不是原地修改
- 读操作根据事务快照读取对应版本
- 旧版本数据在不再需要时由垃圾回收（GC / Vacuum）清理

```
MVCC 版本链示意（PostgreSQL）:

元组头部（tuple header）存储:
  xmin: 创建该版本的事务 ID
  xmax: 删除/更新该版本的事务 ID

Row(id=1, value=100):
  v3: xmin=Tx5, xmax=NULL   ← 当前版本 (Tx5 改为 300)
  v2: xmin=Tx3, xmax=Tx5    ← Tx3 改为 200
  v1: xmin=Tx1, xmax=Tx3    ← Tx1 插入初始值 100

事务可见性判断（简化）:
  visible_if = (xmin < my_xid AND xmin IS COMMITTED)
               AND (xmax IS NULL OR xmax > my_xid OR xmax IS ROLLED BACK)
```

**MVCC 快照隔离（Snapshot Isolation）**：
- 每个事务在开始时获取一个快照时间戳
- 事务只能看到在它开始之前已提交的数据
- 实现了比 Serializable 弱但比 Read Committed 强的隔离

**核心优势**：
- 读不阻塞写，写不阻塞读
- 死锁概率大幅降低
- 对读多写少的 OLTP 场景友好

**核心代价**：
- 存储空间增大（多版本）
- 需要定期 GC/Vacuum 清理旧版本
- 长事务可能导致版本链过长

### 2.3 WAL（Write-Ahead Logging）预写日志

WAL 是保障持久性（Durability）和崩溃恢复的核心机制。

**规则**: 对数据页的修改必须先在日志中记录，日志落盘后才能修改数据页。

```
写入路径:
  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ User SQL │───▶│ WAL 日志 │───▶│ 内存 Buffer │──▶│ 数据页   │
  │  请求    │    │ (顺序写) │    │  Pool     │    │ (异步刷盘)│
  └──────────┘    └──────────┘    └──────────┘    └──────────┘
                       │
                    fsync()
                       ▼
                  ┌──────────┐
                  │ 磁盘日志  │  ← 持久化完成，事务可提交
                  └──────────┘
```

**Redo Log vs Undo Log**:

| 日志类型 | 用途 | 记录内容 | 典型实现 |
|----------|------|----------|----------|
| **Redo Log** | 崩溃恢复（重放已完成但未刷盘的操作） | 物理/逻辑修改：`page_id=N, offset=O, old=A, new=B` | InnoDB Redo |
| **Undo Log** | 事务回滚 + MVCC 读历史版本 | 逆向操作：`将 row(id=1) 的 value 恢复为 100` | InnoDB Undo |

### 2.4 ARIES 崩溃恢复算法

ARIES（Algorithm for Recovery and Isolation Exploiting Semantics）是工业界最广泛采用的恢复算法，InnoDB、SQL Server 都基于它。

**三个核心原则**：

1. **WAL (Write-Ahead Logging)**: 先写日志再写数据
2. **Repeating History During Redo**: 恢复时重放所有操作（包括未提交事务的）
3. **Logging Changes During Undo**: 回滚未提交事务时也要记日志

**恢复三步走**：

```
Phase 1: Analysis（分析阶段）—— 从上一个 Checkpoint 开始扫描日志
  → 确定哪些事务已提交，哪些未提交
  → 构建脏页表（Dirty Page Table）和活跃事务表

Phase 2: Redo（重做阶段）—— 从最早的脏页 LSN 开始
  → 重放所有日志，恢复崩溃时丢失的数据页修改
  → 包括已提交和未提交的事务

Phase 3: Undo（回滚阶段）—— 从日志末尾向前扫描
  → 回滚所有未提交事务的操作
  → 对每个 Undo 操作也记录 CLR (Compensation Log Record)
```

**LSN (Log Sequence Number)** 是 ARIES 的心脏：
- 每个日志记录有全局递增的 LSN
- 每个数据页记录最后一次修改它的 LSN（pageLSN）
- Redo 时：`if log_record.LSN > page.pageLSN → 应用该修改`

## 3. 分布式事务协议

当数据分布在多个节点上，单机事务机制不再够用，需要分布式事务协议。

### 3.1 2PC（Two-Phase Commit，两阶段提交）

2PC 是最经典的原子提交协议，引入**协调者（Coordinator）**角色。

```
Phase 1: Prepare（准备阶段）
  Coordinator → Participants: "Prepare?"
  参与者执行 WAL 写入，锁定资源，回复 "Yes" 或 "No"

Phase 2: Commit/Abort（提交/中止阶段）
  Coordinator: 若全员 "Yes" → "Commit"
              若有任一 "No" 或超时 → "Abort"
  Participants: 执行 Coordinator 的指令，释放锁
```

```
时序列（成功场景）:

Coordinator        Participant A     Participant B
    │                    │               │
    │──── Prepare ─────▶│               │
    │──── Prepare ──────────────────────▶│
    │                    │               │
    │◀─── Yes ──────────│               │
    │◀─── Yes ───────────────────────────│
    │                    │               │
    │──── Commit ──────▶│               │
    │──── Commit ────────────────────────▶│
    │                    │               │
    │◀─── ACK ──────────│               │
    │◀─── ACK ───────────────────────────│
```

**XA 规范**：X/Open 组织定义的 2PC 标准接口。大多数关系型数据库（MySQL、PostgreSQL、Oracle）都支持 XA。

```sql
-- XA 事务示例（MySQL）
XA START 'tx_001';
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
XA END 'tx_001';
XA PREPARE 'tx_001';   -- Phase 1: 进入 Prepared 状态

XA START 'tx_001';
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
XA END 'tx_001';
XA PREPARE 'tx_002';

XA COMMIT 'tx_001';     -- Phase 2: 提交
XA COMMIT 'tx_002';
```

**优点**：
- 协议简单，实现标准统一
- 强一致性保障
- 大量数据库原生支持

**缺点**：
- **同步阻塞**: Prepare 后参与者必须等待 Coordinator 决策，期间资源锁定
- **单点故障**: Coordinator 崩溃导致整个系统阻塞
- **脑裂风险**: Coordinator 恢复后数据可能不一致（需要协调日志）
- **性能差**: 多次网络往返 + 同步写入

### 3.2 3PC（Three-Phase Commit，三阶段提交）

3PC 在 2PC 基础上增加一个阶段，并引入超时机制，试图解决阻塞问题：

```
Phase 1: CanCommit   → "你是否可以提交？"（资源检查，不锁定）
Phase 2: PreCommit   → "准备提交！"（锁资源，写 WAL）
Phase 3: DoCommit    → "执行提交！"
```

**关键改进**：
- 参与者可在 Coordinator 失联后超时自动提交（如果已进入 PreCommit）
- 减少同步阻塞的概率

**为什么 3PC 没有被广泛采用**：
- 只解决了**无分区**场景的阻塞，网络分区时仍然可能不一致
- 多一轮网络往返，延迟更高
- FLP 不可能性：在异步网络中无法完全解决 Coordinator 故障
- 2PC + Paxos/Raft（快速恢复 Coordinator）是更务实的方案

### 3.3 TCC（Try-Confirm-Cancel）

TCC 是**业务层的补偿事务**模式，将每个资源操作拆为三步：

```
Try（尝试）:     预留资源（检查+锁定），例如冻结余额
Confirm（确认）: 真正执行业务操作，例如扣款
Cancel（取消）:  释放已预留资源，例如解冻余额
```

```java
// TCC 转账示例（伪代码）
class TransferTcc {
    // Try 阶段
    boolean try() {
        fromAccount.freeze(100);  // 冻结 100 元
        toAccount.freeze(100);    // 预留 100 元额度
        return true;
    }

    // Confirm 阶段
    void confirm() {
        fromAccount.deductFrozen(100);  // 真实扣款
        toAccount.addBalance(100);      // 真实入账
    }

    // Cancel 阶段（try 失败时调用）
    void cancel() {
        fromAccount.unfreeze(100);      // 解冻
        toAccount.unfreeze(100);
    }
}
```

**与 2PC 的核心差异**：

| 维度 | 2PC | TCC |
|------|-----|-----|
| **层面** | 资源层（数据库协议） | 业务层（应用代码） |
| **锁定粒度** | 事务期间全程锁 | Try 后释放，Confirm 无需锁 |
| **性能** | 差（同步阻塞） | 好（非阻塞） |
| **实现成本** | 低（数据库原生） | 高（需要为每个接口写三套逻辑） |
| **一致性** | 强一致 | 最终一致 |

**适用场景**: 资金交易（冻结/解冻模式）、库存扣减（预占模式）、高并发业务。

### 3.4 SAGA 模式

SAGA（Saga）是处理长事务（Long-Lived Transaction, LLT）的模式，把一个大事务拆分为一系列有序的本地事务。

**两种协调方式**：

```
编排模式 (Choreography)           管控模式 (Orchestration)
  Service A ──▶ Service B           Orchestrator
    │              │                 /    |    \
    │◀─────────────│               A     B     C
    ▼              ▼               │     │     │
  Service C ──▶ Service D         compensate / retry
```

**补偿机制核心**：
```
正向: T1 → T2 → T3 → ... → Tn
反向: Cn → ... → C3 → C2 → C1  (每个 Ti 有对应的补偿 Ci)

例如订单流程:
  T1: 创建订单      C1: 取消订单
  T2: 扣减库存      C2: 恢复库存
  T3: 扣款          C3: 退款
  T4: 更新积分      C4: 扣减积分
```

```python
# SAGA 编排器伪代码
class OrderSagaOrchestrator:
    def execute(self, order):
        steps = [
            ('create_order', 'cancel_order', lambda: order_service.create(order)),
            ('reserve_stock', 'release_stock', lambda: stock_service.reserve(order.items)),
            ('charge_payment', 'refund_payment', lambda: payment_service.charge(order.amount)),
        ]
        executed = []
        for name, compensate_fn, action in steps:
            try:
                result = action()
                executed.append((compensate_fn, result))
            except Exception:
                self._compensate(executed)  # 反向补偿
                raise
        return True

    def _compensate(self, executed):
        for compensate_fn, ctx in reversed(executed):
            compensate_fn(ctx)
```

**SAGA vs TCC**：

| 维度 | SAGA | TCC |
|------|------|-----|
| 事务长度 | 长事务（分钟~小时级） | 短事务（秒级） |
| 资源占用 | 无锁定，补偿式 | Try 阶段暂用资源 |
| 实现模式 | 编排/管控 | Try-Confirm-Cancel 三元组 |
| 隔离性 | 弱（事务间可看到中间状态） | 较好（Try 隔离业务资源） |

### 3.5 事务消息（Transactional Outbox）

以 RocketMQ 为代表的事务消息，保证本地事务与消息发送的原子性：

```
生产者流程:
  ┌─────────────┐
  │ 1. 发送半消息 │──▶ Broker (消息不可见)
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 2. 执行本地事务│
  └──────┬──────┘
     ┌───┴───┐
  Commit    Rollback
     │         │
     ▼         ▼
  消息可见   消息删除

  ┌─────────────────────────────┐
  │ 3. Broker 定期回查（未决事务）│
  │    → 调用 Producer 的 check  │
  │    → 决定 Commit 或 Rollback │
  └─────────────────────────────┘
```

**本地消息表**模式（无事务消息支持时的变通）：

```sql
-- 在同一个本地事务中
BEGIN;
INSERT INTO orders (...) VALUES (...);           -- 业务操作
INSERT INTO outbox (topic, payload, status) ... -- 待发送消息
COMMIT;

-- 独立进程轮询 outbox 表，发送消息后标记已发送
SELECT * FROM outbox WHERE status = 'PENDING';
```

## 4. 现代分布式事务架构

### 4.1 Percolator（乐观锁 + 时间戳 + 2PC）

Google Percolator（2010 OSDI）是一个建在 Bigtable 之上的增量处理系统，其事务模型被 TiDB/TiKV 借鉴并成为 NewSQL 领域的标杆方案。

**核心设计**：
- **无中心化事务管理器**: 客户端参与 2PC，去中心化
- **乐观并发控制**: 提交时才检测冲突
- **时间戳**: 基于 TSO（Timestamp Oracle）分配全局递增时间戳

**数据模型（Bigtable 列族）**：

```
Key: "row_key"
  Column "data"     → 当前值
  Column "lock"     → 分布式锁（primary key 指向）
  Column "write"    → commit 时间戳记录
```

**Percolator 事务流程**：

```
Phase 1: Prewrite（预写）
  ┌──────────────────────────────────────────────────────────┐
  │ 1. 从 TSO 获取 start_ts                                  │
  │ 2. 选一个 key 作为 primary，其余为 secondary             │
  │ 3. 对所有 key 检查写写冲突:                               │
  │    if 存在 write_record.ts > start_ts  → 冲突，abort     │
  │ 4. 对 primary 写 lock 列                                 │
  │ 5. 对所有 key 写 data 列（带 start_ts）                   │
  └──────────────────────────────────────────────────────────┘

Phase 2: Commit
  ┌──────────────────────────────────────────────────────────┐
  │ 1. 从 TSO 获取 commit_ts                                 │
  │ 2. 对 primary 写 write 列（commit_ts + start_ts）        │
  │     ← 此操作成功标志事务提交                              │
  │ 3. 异步删除 primary lock                                 │
  │ 4. 异步清理 secondary 的 lock 和 write                   │
  └──────────────────────────────────────────────────────────┘
```

**冲突检测与锁清理**：

```python
# Percolator 读操作伪代码
def read(key, start_ts):
    # 从最新版本向前扫描
    for version in range(latest, 0, -1):
        if version.write_ts <= start_ts:
            # 检查该版本对应的事务是否提交
            if version.write_ts >= version.start_ts:
                return version.data  # 已提交
        # 检查是否有锁
        if version.lock is not None:
            lock_ts = version.lock.start_ts
            if lock_ts < start_ts:
                # 这个锁是老事务留下的，检查 primary 状态
                primary_status = check_primary_lock(version.lock.primary)
                if primary_status == "COMMITTED":
                    cleanup_secondary_commit(key)  # 补提交
                else:
                    cleanup_secondary_rollback(key) # 回滚
    return None
```

**TiKV 的优化**：
- **并行 Prewrite**: 多个 secondary key 可以并行写入
- **Async Commit**: 允许不经过 TSO 的 commit_ts 获取（1PC 优化）
- **Pessimistic Lock**: 增加悲观锁模式，适应高冲突场景（TiDB 3.0+）

### 4.2 Spanner / TrueTime（全局分布式事务 + 原子钟）

Google Spanner（2012 OSDI）是全球首个实现**外部一致性（External Consistency）**的全球分布式数据库。

**核心创新：TrueTime API**

```
TrueTime 提供三个方法:
  TT.now()   → [earliest, latest]  一个时间区间，保证真实时间在这个区间内
                         ↑
           ε = (latest - earliest) / 2  ≈ 1-7ms（Google 数据中心内）
```

**外部一致性保证（Commit Wait）**：

```
对于读写事务：
  1. 获取所有锁
  2. 选择一个 commit timestamp s ≥ TT.now().latest
  3. 等待直到 TT.now().earliest ≥ s（Commit Wait）
  4. 释放锁，返回结果

  → 保证：若事务 T1 在 T2 开始前提交，
         则 T1.commit_ts < T2.commit_ts
```

**Spanner 架构要点**：

```
                    ┌──────────────────┐
                    │   TrueTime API   │ ← GPS + Atomic Clocks
                    └────────┬─────────┘
                             │
   ┌──────────────────┐      │      ┌──────────────────┐
   │   Zone A (Paxos) │◄─────┼─────►│   Zone B (Paxos) │
   │   Spanserver...  │             │   Spanserver...  │
   └──────────────────┘             └──────────────────┘
             │                               │
             └───────── 2PC ─────────────────┘
                     (跨 Paxos Group 事务)
```

**Spanner vs CockroachDB 的关键区别**：

| 维度 | Google Spanner | CockroachDB |
|------|---------------|-------------|
| 时钟 | TrueTime（原子钟硬件） | HLC（Hybrid Logical Clock） |
| 时钟不确定性 | ε ≈ 1-7ms | max_offset = 500ms（默认） |
| Commit Wait | 需要（2ε） | 不需要（Commit Wait 内置为不确定性等待） |
| 外部一致性 | 硬件保证 | 基于不确定性窗口保证 |
| 部署 | Google 专有 | 开源，任何环境 |

### 4.3 Calvin（确定性数据库）

Calvin（SIGMOD 2012）提出了一种**确定性执行**范式——在事务执行前就确定其全局顺序，从而消除传统 2PC 的协调开销。

**核心思想**：

```
传统数据库:                         Calvin:
  请求 → 锁/调度 → 执行 → 提交       请求 → 全局排序 → 确定性执行 → 提交
  (执行顺序决定提交顺序)             (提交顺序决定执行顺序)
```

**Calvin 架构流程**：

```
Phase 1: Sequencing（排序层）
  ┌──────────────────────────────────────────────────┐
  │ 所有事务请求汇聚到 Sequencer                      │
  │ Sequencer 给每个事务分配全局唯一顺序（批处理）     │
  │ 将顺序广播到所有 Scheduler                        │
  └──────────────────────────────────────────────────┘

Phase 2: Scheduling（调度层）
  ┌──────────────────────────────────────────────────┐
  │ 每个分区独立地按照全局顺序执行事务                │
  │ 由于顺序确定 + 代码确定性，所有副本产生相同结果   │
  │ 无需 2PC！多分区事务直接以相同顺序在各分区执行    │
  └──────────────────────────────────────────────────┘

Phase 3: Storage（存储层）
  ┌──────────────────────────────────────────────────┐
  │ 每个分区的存储节点持久化结果                      │
  │ 确定性执行保证多副本一致                          │
  └──────────────────────────────────────────────────┘
```

**为什么 Calvin 不需要 2PC**：
- 传统: 事务 T 涉及分区 A、B → 需要协调 A、B 同时提交
- Calvin: T 被编号为 #42 → A、B 各自独立看到 #42，执行 #42 → 自动一致

**Calvin 的局限性**：
- **必须提前知道所有读写集合**: 不支持交互式事务（一个查询结果决定下一个操作）
- **Sequencer 是瓶颈**: 虽然是可横向扩展的，但设计上所有事务要过排序层
- **延迟偏大**: 批处理 + 广播排序增加了基础延迟

### 5.1 MySQL InnoDB

**架构概览**：

```
                   ┌──────────────────────────────┐
                   │         Server Layer          │
                   │   SQL Parser → Optimizer     │
                   └──────────────┬───────────────┘
                                  │
                   ┌──────────────▼───────────────┐
                   │        InnoDB Engine          │
                   │  ┌─────────────────────────┐ │
                   │  │  Buffer Pool (内存)      │ │
                   │  │  ┌───────┐ ┌──────────┐ │ │
                   │  │  │ Pages │ │ Undo Log │ │ │
                   │  │  └───────┘ └──────────┘ │ │
                   │  └─────────────────────────┘ │
                   │            │                  │
                   │  ┌─────────▼──────────┐      │
                   │  │  Redo Log Buffer     │      │
                   │  └─────────┬──────────┘      │
                   │            ▼                  │
                   │  ┌─────────────────────┐     │
                   │  │ ib_logfile (磁盘)    │     │
                   │  └─────────────────────┘     │
                   └──────────────────────────────┘
```

**MVCC 实现核心**：

InnoDB 通过 **Undo Log（回滚段）** 实现 MVCC：

```
每行数据隐藏列:
  DB_TRX_ID  : 最后一次修改该行的事务 ID (6 bytes)
  DB_ROLL_PTR: Undo Log 指针，指向回滚段记录 (7 bytes)
  DB_ROW_ID  : 行 ID（无主键时自动生成）

ReadView（读视图）判断可见性:
  m_ids[]     : 创建 ReadView 时活跃事务列表
  min_trx_id  : 活跃事务中的最小 ID
  max_trx_id  : 下一个将分配的事务 ID
  creator_trx_id: 创建该 ReadView 的事务 ID

可见性规则:
  if (trx_id == creator_trx_id) → 可见（自己的修改）
  if (trx_id < min_trx_id)      → 可见（已提交）
  if (trx_id >= max_trx_id)     → 不可见（未来事务）
  if (trx_id in m_ids[])        → 不可见（活跃未提交）
  else                          → 可见（已提交）
```

**隔离级别实现差异**：

| 隔离级别 | ReadView 生成时机 | 特殊机制 |
|----------|-------------------|----------|
| Read Uncommitted | 不使用 ReadView，直接读最新 | - |
| Read Committed | 每次 SELECT 创建新 ReadView | - |
| Repeatable Read | 事务中第一次 SELECT 时创建，后续复用 | Gap Lock 防止幻读 |
| Serializable | 同 RR + 所有 SELECT 隐式加 LOCK IN SHARE MODE | - |

**Redo Log 写入策略**（`innodb_flush_log_at_trx_commit`）：

| 值 | 策略 | 持久性 | 性能 |
|:--:|------|:------:|:----:|
| 0 | 每秒刷盘一次 Redo Log Buffer | ❌ 丢 1s 数据 | 最高 |
| 1 | 每次 COMMIT 都 fsync | ✅ 不丢数据 | 标准 |
| 2 | 每次 COMMIT 写 OS Cache，每秒刷盘 | ⚠️ OS 崩溃丢 | 较高 |

### 5.2 PostgreSQL

**MVCC 实现**：

PostgreSQL 不使用 Undo Log，而是把旧版本直接保留在堆表（Heap）中，通过 XID 和元组可见性规则判断：

```
表元组（Heap Tuple）:
  ┌──────────────────────────────────────┐
  │ t_xmin: 插入/创建该元组的事务 ID      │
  │ t_xmax: 删除/更新该元组的事务 ID      │
  │ t_ctid: 指向新版本的指针（更新时）     │
  │ t_infomask: 标志位                   │
  └──────────────────────────────────────┘
```

**XID 回卷（Wraparound）问题**：

PostgreSQL 使用 32-bit 事务 ID，约 43 亿个。当 XID 回卷时，需要特殊处理：

```
XID 空间逻辑视图（环形）:

  2^31-1
    ↓        过去  ←───  当前  ───→  未来
  ┌────┬────┬────┬────┬────┬────┬────┐
  │   ...  │ Frozen │   │ Current │  │
  └────┴────┴────┴────┴────┴────┴────┘
                    ↑
                 0 (回卷点)

Frozen（冻结）: xmin 设为 FrozenTransactionId (2)
  该元组对所有事务可见，不需要再判断 XID 比较

VACUUM FREEZE 必须定期执行，防止 XID 耗尽变成 wraparound
→ 如果达到 age(datfrozenxid) > 2 亿，触发 aggressive VACUUM
→ 如果达到 20 亿，数据库进入只读模式
```

#### 6.3.1 避免滥用分布式事务

#### 6.3.4 TCC 实现中的幂等性

```java
// TCC 的 Confirm/Cancel 必须幂等
// 典型做法：状态机 + 唯一键

public class TransferTccService {
    // Try
    public void tryFreeze(String tccId, String accountId, BigDecimal amount) {
        // 幂等检查
        if (freezeRecordRepo.existsByTccId(tccId)) return;
        // 乐观锁扣减
        int rows = accountRepo.freezeBalance(accountId, amount);
        if (rows == 0) throw new InsufficientBalanceException();
        freezeRecordRepo.insert(new FreezeRecord(tccId, "TRYING"));
    }

    // Confirm
    public void confirm(String tccId) {
        FreezeRecord record = freezeRecordRepo.findByTccId(tccId);
        if (record.getStatus() == "CONFIRMED") return;  // 幂等
        accountRepo.confirmDeduct(record);
        freezeRecordRepo.updateStatus(tccId, "CONFIRMED");
    }
}
```
